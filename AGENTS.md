# 项目协作规则

这份文件只保留这个项目里高频、长期有效的协作规则。

## 协作目标

- 默认使用中文。
- 优先把项目推进成可执行、可验证、可交接的工程状态。
- 优先最小闭环，不一开始把范围膨胀成全量系统。
- 能写入项目目录的稳定信息，优先落盘，不只留在对话里。

## 文档原则

- 长期稳定规则写入 `AGENTS.md` 或 `docs/`。
- 当前阶段最重要的事项写入 `docs/03_current_plan.md`。
- 线程交接与下一步建议写入 `handoff/latest.md`。
- 具体任务的实现、调试、评审结果分别写入相关目录或相关文档。
- README 只做导航，不承载全部历史。
- 所有项目 Markdown 文档统一使用 UTF-8 编码保存，不使用 GBK/ANSI；若发现乱码，先备份原文件，再修复为 UTF-8。
- 使用 PowerShell 写入中文文档时必须显式指定 `-Encoding UTF8`；避免用默认编码追加或覆盖长期维护文档。

## 线程原则

- 一个线程尽量只解决一个方向上的具体问题。
- 当前方向的最小闭环完成后，应主动建议结束线程。
- 若下一个任务需要完全不同上下文，优先新开线程。
- 若只是当前任务的直接续写、小修、小验证，可以复用当前线程。

## 线程结束前必须做的事

- 更新 `handoff/latest.md`。
- 明确写出：
  - 当前线程完成了什么。
  - 下一步候选事项是什么。
  - 每个候选事项建议新开还是复用。
- 如果建议归档当前线程，必须同时给出下一线程名称和可直接复制的启动 prompt。

## 运行规则共识

- planning / execution / validation / control 尽量分阶段思考，不混成一团。
- 默认先判断当前任务处于哪个阶段，再决定怎么推进。
- 共享状态优先压缩，避免无节制新增文档。
- 真正需要人类拍板时，再停机汇报；汇报时应尽量带完整决策包。

## 设计与沟通原则

- 设计阶段默认采用“无情审问式”访谈：持续追问关键分支、约束、依赖和取舍，直到形成可执行共识。
- 每次只问一个问题，不把多个互相依赖的问题打包丢给用户。
- 每个问题都必须附带一个推荐答案，并说明推荐理由、影响范围和可能代价。
- 追问顺序遵循设计树依赖关系：先确定上层目标和边界，再追问角色、流程、数据、状态、异常、权限、接口、验证。
- 对每个设计分支逐一收敛：未达成共识的分支不得假装完成；已达成共识的分支应写入当前设计文档或交接文档。
- 当用户给出模糊回答时，优先将其转化为可选择、可验证、可落地的设计判断。
- 当出现冲突时，优先指出冲突并给出推荐取舍，不直接把冲突揉进文档。

## 代码生成约束

- 代码实现阶段必须逐个模块、逐个功能推进，不得一次性生成大范围系统。
- 每个模块或功能开始编码前，必须先写对应测试用例。
- 编码前需要确认：在该功能尚未实现时，这批测试用例应当失败。
- 完成功能实现后，必须运行并通过本批测试以及必要的既有回归测试，才能进入下一模块或下一功能。
- 所有测试设计、失败记录、修复记录和通过结果都需要留档。
- 详细约束见：`docs/12_code_generation_constraints_v1.md`。
