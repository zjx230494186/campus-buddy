# 09 CodeArts Requirement Tables V1

这份文档用于给小组内部讨论和分工对齐使用。

目标：

1. 按 `RR / SF / IR / User Story` 分别提供一批清晰表格
2. 让你可以直接发给队友一起讨论
3. 让讨论结果可以平滑迁移到 `CodeArts`
4. 尽量兼容队友已经录入的平台内容，减少返工

建议使用方式：

- 先以本文为会议讨论底稿
- 在“是否确认 / 是否修改 / 备注”列中补充意见
- 确认后再统一录入 `CodeArts`

本文主要承接：

- `D:\big_homework\docs\08_requirement_mapping_for_codearts_v1.md`

## 2026-05-14 口径更新说明

本文为早期 CodeArts 讨论表格，保留了 5 条 RR、5 条 SF 的旧结构。最新 LaTeX 版《校园搭子平台需求分析报告》已经形成新的编号基线，后续录入、建模和设计衔接应以最新报告为准：

## 1. RR 表格

> RR 用来表达“真实问题是什么”，不要直接写成功能。

| RR编号 | 当前建议标题 | 问题说明 | 来源依据 | 是否确认 | 是否需要修改 | 备注 |
|---|---|---|---|---|---|---|
| RR1 | 学生想找人结伴做某件事，但现有渠道效率极低 | 学生在找饭搭子、学习搭子、运动搭子、小组作业队友、大创项目队友时，主要依赖群聊、朋友圈、熟人推荐等方式，信息触达低、容易错过时机。 | 问卷与访谈共同指向“找搭子困难”是最高优先级问题 |  |  |  |
| RR2 | 学生需要找到条件匹配的同伴，但缺乏精准筛选手段 | 现有渠道很难按时间、地点、技能、水平、作息、标签等条件筛选，导致即使找到人，也经常不匹配。 | 访谈痛点：匹配精度不足 |  |  |  |
| RR3 | 与陌生人建立联系存在社交压力，缺少低压力的沟通入口 | 用户想联系合适对象，但不愿频繁在群里发消息，也不想一开始暴露外部联系方式。 | 访谈痛点：社交压力 |  |  |  |
| RR4 | 找到的搭子经常爽约、失联、划水，缺乏筛选和约束手段 | 用户担心遇到不守时、不负责、不交付的对象，现有方式无法提前识别和约束。 | 访谈痛点：伙伴稳定性差 |  |  |  |
| RR5 | 平台需要有管理员介入处理违规行为、争议与信用纠纷 | 骚扰、虚假信息、恶意评价、申诉争议等问题需要平台提供审核、处理、纠偏和处罚能力。 | 治理与平台秩序要求 |  |  |  |

## 4. User Story 表格

> User Story 用来表达“谁在什么场景下想达成什么目标，以及为什么”。

> 建议 CodeArts 中每条 US 至少录入：
> - 标题
> - 标准格式
> - 优先级
> - 验收标准（至少2条）
> - 对应 IR

| US编号 | 当前建议标题 | 标准格式 | 对应IR | 优先级 | 验收标准1 | 验收标准2 | 是否确认 | 是否需要修改 | 备注 |
|---|---|---|---|---|---|---|---|---|---|
| US1 | 快速匹配同时间同食堂的饭搭子 | 作为一名想找饭搭子的学生，我希望按时间、食堂和预算快速找到合适对象，这样我不用在群里反复询问却没人回应。 | IR3、IR4、IR5、IR7 | 高 | 用户发布饭搭子需求时可填写食堂、预算、口味偏好等字段。 | 其他用户可在广场中按时间和地点筛选到该需求。 |  |  |  |
| US2 | 按学习目标和作息时间匹配固定学伴 | 作为一名备考或自习学生，我希望按学习目标、作息节奏和监督方式筛选学习搭子，这样我能找到节奏一致的人长期配合。 | IR3、IR4、IR5、IR7 | 高 | 学习搭子需求支持填写学习目标、作息节奏、监督方式。 | 用户可基于这些字段筛选学习搭子需求。 |  |  |  |
| US3 | 按运动水平和场地筛选球友 | 作为一名想找运动搭子的学生，我希望按运动项目、水平和场地筛选对象，这样可以减少水平差异过大带来的沟通成本。 | IR3、IR4、IR5、IR7 | 中 | 运动搭子需求支持填写项目、水平、场地/器材信息。 | 广场中可按项目或水平进行筛选。 |  |  |  |
| US4 | 发布的需求不被群聊淹没，能被真正有空的人看到 | 作为一名发布需求的学生，我希望我的需求在审核通过后进入公开广场，这样真正有相同需求的人可以稳定看到，而不是被普通群消息迅速淹没。 | IR3、IR6、IR7 | 高 | 用户提交需求后，需求状态进入待审核而不是立即公开。 | 审核通过后，该需求可在广场中被其他用户浏览到。 |  |  |  |
| US5 | 按编程等特定技能精准找大创队友 | 作为一名发起大创项目的学生，我希望按项目方向、所需技能和角色筛选队友，这样可以更快找到真正能参与交付的人。 | IR3、IR4、IR5、IR7、IR8 | 高 | 大创项目需求支持填写方向、技能、角色、周期和交付目标。 | 广场或推荐结果中可基于技能与标签进行筛选或排序。 |  |  |  |
| US6 | 组队时过滤经常划水拖延的人 | 作为一名需要长期协作的学生，我希望在联系他人前看到其靠谱分和评价摘要，这样我可以尽量避开经常划水、拖延或不交付的人。 | IR13、IR16 | 高 | 需求详情页或用户主页可展示对方靠谱分。 | 需求详情页或用户主页可展示近期评价摘要标签。 |  |  |  |
| US7 | 在平台内先发起邀约或联系，再决定是否交换联系方式 | 作为一名不想一开始就暴露外部联系方式的学生，我希望先通过低压力邀约或站内联系确认意向，再决定是否交换联系方式，这样联系更有安全感。 | IR9、IR10 | 高 | 用户可从需求详情页发起邀约、意向确认或站内私信。 | 系统至少支持一种不直接暴露外部联系方式的联系路径。 |  |  |  |
| US8 | 发布需求后尽快收到响应而不是石沉大海 | 作为一名发布需求的学生，我希望系统能够通过广场展示和规则推荐提升需求曝光，这样我发布后不至于长期没有回应。 | IR7、IR8 | 中 | 用户发布成功并审核通过后，需求可进入广场公开展示。 | 系统可生成基于规则的推荐结果供其他用户查看。 |  |  |  |
| US9 | 活动结束后记录对方是否守时和合作态度 | 作为一名完成活动或合作的学生，我希望在结束后评价对方的守时、失联情况和合作态度，这样平台能够逐步积累可信记录。 | IR12、IR13 | 高 | 平台能够为已结束活动或合作生成待评价入口。 | 评价结果可影响对方靠谱分或评价摘要展示。 |  |  |  |
| US10 | 如果被误评或被误判，可以提交申诉纠正 | 作为一名被误评或被误处理的学生，我希望提交申诉并上传证据，这样管理员可以结合事实纠正不合理结果。 | IR14、IR15、IR16 | 高 | 用户可针对评价或处理结果发起申诉。 | 申诉时支持填写说明并上传证据材料。 |  |  |  |
| US11 | 管理员可查看并处理举报、申诉和违规账号 | 作为系统管理员，我希望查看用户举报、申诉案件和违规账号信息，这样我可以维持平台秩序并处理争议。 | IR11、IR14、IR15、IR16 | 高 | 管理员可查看待审核内容或案件列表。 | 管理员可执行通过、驳回、警告、禁言、封禁等处理动作。 |  |  |  |
| US12 | 快速找到之前合作愉快的搭子再次联系 | 作为一名已有历史合作记录的学生，我希望在个人空间中查看历史搭子或合作记录，这样我能再次联系合作体验较好的对象。 | IR13 | 中 | 用户可以查看近期评价摘要或历史合作相关记录入口。 | 历史记录能帮助用户辨识曾合作过的对象。 |  |  |  |

## 5. 讨论时建议重点关注的问题

你把这份文档发给队友后，建议重点讨论以下问题：

1. 五类场景是否全部统一为：
   - 饭搭子 / 学习搭子 / 运动搭子 / 小组作业队友 / 大创项目队友
2. “匿名邀约 / 双向确认 / 一键拒绝”是否保留在当前统一版本中，并作为 SF3/IR9/IR10 下的子机制存在
3. “实名认证 / 先审后发 / 站内私信 / 申诉机制”是否作为当前 V1 主线能力
4. 管理端是否明确承担：
   - 审核
   - 案件处理
   - 账号处置
   - 靠谱分干预
5. 这批需求是否能支撑：
   - 总体用例图
   - 两个模块用例图
   - 后续活动图 / 类图 / 顺序图
