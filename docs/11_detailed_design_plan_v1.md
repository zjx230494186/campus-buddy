## 6. 推荐绘制顺序

推荐顺序如下：

1. 先确认模块边界。
2. 再确认核心类图。
3. 再确认状态图。
4. 再绘制活动图。
5. 最后绘制顺序图。

理由：

- 模块边界决定类图的职责划分。
- 类图决定状态和对象关系。
- 状态图决定活动图中的关键分支。
- 顺序图依赖接口、对象和流程都已经稳定。

## 11. 已确认问题清单

### Q1 详细设计第一版覆盖范围

问题：

- 详细设计文档第一版应该覆盖“全部 V1 能力”，还是只覆盖“最小 MVP 闭环”？

结论：

- 覆盖全部 V1 能力，但按 MVP 优先级分层书写。

具体口径：

- P0 / MVP 主链路需要详写，包括认证、发布审核、广场发现、低压力联系等核心闭环。
- P1 能力需要完整覆盖设计边界和关键状态，包括评价、信用摘要、投诉申诉、管理端治理等内容，但可在图表深度和接口细节上低于 P0。
- 暂不实现或后续扩展能力需要明确保留扩展点，避免在详细设计阶段写死结构。
- 分层书写必须同时服务分层开发：每一层都应说明当前要做到什么、后续可以如何扩展、哪些对象或接口需要预留。

影响：

- 正式详细设计文档不能只写最小演示链路。
- 图表和模块说明需要标注优先级。
- 后续代码实现仍应按模块和功能逐步推进，不因设计覆盖全部 V1 而一次性实现全部能力。

### Q2 类图建模方式

问题：

- 类图应该采用“领域对象优先”，还是“数据库表优先”？

结论：

- 采用领域对象优先。

具体口径：

- 详细设计阶段先确认领域对象、职责边界、对象关系和状态行为。
- 类图中的对象不直接等同于数据库物理表。
- 后续数据库表结构可由领域对象、接口契约和实现约束继续推导。
- 核心类图优先覆盖 `User`、`UserProfile`、`AuthenticationRecord`、`PartnerPost`、`Conversation`、`Message`、`Review`、`CreditSummary`、`CaseTicket`、`ModerationRecord`、`Notification`、`Attachment` 等领域对象。

影响：

- 详细设计文档应避免直接写死字段级数据库设计。
- 类图应重点表达对象职责、关联关系、聚合关系和关键状态。
- 后续实现时仍可根据技术选型拆分为数据表、DTO、服务对象或接口对象。

问题：

- 活动图是否优先覆盖 5 条主流程？

结论：

- 是。活动图优先覆盖 5 条主流程。

具体口径：

1. 用户注册、登录与校内身份认证。
2. 需求发布、管理员审核与广场展示。
3. 用户浏览需求、发起邀约或站内消息、交换联系方式。

### Q5 状态图覆盖范围

问题：

- 状态图是否只画“有状态流转风险”的对象？

结论：

- 是。状态图只覆盖状态复杂、容易出错、影响流程判断的对象。

具体口径：

第一版状态图对象为：

1. 用户认证状态。
2. 搭子需求状态。
3. 联系方式解锁状态。
4. 评价任务状态。
5. 投诉/申诉案件状态。

不单独绘制状态图的对象：

- 普通通知。
- 附件。
- 用户资料。
- 静态配置类对象。

影响：

- 状态图用于控制关键业务流转，不作为所有对象的形式化补图。
- 普通对象的状态或字段可在类图、字段说明或接口说明中表达。
- 后续若某个对象出现复杂状态分支，可追加模块级状态图。

### Q7 低压力联系机制模块定位

问题：

- 低压力联系机制是否作为独立模块设计，而不是并入普通消息模块？

结论：

- 是。低压力联系机制作为独立模块设计。

具体口径：

- 低压力联系机制不等同于普通聊天或完整 IM。
- 该模块至少覆盖：
  - 从需求详情发起邀约或站内联系。
  - 会话绑定原始需求。
  - 文本、图片/截图、联系方式卡片。
  - 联系方式解锁条件。
  - 消息举报入口。
  - 后续评价任务的候选关系挂接。
- 普通消息能力只作为低压力联系机制中的沟通载体，不单独扩展为完整社交聊天系统。

影响：

- 正式详细设计文档中需要单列“低压力联系模块”。
- 类图中 `Conversation`、`Message`、`ContactCard` 需要与 `PartnerPost`、`ReviewTask` 或评价候选关系建立联系。
- 顺序图中需要覆盖“发起联系并解锁联系方式”的关键用例。
- 后续开发时不得把该模块扩展为完整 IM，以免超出 V1 边界。

### Q8 信用摘要领域对象定位

问题：

- 信用摘要是否作为独立领域对象设计，而不是只作为 `User` 的一个字段？

结论：

- 是。信用摘要作为独立领域对象设计。

具体口径：

- 信用摘要不等同于用户表中的单一分数字段。
- 详细设计中至少保留以下对象：
  - `CreditSummary`：面向展示的当前信用摘要。
  - `CreditChangeLog`：记录评价、申诉、人工干预等造成的信用变化。
- 信用摘要需要关联：
  - 用户。
  - 评价记录。
  - 投诉/申诉案件。
  - 管理员干预记录。
  - 推荐排序或筛选权重。

影响：

- 类图中 `User` 与 `CreditSummary` 关联，而不是直接把全部信用逻辑塞进 `User`。
- `Review`、`CaseTicket`、`ModerationRecord` 等对象需要能追溯到信用摘要变化。
- 后续实现可以先做简单分值，但设计上必须保留来源解释、纠偏和留痕能力。

### Q11 软件工程开发模型

### TQ6 数据库访问方式

问题：

- 后端数据库访问方式是否采用 Spring Data JPA + Hibernate，而不是 MyBatis？

结论：

- 是。V1 后端采用 Spring Data JPA + Hibernate 作为主要数据库访问方式。

具体口径：

- 常规实体持久化、分页、事务和实体关系由 JPA/Hibernate 承担。
- 复杂查询可按需使用 JPQL、Specification 或少量 native query 补充。
- 当前不采用 MyBatis 作为 V1 主数据访问框架。

推荐理由：

- 当前详细设计采用领域对象优先，JPA/Hibernate 与领域对象、实体关系和状态流转设计更一致。
- Spring Boot 对 Spring Data JPA、事务管理、测试切片和数据库集成有成熟支持。
- V1 的主要数据访问场景以常规 CRUD、分页筛选和状态更新为主，适合使用 JPA 降低样板代码。

影响：

- 后续领域对象设计需要考虑实体边界、关系方向、懒加载边界和事务边界。
- 数据库物理表仍不应反向主导领域模型，但需要在详细设计阶段说明关键实体与表结构的映射原则。
- 若后续出现复杂报表或高复杂查询，应单独标注查询实现策略，避免在通用 Repository 中堆积不可读逻辑。

### TQ7 数据库迁移工具

### TQ8 后端接口形式

- RESTful HTTP + JSON 与 Spring Boot 集成成熟，适合课程项目、小组协作和接口测试。
- Qt 桌面端、移动端 App、接口调试工具和自动化测试均可稳定接入。
- 对当前 V1 的认证、发布、审核、浏览、联系、评价和投诉申诉流程已经足够。

影响：

- 后续详细设计中的客户端与服务端交互应按 API 资源、请求方法、请求体、响应体和错误码组织。
- 接口测试可以作为后端测试留档的重要组成部分。
- 低压力联系机制中的实时性要求仍需单独确认，不能由 REST 接口形态自动推导。

### TQ9 V1 站内消息传输方式

问题：

- V1 的站内消息是否先采用 REST 轮询，而不引入 WebSocket？

结论：

- 是。V1 站内消息先采用 REST 轮询，但后续迭代几乎必然引入 WebSocket，详细设计必须预留传输层扩展边界。

具体口径：

- V1 低压力联系机制不扩展为完整即时聊天系统。
- 站内消息通过 REST 接口提交、分页查询、状态更新和未读数查询。
- 客户端可按固定间隔或页面触发方式轮询消息。
- WebSocket 不纳入 V1 主实现范围，但作为确定性较高的后续演进方向处理。
- 消息模块不应把业务逻辑直接绑定在 REST 轮询上，需要预留消息传输层或消息网关抽象。

推荐理由：

- 低压力联系机制的重点是邀约、站内消息、联系方式卡片、联系方式解锁和评价关系挂接，不是实时聊天体验。
- REST 轮询与当前已确认的 RESTful HTTP + JSON 接口形态一致，实现、测试和调试成本较低。
- 对 V1 闭环而言，REST 轮询能降低初版交付风险，但不能牺牲后续实时通信演进空间。

影响：

- 后续消息相关顺序图和接口设计按 REST 请求/响应建模。
- 消息实时性弱于 WebSocket，可能存在秒级延迟。
- 后续扩展为完整 IM 或更好聊天体验时，需要新增 WebSocket 通道、连接管理、在线状态和更复杂的消息投递策略。

### TQ13 V1 数据库部署方式

问题：

- V1 是否不追求物理独立数据库服务器，改为低成本云服务器自建 PostgreSQL？

结论：

- 是。V1 不追求物理独立数据库服务器。
- V1 采用低成本云服务器自建 PostgreSQL。
- 数据库仍作为独立服务运行，并保留后续迁移路径。

具体口径：

- PostgreSQL 与后端服务可部署在同一台低成本云服务器上。
- PostgreSQL 仍需要独立端口、独立数据目录、独立账号、独立备份和独立迁移脚本。
- 数据库端口默认不直接开放公网。
- 后端通过配置化连接串访问数据库，避免业务代码绑定具体部署位置。
- 后续可迁移到独立数据库服务器或云托管 RDS。

推荐理由：

- 当前阶段需要控制基础设施成本，不适合为 V1 单独购买数据库服务器或立即使用 RDS。
- 自建 PostgreSQL 能满足短期试用、课程交付和小规模真实使用验证。
- 通过 Flyway、备份脚本、配置化连接串和账号隔离，可以保留正式上线演进路径。

影响：

- 正式详细设计可描述为“可迁移的自建 PostgreSQL 部署”，不强调物理同机带来的短期成本策略。
- 内部部署文档需要明确数据库备份、恢复、端口暴露、账号权限和迁移路径。
- 若后续用户量或稳定性要求上升，应优先评估迁移到云托管 RDS。

### TQ16 后端认证方式

问题：

- 后端认证方式是否采用 JWT access token + refresh token？

结论：

- 是。后端认证方式采用 JWT access token + refresh token。

具体口径：

- 桌面端和后续移动端均通过令牌访问后端接口。
- access token 短期有效，用于普通接口访问。
- refresh token 用于续期，并需要服务端保存必要状态以支持注销、异常设备处理和令牌撤销。
- 登录、刷新、注销、认证失效和权限不足需要定义统一错误码与客户端处理规则。

推荐理由：

- Token 认证适合桌面端和未来移动端共用同一后端接口体系。
- access token + refresh token 能兼顾使用体验与安全边界。
- 相比服务端 session，JWT 更适合后续多端接入和接口扩展。

影响：

- 账号认证模块需要设计令牌签发、刷新、失效、注销和异常设备处理流程。
- 客户端需要设计本地凭据保存、过期刷新、退出登录和认证失败回退逻辑。
- 后续安全设计需要确认 refresh token 存储方式、过期策略、轮换策略和撤销策略。

### TQ18 后端安全框架

- 桌面客户端单元测试以 Qt Test 为基础。

- 复杂 UI 端到端自动化测试不作为 V1 主测试框架，后续可结合手工测试清单或专门 GUI 自动化方案补充。

推荐理由：

- 桌面端技术路线已确定为 C++ + Qt Widgets，Qt Test 与 Qt 工程体系最贴近。
- Qt Test 适合验证客户端核心逻辑、数据模型、输入校验和基础交互行为。
- 对 V1 来说，先建立可执行的客户端测试基础，比追求完整 GUI 自动化更稳。

影响：

- 后续客户端实现需要将业务逻辑、API 调用封装和界面展示适度解耦，避免全部逻辑堆在 Widget 事件处理函数中。
- 客户端测试留档需要记录 Qt Test 用例、运行命令和覆盖范围。
- 若后续需要更完整的 UI 端到端测试，需要另行确认 GUI 自动化工具或手工验收规范。

### TQ22 Qt 客户端分层结构

问题：

- 客户端是否采用分层结构：UI Widget 层、ViewModel/Presenter 层、API Client 层、Domain Model 层？

结论：

- 是。Qt 桌面客户端采用分层结构。

具体口径：

- UI Widget 层负责界面展示、用户输入采集和事件转发。
- ViewModel/Presenter 层负责界面状态、表单校验、交互流程和错误提示组织。
- API Client 层负责 HTTP 请求、响应解析、认证令牌附加、错误响应转换和重试/超时边界。
- Domain Model 层负责表达客户端侧业务对象、枚举、状态和简单业务规则。
- 不允许将主要业务逻辑堆入 Widget 事件处理函数。

推荐理由：

- Qt Widgets 项目容易把业务逻辑写进按钮点击事件和界面类中，后续难测试、难维护。
- 分层结构能让 Qt Test 覆盖非 UI 逻辑、状态转换、表单校验和 API 封装。
- 该结构有利于后续移动端或其他客户端复用接口契约和业务对象概念。

影响：

- 客户端详细设计需要说明各层职责、依赖方向和典型调用链。
- 客户端实现时文件和类会增加，但可维护性和可测试性更强。
- 后续 AI coding 需要按层生成客户端代码，不能一次性把页面、网络请求和业务判断混在同一个 Widget 类中。

### TQ23 Qt 客户端 HTTP 通信

问题：

- Qt 客户端的 HTTP 通信是否使用 Qt 原生 `QNetworkAccessManager`？

结论：

- 是。Qt 客户端 HTTP 通信使用 `QNetworkAccessManager`。

具体口径：

- API Client 层基于 `QNetworkAccessManager` 封装后端 HTTP/JSON 调用。
- API Client 层负责统一请求构造、认证头附加、JSON 序列化/解析、错误响应转换、超时和重试边界。
- UI Widget 层不得直接调用 `QNetworkAccessManager`。
- 不在 V1 引入额外第三方 C++ HTTP 客户端库。

推荐理由：

- `QNetworkAccessManager` 是 Qt 原生网络能力，与 Qt 事件循环、信号槽、异步模型和 Qt Test 更贴合。
- 减少第三方依赖，有利于 Windows 桌面端构建、打包和交付。
- 配合 API Client 层可以屏蔽底层网络细节，保证客户端调用后端接口的统一性。

影响：

- 客户端详细设计需要补充 API Client 层的职责、错误模型、超时策略和认证令牌附加规则。
- 后续客户端测试需要覆盖 API Client 对成功响应、错误响应、认证失败和网络异常的处理。
- 若后续引入 WebSocket，实时通信通道应独立封装，不与 REST API Client 混杂。

### TQ24 Qt 客户端令牌存储

问题：

- Qt 客户端本地令牌保存是否采用 `SecureTokenStore` 抽象 + Windows Credential Manager 实现，并禁止明文保存 token？

结论：

- 是。Qt 客户端令牌存储采用 `SecureTokenStore` 抽象。
- Win11 端优先使用 Windows Credential Manager 实现。
- 禁止明文保存 access token 和 refresh token。

具体口径：

- UI Widget 层和 API Client 层不直接操作系统凭据 API。
- 客户端通过 `SecureTokenStore` 保存、读取、覆盖和删除令牌。
- Windows 实现封装 `CredWriteW`、`CredReadW`、`CredDeleteW` 等系统 API。
- 需要处理凭据不存在、读取失败、删除失败、覆盖写入和退出登录清理。

推荐理由：

- JWT access token 和 refresh token 属于敏感凭据，正式上线软件不能明文写入 ini/json 配置文件。
- Windows Credential Manager 是 Win11 桌面端更合适的系统凭据保存位置。
- `SecureTokenStore` 抽象可以隔离平台实现，后续移动端或其他客户端可使用不同安全存储机制。

影响：

- 后续客户端详细设计需要补充令牌保存、读取、刷新、注销和清理流程。
- Qt Test 需要覆盖 `SecureTokenStore` 接口行为和 Windows 实现的关键读写删除场景。
- AI coding 必须先写测试，再实现凭据存储封装，不得绕过抽象直接读写本地文件。

### TQ25 Qt 客户端普通配置边界

问题：

- 客户端配置是否只保存非敏感配置，例如 API base URL、超时时间、轮询间隔，不保存 token 或密码？

结论：

- 是。客户端普通配置只保存非敏感配置。
- token、密码和 refresh token 不得写入普通配置文件。

具体口径：

- 普通配置可使用 Qt 的 `QSettings` 管理。
- `QSettings` 可保存 API base URL、请求超时时间、消息轮询间隔、窗口偏好等非敏感项。
- access token、refresh token、密码和其他敏感凭据只能通过 `SecureTokenStore` 或同等级安全存储机制保存。
- 不得将敏感凭据写入 ini/json/文本文件、日志或调试输出。

推荐理由：

- 普通配置与敏感凭据分离，可以降低误写、误传和日志泄露风险。
- `QSettings` 适合保存普通客户端偏好和运行参数，但不适合存储认证凭据。
- 该边界便于 AI coding 执行：普通配置走配置服务，敏感凭据走 `SecureTokenStore`。

影响：

- 后续客户端详细设计需要补充配置服务或配置管理模块。
- 客户端测试需要覆盖非敏感配置读写和敏感字段禁止写入规则。
- AI coding 约束需要明确禁止把 token、密码或 refresh token 写入 `QSettings`。

### TQ26 后端统一错误响应结构

问题：

- 后端统一错误响应是否采用固定结构，例如 `code`、`message`、`details`、`traceId`？

结论：

- 是。后端统一错误响应采用 `code`、`message`、`details`、`traceId` 结构。

具体口径：

```json
{
  "code": "AUTH_TOKEN_EXPIRED",
  "message": "登录状态已过期，请重新登录",
  "details": {},
  "traceId": "..."
}
```

- `code` 表示稳定错误码，供客户端分支处理和测试断言使用。
- `message` 表示面向用户或客户端展示层的简要说明。
- `details` 表示字段级错误、业务约束错误或附加上下文。
- `traceId` 用于关联后端日志，便于排查问题。

推荐理由：

- Qt 客户端 API Client 层需要把后端错误稳定转换成界面提示、认证状态处理和重试/退出流程。
- OpenAPI 文档和接口测试需要稳定错误结构。
- `traceId` 有利于问题排查和日志关联。

影响：

- 后端需要统一异常处理、错误码表和错误响应 DTO。
- 客户端需要统一错误模型，避免每个页面自行解析错误响应。
- 后续详细设计需要补充错误码分类，例如认证错误、权限错误、参数校验错误、状态冲突、资源不存在和系统异常。

### TQ27 后端结构化日志与 traceId

问题：

- 后端日志是否采用结构化日志，并在每个请求中生成/传播 `traceId`？

结论：

- 是。后端日志采用结构化日志。
- 每个请求需要生成或传播 `traceId`。

具体口径：

- 请求进入后端时，如果请求头中已有 `traceId`，可校验后继续传播；否则由后端生成新的 `traceId`。
- `traceId` 需要写入统一错误响应，并通过 MDC 或同类上下文机制写入日志。
- 认证失败、权限不足、审核操作、投诉申诉处理、消息异常、对象存储上传失败等关键事件需要记录结构化日志。
- 日志不得输出 token、密码、认证材料、联系方式、对象存储临时访问凭据等敏感信息。

推荐理由：

- 统一错误响应中的 `traceId` 只有能关联日志时才有排查价值。
- 结构化日志有利于定位接口异常、状态冲突和治理操作问题。
- 正式上线软件需要具备基本可观测性，而不仅是控制台打印。

影响：

- 后端需要配置日志格式、请求过滤器或拦截器、MDC 上下文和统一异常处理联动。
- 接口测试需要验证错误响应包含 `traceId`。
- 代码生成约束需要明确禁止在日志中输出敏感凭据和敏感材料。

### TQ28 管理端审计日志

问题：

- 是否在后端为管理端关键操作建立审计日志表，而不是只依赖普通运行日志？

结论：

- 是。后端需要为管理端关键操作建立独立审计日志表。

具体口径：

- 审计日志用于记录管理员审核、投诉申诉处理、账号处置、信用干预和其他关键治理操作。
- 审计日志应独立于普通运行日志，不能只依赖控制台日志或结构化运行日志。
- 审计日志至少需要记录操作人、操作类型、目标对象、操作前后关键状态、操作理由、发生时间、来源 IP/设备信息、`traceId`。
- 审计日志应作为管理端治理模块的领域对象或持久化对象，例如 `AdminOperationLog` / `AuditLog`。

推荐理由：

- 管理端治理动作涉及审核责任、申诉裁定、账号处置和信用干预，需要可追踪、可查询、可回溯。
- 普通运行日志主要服务排错，审计日志服务业务责任链和治理记录，两者不能混用。
- 审计日志有利于后续管理端详情页、操作历史、导出和争议复核。

影响：

- 后续详细设计需要补充审计日志领域对象、字段、写入时机和查询权限。
- 管理端关键接口测试需要验证审计日志是否正确写入。
- 审计日志不得记录不必要的敏感原文；涉及材料、联系方式等内容时应记录引用关系或摘要，而不是直接复制敏感内容。

### TQ29 附件访问控制方式

问题：

- 是否为附件访问采用“后端签发临时访问 URL / 受控下载接口”，而不是直接公开对象存储地址？

结论：

- 不采用复杂的临时签名 URL 流程。
- 附件安全由后端业务权限校验和后端访问控制逻辑承担。
- 不直接公开对象存储长期地址。

具体口径：

- 附件访问需要经过后端登录状态、业务归属和操作权限校验。
- 对象存储 key 不作为客户端可长期依赖的公开访问地址。
- V1 不引入预签名 URL、短期 URL 签发、URL 过期校验等复杂机制。
- 后端可以通过受控附件接口或等效方式返回附件内容或可访问结果，但实现重点是后端权限校验，而不是对象存储签名 URL 体系。

推荐理由：

- 认证材料、聊天图片、投诉证据等附件确实需要权限保护。
- 但 V1 引入临时签名 URL 会增加对象存储适配、过期策略、客户端处理和测试复杂度。
- 通过后端统一权限校验可以在复杂度与安全边界之间取得更合适的平衡。

影响：

- 后续详细设计需要补充附件权限规则，例如上传者、被授权会话参与者、投诉案件相关方、管理员等访问边界。
- 后端附件接口需要纳入权限测试，避免越权访问。
- 若未来附件访问量较大或需要 CDN/大文件优化，可重新评估临时签名 URL 或对象存储直传/直下方案。

### TQ30 附件上传方式

问题：

- 对象存储上传是否也由后端中转处理，而不让客户端直传对象存储？

结论：

- 是。V1 附件上传由后端中转处理。
- 客户端不直传对象存储。

具体口径：

- 客户端将附件提交给后端附件接口。
- 后端负责登录状态校验、业务归属校验、文件类型校验、大小限制、病毒/风险扫描预留、对象 key 生成、对象存储写入、附件元数据保存和审核状态初始化。
- 客户端不持有对象存储访问凭据。
- V1 不引入客户端直传签名流程。

推荐理由：

- 后端中转上传能集中处理权限、校验、元数据和审核状态。
- 不需要客户端持有对象存储凭据，降低泄露风险。
- 与“不采用复杂临时签名 URL 流程”的访问控制决策一致。

影响：

- 后端需要控制单文件大小、请求体大小和上传超时。
- 大文件上传会占用后端带宽和处理时间，因此 V1 需要限制附件大小和类型。
- 若未来附件体量增长，可重新评估对象存储直传、分片上传或 CDN 方案。

### TQ31 附件上传类型与大小限制

### DQ2 用户认证状态集合

问题：

### DQ17 密码重置后的令牌撤销规则

问题：

- 密码重置成功后，是否撤销该用户所有已签发的 refresh token，要求其他设备重新登录？

结论：

- 是。密码重置成功后，后端必须撤销该用户全部已签发的 refresh token。

具体口径：

- 用户通过邮箱验证码完成密码重置并设置新密码后，服务端撤销该账号当前所有有效 refresh token。
- 已签发且尚未过期的 access token 可以在短期有效期内自然失效，但不得再通过旧 refresh token 续期。
- 其他设备或客户端会话在 access token 过期后，需要重新登录。
- 该规则同时适用于邮箱密码登录与邮箱验证码登录产生的登录态。

推荐理由：

- 密码重置通常意味着原密码可能遗忘、失控或存在泄露风险。
- 保留旧 refresh token 会让旧设备继续维持长期登录态，削弱密码重置的安全意义。
- 全量撤销 refresh token 可以用较低实现复杂度换取清晰的安全边界。

影响：

- `D:\big_homework\docs\13_detailed_design_v1.md` 已补充密码重置后的 refresh token 撤销规则。
- 后续接口契约和测试用例需要覆盖密码重置成功后旧 refresh token 刷新失败。
- 客户端需要能处理刷新失败并引导用户重新登录。

### DQ18 令牌有效期与刷新轮换规则

问题：

- access token 与 refresh token 的有效期是否设定为 access token 30 分钟、refresh token 14 天，并在刷新时轮换 refresh token？

结论：

- 是。V1 采用 access token 30 分钟、refresh token 14 天，并启用 refresh token 轮换。

具体口径：

- access token 有效期为 30 分钟。
- refresh token 有效期为 14 天。
- 用户刷新登录态时，后端签发新的 access token，并同时签发新的 refresh token。
- 旧 refresh token 在成功刷新后立即失效，不得再次用于续期。
- 服务端需要保存 refresh token 的有效状态、过期时间、撤销时间和轮换关系。

推荐理由：

- 30 分钟 access token 能控制访问令牌泄露后的风险窗口。
- 14 天 refresh token 能兼顾桌面端使用体验，避免用户过于频繁登录。
- refresh token 轮换可以识别旧 token 被重复使用的异常情况，并为后续设备管理、异常登录处置预留基础。

影响：

- `D:\big_homework\docs\13_detailed_design_v1.md` 已补充令牌有效期与 refresh token 轮换规则。
- 后续接口契约需要明确登录、刷新、退出登录、密码重置中的令牌行为。
- 测试用例需要覆盖 access token 过期、refresh token 过期、刷新成功、旧 refresh token 复用失败和撤销后刷新失败。
- 客户端 API Client 层需要在 access token 过期时发起刷新，并在刷新失败时清理本地令牌、引导用户重新登录。

### DQ19 退出登录的令牌撤销范围

问题：

- 退出登录时，是只撤销当前设备的 refresh token，还是撤销该账号全部设备的 refresh token？

结论：

- 普通退出登录只撤销当前设备或当前会话的 refresh token。
- 另设“退出所有设备”能力，用于撤销该账号全部有效 refresh token。

具体口径：

- 用户在当前客户端执行普通退出登录时，后端只撤销当前会话对应的 refresh token。
- 同一账号在其他设备或其他客户端会话中的登录态不受普通退出登录影响。
- 用户主动执行“退出所有设备”时，后端撤销该账号当前全部有效 refresh token。
- `RefreshTokenRecord` 需要记录会话或设备标识、签发来源、失效原因和撤销时间。

推荐理由：

- 普通退出登录通常只表示用户结束当前客户端会话，不应默认打断其他设备的正常使用。
- “退出所有设备”可以覆盖账号风险、设备丢失、共享电脑误登录等安全场景。
- 将二者分离后，客户端文案、接口语义和测试边界都更清晰。

影响：

- `D:\big_homework\docs\13_detailed_design_v1.md` 已补充普通退出当前会话与退出所有设备的区别。
- 后续接口契约需要区分当前会话退出接口和全设备退出接口。
- 测试需要覆盖普通退出不影响其他 refresh token、退出所有设备后全部 refresh token 刷新失败。
- 后续设备管理能力可以基于 `RefreshTokenRecord` 的会话或设备标识扩展。

### DQ20 多设备在线与移动端扩展边界

问题：

- 同一账号是否允许多设备同时在线？

结论：

- 是。同一账号允许多设备或多客户端会话同时在线。
- V1 正式展示和主要实现重心仍以 Win11 桌面 PC 端为主。
- 移动端仅作为后续可扩展接入方向预留，不作为 V1 必交付范围。

具体口径：

- 同一用户可以存在多个有效 refresh token，对应不同设备或不同客户端会话。
- 用户在新设备或新客户端登录时，不默认踢掉旧设备登录态。
- V1 不要求实现完整设备管理页面。
- 后端账号、令牌和接口设计需要保留后续移动端接入空间，避免绑定为单一 PC 客户端专用实现。
- 正式详细设计可以写“预留移动端接入能力”，但不得将移动端写成 V1 必须完成的交付内容。

推荐理由：

- 项目初版展示以 PC 端为主，移动端很可能不会真正实现，因此不应为了移动端提前膨胀 V1 范围。
- 允许多会话在线与后续双端互通目标兼容。
- 后端保留会话记录和接口扩展空间，可以在不增加当前展示负担的前提下维持正式软件设计质量。

影响：

- 后续文档表述应坚持“PC 端为 V1 重心，移动端为扩展预留”，避免把移动端误写成当前实现任务。

### DQ21 密码登录失败限制

问题：

- 连续登录失败是否需要锁定账号或触发临时限制？

结论：

- 需要。连续 5 次密码登录失败后，对该账号的密码登录临时锁定 15 分钟。
- 邮箱验证码登录不使用账号锁定机制，继续使用验证码错误次数失效规则。

具体口径：

- 后端记录账号维度的密码登录失败次数。
- 连续 5 次密码登录失败后，系统对该账号的密码登录临时锁定 15 分钟。
- 锁定期内，后端拒绝继续使用密码登录。
- 密码登录成功后，清零该账号的密码登录失败计数。
- 邮箱验证码登录不因密码登录失败次数被锁定；验证码登录仍按 10 分钟有效、60 秒重发、连续错误 5 次验证码失效处理。

推荐理由：

- 密码登录需要限制暴力猜测风险。
- 邮箱验证码登录已有独立的有效期、重发间隔和错误次数控制，不宜叠加账号锁定，避免过度影响正常用户。
- 将密码登录限制与验证码限制分开，可以让安全策略更清晰，测试边界更明确。

影响：

- `D:\big_homework\docs\13_detailed_design_v1.md` 已补充密码登录失败锁定规则。
- 后端需要记录密码登录失败次数、锁定截止时间和成功登录后的清零规则。
- 客户端需要展示账号密码登录临时锁定提示。
- 测试需要覆盖失败累计、触发锁定、锁定期内拒绝登录、锁定后恢复、成功登录清零，以及验证码登录不受该锁定影响。

### DQ23 认证资料重新提交次数与冷却时间

- 低压力联系会进入真实互动和联系方式交换链路。

- 该规则与 P0-1 的认证边界、P0-2 的发布状态和 P0-3 的自有需求联系限制保持一致。

影响：

- `D:\big_homework\docs\13_detailed_design_v1.md` 已补充 P0-4 发起联系前置条件。
- 发起联系接口需要校验发起者状态、发布者状态、需求状态和发起者与发布者关系。
- 测试需要覆盖未认证发起失败、发布者重新认证期间发起失败、非 `PUBLISHED` 需求发起失败、自我联系失败、双方均符合条件时发起成功。

### DQ65 发起联系后的会话创建

问题：

- 发起联系后是否立即建立会话？

结论：

- 是。发起联系成功后，系统立即创建绑定目标 `PartnerPost` 的 `Conversation`，并发送一条初始邀约消息。

具体口径：

- 发起联系成功后创建 `Conversation`。
- `Conversation` 绑定目标 `PartnerPost`。
- `Conversation` 参与者为发起者和需求发布者。
- 系统发送一条初始邀约消息。
- 初始邀约消息用于表达联系意图。
- 会话创建时联系方式卡片保持未解锁。
- 联系方式卡片后续按联系方式解锁规则处理。

推荐理由：

- 用户发起联系后需要有明确沟通入口。
- 初始邀约消息能表达联系意图，降低直接索取联系方式的压力。
- 先站内沟通、后交换联系方式符合低压力联系原则。

影响：

- `D:\big_homework\docs\13_detailed_design_v1.md` 已补充发起联系后的会话创建规则。
- `Conversation` 需要绑定 `PartnerPost`、发起者和发布者。
- `Message` 需要支持初始邀约消息类型。
- `ContactUnlockRecord` 初始为未解锁。
- 测试需要覆盖发起联系成功创建会话、生成初始邀约消息、联系方式未解锁。

### DQ66 两个用户之间的有效会话唯一性

问题：

- 同一发起者对同一需求是否只能存在一个有效会话？

结论：

- 不按“同一发起者 + 同一需求”限制会话唯一性。
- 按“两个人之间只能存在一个有效会话”限制。

具体口径：

- 同一两个用户之间只能存在一个有效 `Conversation`。
- 用户基于某个 `PartnerPost` 发起联系时，系统先按参与双方查找是否已有有效会话。
- 如果双方已有有效会话，则不重复创建新会话。
- 第二次或后续邀约进入既有会话。
- 既有会话保留先前聊天记录。
- 系统可在既有会话中追加本次邀约上下文和初始邀约消息。
- 不按 `postId + initiatorId` 作为会话唯一性约束。

否定推荐方案的理由：

- 用户认为真正需要避免的是同样两个人之间重复开多个有效对话。
- 两个人第二次因其他需求产生邀约时，先前聊天记录应继续保留。
- 按需求维度禁止多开没有必要，反而会割裂同一双方之间的沟通历史。

影响：

- `D:\big_homework\docs\13_detailed_design_v1.md` 已补充两人之间有效会话唯一规则。
- `Conversation` 唯一性应围绕参与双方，而不是围绕 `postId + initiatorId`。
- `Conversation` 需要能关联或记录多次邀约上下文。
- 测试需要覆盖同一双方首次发起创建会话、同一双方再次邀约复用会话、不同双方可创建不同会话。

### DQ67 多次邀约上下文对象

问题：

- 既然一个会话可承载同一双方多次邀约，是否需要单独设计 `Invitation` 对象来记录每次邀约上下文？

结论：

- 需要。P0-4 单独设计 `Invitation` 对象。

具体口径：

- `Conversation` 表示两个人之间的持续沟通空间。
- `Invitation` 表示某一次基于具体 `PartnerPost` 的邀约事件。
- 发起联系时创建或复用 `Conversation`。
- 发起联系时同时创建一条关联目标需求的 `Invitation`。
- 初始邀约消息可以关联 `Invitation`。
- 后续评价关系可基于具体 `Invitation` 进行挂接。

推荐理由：

- 同一会话可能承载同一双方因不同需求产生的多次邀约。
- 如果没有 `Invitation`，多次邀约上下文会被迫塞进消息文本或会话备注中。
- `Invitation` 可以保持聊天历史连续，同时区分每次联系来源。
- 后续评价和信用摘要更容易挂接到具体邀约事件。

影响：

- 本结论已由 DQ68 撤销，不再作为后续设计基线。
- P0-4 不再新增 `Invitation` 对象。
- 后续不再确认 `Invitation` 状态集合。

### DQ68 撤销 Invitation 对象

问题：

- 是否撤销 DQ67 中“P0-4 新增 `Invitation` 对象”的结论？

结论：

- 是。撤销 DQ67。
- P0-4 不单独设计 `Invitation` 对象。

具体口径：

- 同一双方之间的聊天记录由 `Conversation` 统一维护。
- 基于具体 `PartnerPost` 的邀约上下文直接通过 `Conversation` 及其消息记录表达。
- 不要求划分每一条消息归属于哪一次邀约。
- 同一双方因新的需求再次沟通时，复用既有会话，并追加新的初始邀约消息。
- P0-4 核心对象为 `Conversation`、`Message`、`ContactCard`、`ContactUnlockRecord`、`PartnerPost`。

否定 DQ67 的理由：

- 用户认为按“某一次基于具体 `PartnerPost` 的邀约事件”拆出对象过于啰嗦。
- 当前低压力联系模块不需要精确维护每条消息属于哪一次邀约。
- 使用 `Conversation` 统一维护聊天记录和邀约上下文，模型更克制。

影响：

- `D:\big_homework\docs\13_detailed_design_v1.md` 已移除 P0-4 核心对象清单中的 `Invitation`。
- 后续不设计 `Invitation` 状态集合。
- 后续 P0-4 类图不绘制独立 `Invitation`。
- 测试聚焦会话复用、初始邀约消息追加和联系方式解锁，不测试 `Invitation` 对象创建。

### DQ69 初始邀约消息内容方式

问题：

- 初始邀约消息是否允许用户自定义内容？

结论：

- 允许，并且采用完全自由输入。
- 不采用固定系统模板或“系统模板 + 用户附言”模式。
- 初始邀约消息需要设置长度上限。

具体口径：

- 用户发起联系时自行输入初始邀约消息。
- 系统不强制套用固定模板。
- 系统不将初始邀约拆分为模板正文和用户附言。
- 初始邀约消息必须非空。
- 初始邀约消息需要校验长度上限。
- 具体字数上限已在 DQ70 中确认为 30 个中文字符或等效长度字符。

否定推荐方案的理由：

- 用户认为初始邀约直接完全自由输入即可。
- 模板约束会削弱用户表达空间。
- 通过消息长度限制即可控制边界，不必引入模板结构。

影响：

- `D:\big_homework\docs\13_detailed_design_v1.md` 已补充初始邀约自由输入规则。
- 客户端发起联系界面提供自由文本输入。
- 后端需要校验初始邀约消息非空和长度限制。
- 普通站内消息和初始邀约消息的长度上限已在 DQ70 中统一确认。

### DQ70 消息长度上限

问题：

- 初始邀约消息和普通站内消息的长度上限是否统一？

结论：

- 统一。
- 初始邀约消息和普通站内文本消息均限制为 30 个中文字符或等效长度字符。
- 消息去除首尾空白后必须非空。
- V1 不引入图片、附件、表情等非纯文本消息类型。

推荐方案调整：

- 原推荐为 300 字统一上限。
- 用户认为低压力联系应更简短，过长文本会增加表达压力。
- 因此最终采用 30 字短消息约束。

理由：

- 30 字更符合低压力联系定位，鼓励简短表达联系意图。
- 统一上限便于客户端、后端和测试保持一致。
- 短消息边界可以降低内容审核、界面展示和沟通负担。

影响：

- `Message` 文本字段需要设置 30 字上限。
- Qt 客户端输入框需要提供剩余字数或超限提示。
- 后端接口需要校验空消息、全空白消息、超长消息和 30 字边界。
- 后续如需要更完整的聊天体验，可通过新增消息类型、扩展长度策略或 WebSocket 通道演进，不改变 V1 低压力联系边界。

### DQ71 会话关闭与重新激活

问题：

- 会话是否需要设置“关闭/结束”状态？

结论：

- 需要。
- `Conversation` 设置 `ACTIVE` 与 `CLOSED` 两种业务状态。
- 会话创建后默认为 `ACTIVE`。
- 任一参与方可以主动关闭会话。
- `CLOSED` 状态下不允许继续发送普通消息或解锁联系方式。
- 历史消息保留，用于用户回看和必要治理。
- 同一双方后续因新的 `PartnerPost` 再次发起联系时，复用既有会话，并通过新的初始邀约消息重新激活为 `ACTIVE`。
- 关闭后的会话不得在没有新需求和新初始消息的情况下静默恢复。

理由：

- 主动关闭为用户提供低压力退出机制。
- 两态模型足够支撑 V1，不引入复杂会话生命周期。
- 重新激活必须绑定新的需求和初始邀约，避免用户关闭会话后被无上下文打扰。
- 与“同一双方只能存在一个有效会话”的规则保持一致。

影响：

- `Conversation` 需要状态字段。
- 发送消息、联系方式解锁接口需要校验会话状态。
- 客户端会话列表需要区分进行中会话和已关闭会话。
- 再次发起联系时，后端需要识别既有 `CLOSED` 会话，追加新的初始邀约消息并重新激活。
- 测试需要覆盖关闭后禁止发送普通消息、禁止解锁联系方式、再次基于新需求发起联系后重新激活。

### DQ72 联系方式解锁确认机制

问题：

- 联系方式解锁是否必须双方确认？

结论：

- 是。
- 联系方式卡片默认锁定。
- 只有当双方都在同一 `ACTIVE` 会话中明确确认愿意交换联系方式后，系统才解锁双方联系方式。
- 单方确认不能直接查看对方联系方式。
- 后端返回联系方式前必须校验会话状态、参与关系和双方确认结果。

理由：

- 双方确认更符合低压力联系定位。
- 避免用户一发起联系就变成索取隐私。
- 双方都保留拒绝或退出空间。

影响：

- `ContactCard` 需要表示用户可交换的联系方式内容。
- `ContactUnlockRecord` 需要记录双方确认状态和最终解锁时间。
- 客户端需要展示“我愿意交换”“等待对方确认”“已解锁”等状态。
- 后端接口需要校验会话为 `ACTIVE` 且双方均确认后才返回联系方式。
- 测试需要覆盖单方确认不可见、双方确认后可见、关闭会话后不可解锁等场景。

### DQ73 联系方式卡片内容范围

问题：

- 联系方式卡片 V1 应允许填写哪些联系方式？

结论：

- V1 支持三类联系方式：微信号、手机号、QQ 号。
- 用户可以填写其中一种或多种。
- 用户至少需要维护一种可交换联系方式。
- 校园邮箱不作为联系方式卡片内容。
- 解锁后只返回用户主动填写并确认可交换的联系方式项。

推荐方案调整：

- 原推荐只支持微信号和手机号。
- 用户补充认为 QQ 号也应加入。
- 因此最终将 QQ 号纳入 V1 联系方式卡片范围。

理由：

- 微信号、手机号、QQ 号符合校园线下协作和即时沟通习惯。
- 将联系方式范围控制在三类，可以降低隐私暴露和界面复杂度。
- 校园邮箱已经承担认证职责，不应默认进入社交联系方式交换链路。

影响：

- `ContactCard` 需要支持联系方式类型和值。
- 联系方式类型至少包括 `WECHAT_ID`、`PHONE`、`QQ`。
- 后端需要校验至少一项非空，并分别校验手机号、微信号、QQ 号的基本格式。
- 客户端需要提供联系方式维护界面。
- 解锁接口只返回用户主动维护并允许交换的联系方式项。

### DQ74 联系方式卡片维护时机

问题：

- 用户是否必须在发起联系前先维护联系方式卡片？

结论：

- 是。
- 发起联系前，发起者必须至少维护一种有效的可交换联系方式。
- 如果发起者尚未维护联系方式，则不能发起联系，应先进入联系方式维护流程。
- 需求发布者可以在尚未维护联系方式的情况下发布需求并接收邀约、进行站内沟通。
- 在双方联系方式解锁前，需求发布者也必须补全联系方式卡片。
- 联系方式解锁接口需要校验双方均存在有效 `ContactCard`。

理由：

- 发起方主动进入联系链路，应先具备可交换内容。
- 发布方早期只发布需求，不宜在发布前强制维护联系方式。
- 将发布方补全要求放在解锁前，更符合低压力联系和渐进披露。

影响：

- 发起联系接口需要校验发起者 `ContactCard` 是否有效。
- 联系方式解锁接口需要校验双方是否都有至少一种可交换联系方式。
- 客户端需要在发起联系时引导未维护联系方式的用户补全。
- 客户端需要在解锁联系方式前提示尚未维护联系方式的一方补全。
- 测试需要覆盖发起者未维护联系方式时发起失败、发布者未维护联系方式时可沟通但不可完成解锁、双方补全后可继续解锁。

### DQ75 解锁后联系方式变更

问题：

- 联系方式解锁后，如果用户后来修改或删除自己的联系方式，对已解锁会话是否同步变化？

结论：

- 同步变化。
- 已解锁会话不保存联系方式值的历史快照，只保存双方已完成解锁的事实。
- 用户后续修改联系方式后，对方再次查看时读取最新 `ContactCard` 内容。
- 若用户删除到没有任何可交换联系方式，则已解锁会话中对方暂时看不到联系方式，并提示对方暂未维护可交换联系方式。

理由：

- 不保存历史联系方式快照更利于隐私保护。
- 可以避免旧手机号、旧 QQ 号或旧微信号在系统中长期残留。
- 解锁记录只证明双方曾经同意交换，不等于永久保留对方旧联系方式。

影响：

- `ContactUnlockRecord` 记录解锁事实，不冗余保存联系方式值。
- 联系方式展示接口需要实时读取当前 `ContactCard`。
- 用户删除全部联系方式后，已解锁状态不撤销，但联系方式展示为空。
- 测试需要覆盖解锁后修改联系方式、删除联系方式、重新维护联系方式后的展示行为。

### DQ76 站内消息撤回与删除

问题：

- 站内消息是否允许用户撤回或删除？

结论：

- 不允许。
- V1 不提供站内消息撤回能力。
- V1 不提供站内消息删除能力。
- `Message` 一经发送即作为会话历史保留。
- 客户端不提供撤回、删除单条消息或删除服务端会话记录的入口。

理由：

- 撤回和删除会引入证据链、通知、对方视图一致性等复杂问题。
- 站内消息长度上限较短，定位是低压力联系，不需要完整 IM 的消息管理能力。
- 保留完整会话上下文更有利于后续投诉、申诉和管理员治理。

影响：

- `Message` 不需要设计撤回状态。
- `Message` 不需要设计用户删除状态。
- 客户端不提供撤回或删除入口。
- 投诉治理模块后续可引用必要消息上下文。
- 测试不覆盖撤回或删除成功路径，只覆盖不存在相关操作入口或接口。

### DQ77 站内消息举报入口

问题：

- 站内消息是否需要举报入口？

结论：

- 需要。
- V1 按会话维度提供举报入口。
- 不做逐条消息举报。
- 用户可在会话页发起“举报该会话”，选择原因并填写补充说明。
- 系统将会话、双方用户、关联需求和最近消息上下文提交给后续投诉/申诉模块处理。
- P0-4 只负责提供入口和上下文挂接，具体案件状态、证据补充和管理员处置流程归入 P1 “投诉申诉与案件”。

理由：

- P0-4 需要为骚扰、虚假邀约、恶意沟通保留治理入口。
- 逐条消息举报会增加界面和证据选择复杂度。
- 会话维度举报更适合 30 字短消息和低压力联系定位。

影响：

- 客户端会话页需要提供举报入口。
- 后端需要支持将会话上下文传给投诉/申诉模块。
- P0-4 不展开投诉案件状态机。
- 后续 P1 投诉申诉模块需要承接会话举报来源。
- 测试需要覆盖会话举报入口可用、上下文包含会话与关联需求、不存在逐条消息举报入口。

### DQ78 站内消息未读数

问题：

- 站内消息轮询是否需要未读数？

结论：

- 需要。
- V1 采用 REST 轮询时，支持会话列表未读数和总未读数。
- 用户进入会话详情后，将该会话中对方发送且尚未计入已读位置的消息标记为已读。
- 不做对方可见的已读回执。
- 系统不向消息发送方展示“对方已读”状态。

理由：

- 未读数是消息体验的基础能力。
- 未读数只服务于当前用户的信息提醒，不等于完整 IM 的已读回执。
- 不展示对方已读状态，可以降低社交压力。

影响：

- 需要记录每个用户在每个会话中的 `lastReadMessageId` 或等价已读位置。
- 会话列表接口需要返回单个会话未读数。
- 全局消息入口需要返回总未读数。
- 消息拉取接口需要支持分页。
- 进入会话后需要更新当前用户的已读位置。
- 测试需要覆盖未读数增加、进入会话后清零、只清理当前会话未读数、不向对方暴露已读状态。

### DQ79 站内消息轮询频率

问题：

- 站内消息轮询频率如何设定？

结论：

- V1 站内消息轮询频率统一按 1 秒一次设计。
- Qt 客户端在会话列表和会话详情页均可按 1 秒间隔查询消息更新、会话未读数和总未读数。
- 网络异常时不得阻塞界面，恢复后应重新拉取最新数据。
- V1 仍不引入 WebSocket，但保留后续演进边界。

推荐方案调整：

- 原推荐为会话列表 30 秒、会话详情 5 秒。
- 用户明确否定该方案，认为 1 秒查询一次不会占用过多资源。
- 因此最终采用 1 秒统一轮询。

理由：

- 1 秒轮询更接近即时沟通体验。
- 当前项目规模和 V1 使用场景下，资源占用可接受。
- 业务接口仍保持 REST 查询边界，后续可演进到 WebSocket 或其他推送机制。

影响：

- Qt 客户端需要实现 1 秒轮询调度。
- 会话列表、会话详情、未读数查询需要避免重复请求堆积。
- 后端消息查询接口需要支持按 `afterMessageId` 或分页拉取增量。
- 网络异常、超时和恢复后刷新需要有明确处理。
- 测试需要覆盖 1 秒轮询触发、重复请求控制、异常后恢复刷新和增量拉取。

### DQ80 消息查询增量拉取

问题：

- 消息查询接口是否采用增量拉取？

结论：

- 是。
- 会话详情页消息查询采用 `afterMessageId` 增量拉取新消息。
- 同时保留分页查询历史消息的能力。
- 进入会话时先加载最近一页历史消息。
- 之后客户端维护当前最大 `messageId`，每 1 秒通过 `afterMessageId` 拉取新增消息。
- 用户向上滚动查看历史时，分页加载更早消息。

理由：

- 1 秒轮询如果每次拉取完整消息列表，会造成重复传输和不必要查询。
- 增量拉取能降低轮询成本。
- 历史分页可以保留会话回看能力。

影响：

- 消息接口需要支持最近一页、更早一页、某消息之后新增消息三类查询。
- 客户端需要维护当前会话最大 `messageId`。
- 后端需要保证消息排序稳定。
- 无新消息时接口应返回空列表。
- 测试需要覆盖首次加载、增量拉取、无新消息返回空列表、历史分页和消息顺序稳定。

### DQ81 关闭会话后的历史与联系方式可见性

问题：

- 关闭会话后，是否仍允许查看历史消息和已解锁联系方式？

结论：

- 关闭会话后仍保留历史消息记录。
- 关闭会话后不再提供联系方式查看能力。
- `CLOSED` 表示一段联系关系已经终结。
- 即使关闭前双方已经完成联系方式解锁，系统也不应在关闭后的会话中继续展示对方联系方式。
- 关闭后不得发送新消息、不得发起新的联系方式解锁确认，也不得通过既有解锁记录继续访问联系方式。
- 同一双方后续因新的 `PartnerPost` 再次发起联系时，需要通过新的初始邀约消息重新激活会话，并重新完成双方联系方式解锁确认。

推荐方案调整：

- 原推荐为关闭后仍允许查看历史消息和已解锁联系方式。
- 用户明确否定继续提供联系方式，认为关闭对话意味着一段关系的终结。
- 因此最终采用“历史消息保留、联系方式关闭后不可见”的规则。

理由：

- 关闭会话应当具备明确的关系终止含义。
- 联系方式属于高敏感交换信息，不应在关系终结后继续通过系统展示。
- 历史消息保留用于用户回看必要上下文和后续治理，但不继续提供联系方式。

影响：

- 会话详情接口可以允许参与方查看 `CLOSED` 会话历史消息。
- 联系方式展示接口需要拒绝 `CLOSED` 会话。
- 发送消息和解锁确认接口需要拒绝 `CLOSED` 会话。
- 会话重新激活后，需要重新完成联系方式解锁确认，不能直接复用关闭前的可见性。
- 测试需要覆盖关闭后历史消息可见、联系方式不可见、重新激活后联系方式仍需重新解锁。

### DQ82 关闭会话后的重新激活入口

问题：

- 会话关闭后，是否允许双方互相重新打开，还是只能通过新的需求邀约重新激活？

结论：

- 只能通过新的 `PartnerPost` 邀约重新激活。
- 用户不能在旧会话中直接点击“重新打开会话”。
- 重新激活必须满足新的发起联系前置条件。
- 重新激活必须发送新的初始邀约消息。
- 重新激活后仍需重新完成双方联系方式解锁确认。
- 邀约帖子可以支持手动指定，定向推送给先前有过沟通记录的用户。
- 定向推送只作为新需求邀约入口，不绕过会话状态、发起联系校验或联系方式重新解锁规则。

理由：

- 关闭会话表示一段联系关系终结，旧会话不应被任意重新打开。
- 通过新的需求邀约重新激活，可以保证复联有明确上下文。
- 定向推送可以支持与曾经沟通过的用户再次协作，但必须以新的 `PartnerPost` 为业务入口。

影响：

- 客户端不提供“重新打开会话”按钮。
- 发起联系接口负责识别旧 `CLOSED` 会话并重新激活。
- 重新激活仍需校验双方认证状态、目标需求状态、发起方联系方式卡片和新的初始邀约消息。
- 发布或发布后管理入口需要预留定向推送给历史沟通用户的能力。
- 后续需要继续确认定向推送的接收方范围、是否需要对方可拒收、是否计入通知模块。
- 测试需要覆盖旧会话不能直接重开、基于新需求可重新激活、定向推送不绕过发起联系前置条件。

### DQ83 定向推送接收方范围

问题：

- 定向推送的接收方范围是否只限于“与发布者有过会话记录的用户”？

结论：

- 是。
- V1 定向推送只允许发布者从历史沟通用户中选择接收方。
- 接收方必须与发布者存在历史 `Conversation`。
- 不允许按学号、手机号、邮箱、昵称或其他个人标识搜索任意用户进行定向推送。
- 双方当前不得存在阻断互动的治理限制。

理由：

- 定向推送应服务于复联，而不是陌生人定向打扰。
- 历史会话可以证明双方曾经存在沟通关系。
- 禁止任意用户搜索推送可以降低骚扰和隐私风险。

影响：

- 发布端需要提供历史沟通用户选择入口。
- 后端需要校验发布者与接收方存在历史会话。
- 不提供任意用户搜索推送接口。
- 通知模块后续需要承接定向推送通知。
- 测试需要覆盖无历史会话不可定向推送、有历史会话可选择、治理限制下不可推送、不能通过个人标识搜索任意用户。

### DQ84 拉黑功能边界

问题：

- P0-4 是否需要增加拉黑功能？

结论：

- 需要。
- 任一用户可以拉黑对方。
- 拉黑后，拉黑方不再收到被拉黑方发送的站内消息、定向推送或新的邀约入口。
- 拉黑后，被拉黑方也不得继续查看拉黑方在该关系链路中的消息内容、联系方式或后续定向推送信息。
- 拉黑关系应作为会话、发起联系、消息发送、联系方式查看和定向推送接口的统一前置校验条件。

理由：

- 拉黑是低压力联系中的强退出机制。
- 关闭会话表示关系终结，拉黑则进一步表示不希望再接收对方触达。
- 双向阻断可见性可以避免被拉黑方继续通过历史关系获取拉黑方信息。

影响：

- 需要新增或明确 `BlockRelation` / 黑名单关系对象。
- 发起联系、发送消息、定向推送、联系方式查看接口需要校验双方是否存在拉黑关系。
- 客户端会话页需要提供拉黑入口。
- 拉黑后当前会话应进入不可继续互动状态。
- 后续需要继续确认拉黑是否允许解除、解除后历史消息是否恢复可见、是否需要通知对方。
- 测试需要覆盖拉黑后消息不可达、定向推送不可达、联系方式不可见、被拉黑方不可继续查看拉黑方信息。

### DQ85 解除拉黑规则

问题：

- 拉黑是否允许用户主动解除？

结论：

- 允许。
- 解除拉黑只移除阻断关系。
- 解除后不自动恢复旧会话互动。
- 若旧会话已关闭，则仍保持 `CLOSED`。
- 解除后历史联系方式仍不可见。
- 双方后续如需再次沟通，仍需通过新的 `PartnerPost` 邀约、发送新的初始邀约消息，并重新完成联系方式解锁确认。
- 解除拉黑不自动通知对方。
- 解除拉黑不自动恢复会话、消息可见性或联系方式可见性。

理由：

- 允许解除可以处理误操作。
- 解除拉黑不等于重新建立联系关系。
- 不自动恢复可以保护关闭和拉黑动作的边界。
- 不通知对方可以避免解除动作本身造成新的社交压力。

影响：

- 需要提供黑名单管理入口。
- 解除拉黑接口只删除或失效化 `BlockRelation`。
- 旧会话状态不因解除拉黑改变。
- 旧联系方式可见性不因解除拉黑恢复。
- 测试需要覆盖解除拉黑不恢复旧会话、不恢复联系方式、不通知对方，以及解除后可通过新需求重新发起联系。

### DQ86 拉黑后的历史消息可见性

问题：

- 拉黑后历史消息是否双方都不可见，还是只限制被拉黑方查看拉黑方内容？

结论：

- 双方都不可见。
- 拉黑后，该关系链路中的历史会话消息在双方普通客户端视图中均不可见。
- 客户端会话列表不再展示该会话。
- 普通会话详情接口不得向任一方返回历史消息内容。
- 服务端仍保留消息记录，用于投诉、申诉和管理员治理。
- 解除拉黑后，不自动恢复历史消息可见性。
- 若双方后续通过新的 `PartnerPost` 邀约重新建立联系关系，再按新的会话激活和消息展示规则处理。

理由：

- 用户明确要求拉黑后双方都不可见。
- 双向隐藏比只隐藏单方内容更清晰，避免会话上下文割裂。
- 服务端保留记录可以维持治理证据链。

影响：

- 客户端会话列表需要过滤被拉黑关系会话。
- 普通会话详情接口需要拒绝返回被拉黑关系的历史消息。
- 管理员治理接口仍可按权限查看必要上下文。
- 解除拉黑接口不恢复历史消息普通可见性。
- 测试需要覆盖拉黑后双方普通视图不可见、服务端治理可查、解除拉黑不自动恢复。

### DQ87 拉黑与会话关闭

问题：

- 拉黑后，是否需要自动关闭当前会话？

结论：

- 需要。
- 拉黑动作触发后，当前双方 `Conversation` 应立即进入 `CLOSED`。
- 系统记录关闭原因或阻断来源为 `BLOCKED`。
- 不额外增加第三种会话业务状态。
- 会话业务状态仍保持 `ACTIVE` 与 `CLOSED` 两态。
- 拉黑语义由 `BlockRelation` 或等价黑名单关系对象表达。

理由：

- 自动关闭会话可以阻断后续发送消息、解锁联系方式和旧会话重开。
- 会话状态保持简单，避免新增 `BLOCKED` 状态导致状态机膨胀。
- 拉黑属于用户关系阻断，应由黑名单关系对象承载。

影响：

- 拉黑接口需要关闭双方当前会话。
- `Conversation` 可以记录关闭原因或阻断来源。
- 发消息、联系方式解锁、定向推送和重新激活会话时，需要同时校验 `Conversation.status` 与拉黑关系。
- 测试需要覆盖拉黑后会话进入 `CLOSED`、关闭原因记录为 `BLOCKED`、不产生第三种会话状态。
