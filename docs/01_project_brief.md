# 01 Project Brief

这份文档用于保存这个真实项目的稳定背景。需要长期反复使用的信息，优先写在这里。

## 项目名称

- `校园搭子`

## 项目一句话目标

- 面向校内有明确结伴需求的学生，建设一个可真实部署的校园同伴匹配系统，帮助其更快找到时间、目标与条件更匹配且更可靠的同伴。

## 项目背景

- 当前项目起点来自《软件系统设计与分析》课程作业 PPT。
- 现有材料包含 125 份问卷与 12 人深度访谈的整理结果，但这些内容仍不足以直接支撑开发，需要继续做需求收敛。
- 当前线程处于 planning 阶段，目标是先把“项目真正要做什么”收敛清楚，再进入后续执行线程。
- 用户已确认该项目希望走向真实可落地部署：
  - 后端独立服务器
  - 独立数据库服务器
  - 前端使用成熟网页框架搭建
  - 第一版产品载体定为纯 Web

## 核心用户 / 使用场景

- 核心用户：校内有明确结伴需求、社交圈有限、对伙伴质量有要求的在校大学生。
- 当前已出现的高频场景：
  - 饭搭子
  - 学习搭子
  - 运动搭子
  - 小组作业队友
  - 大创项目队友
- 用户已确认第一版希望直接覆盖以上 5 类场景。

## 当前已知硬约束

- 当前线程只做 planning，不直接进入代码实现。
- 需要遵循项目内约定的 planning / execution / validation / control 分阶段推进方式。
- 本线程需要持续维护 `handoff/latest.md`、`docs/03_current_plan.md`、`docs/01_project_brief.md`。
- 第一版产品要按真实可部署方向规划，而不是只做课堂演示壳子。
- `互评 / 靠谱分 / 实名` 属于第一版必须考虑的能力，不是可随意删除的边缘项。
- 实名强度：学号 + 校园邮箱认证即可。
- 靠谱分同时覆盖爽约 / 迟到 / 失联，以及小组协作中的划水 / 不交付。
- 第一版需要申诉机制。
- 第一版需要完整管理端，覆盖内容审核、用户管理、信用分干预、投诉处理。
- 五类场景采用统一发布 / 匹配流程 + 场景化字段。
- 匹配方式以广场筛选为主、系统推荐为辅。
- 第一版需要站内私信。
- 申诉机制需要用户上传证据，并支持被投诉方回应，管理员人工处理。
- 内容审核覆盖搭子需求、用户资料 / 头像 / 昵称、评价内容、投诉与申诉材料。
- 搭子需求发布采用先审后发。
- 站内私信第一版支持纯文本消息、图片 / 截图、联系方式卡片、消息举报。
- 统一发布公共字段包括场景类型、标题 / 简短说明、时间或时间段、地点、人数、性别偏好、期望条件 / 标签、有效期。
- 五类场景差异字段已确认：
  - 饭搭子：食堂 / 预算 / 口味偏好
  - 学习搭子：学习目标 / 作息节奏 / 监督方式
  - 运动搭子：项目 / 水平 / 场地 / 器材
  - 小组作业：课程 / 截止时间 / 分工角色 / 预期投入
  - 大创项目：项目方向 / 所需技能 / 角色 / 周期 / 交付目标
- 靠谱分 / 申诉处理结果包括不处理、删除 / 纠正评价、扣减靠谱分、恢复靠谱分、警告、临时禁言 / 封禁。
- 靠谱分采用简单可解释规则：初始分 100，正常履约少量加分，违规按严重程度扣分，申诉成功可恢复分数或删除不当评价。
- 系统推荐第一版采用规则推荐，按同场景、时间匹配、地点接近、标签 / 技能 / 水平匹配、靠谱分较高优先排序。
- 第一版只做单校部署，并在数据结构上预留 `school_id`。

## 当前已知软偏好

- 优先最小闭环，不一开始扩展成全量平台。
- 优先把稳定信息写入项目目录，而不是只留在对话里。
- 需求不明确时，不预设过深技术架构。

## 当前暂不做

- 不直接进入实现。
- 不在需求仍模糊时写死架构方案。
- 不同时扩展成多个独立方向讨论。

## 成功信号

- 形成一版可交接的项目 planning 结果。
- 明确项目目标、范围、角色、核心流程、约束、输入输出与验证方式。
- 明确哪些内容已经稳定，哪些仍待用户确认。
- 后续具备继续找真实学生做验证的条件。
