## 2. 原始需求 RR

原始需求（RR）强调“真实问题是什么”，建议保持偏问题陈述，而不是直接写成系统功能。

### RR1

- 标题：
  - 学生想找人结伴做某件事，但现有渠道效率极低
- 问题说明：
  - 学生在找饭搭子、学习搭子、运动搭子、小组作业队友、大创项目队友时，常通过宿舍群、班级群、朋友圈、熟人推荐等方式寻找，但信息容易被淹没，触达效率低，常常错过最佳时间窗口。
- 来源依据：
  - 问卷与访谈共同指向“找搭子困难”是最高优先级问题

### RR2

- 标题：
  - 学生需要找到条件匹配的同伴，但缺乏精准筛选手段
- 问题说明：
  - 现有渠道无法按时间、地点、技能、水平、作息、标签等条件筛选，导致用户即使找到人，也常出现时间冲突、能力不匹配、需求不一致等问题。

### RR3

- 标题：
  - 与陌生人建立联系存在社交压力，缺少低压力的沟通入口
- 问题说明：
  - 用户并不是没有结伴需求，而是不好意思频繁在群里发消息，也不愿意直接暴露外部联系方式，导致很多潜在匹配停留在“想联系但不敢联系”的阶段。
- 说明：
  - 该 RR 保留“社交压力”问题本身，但不等价于必须用匿名邀约解决。

### RR4

- 标题：
  - 找到的搭子经常爽约、失联、划水，缺乏筛选和约束手段
- 问题说明：
  - 无论是饭搭子、学习搭子，还是小组作业、大创项目队友，用户都担心遇到不守时、不负责、不交付的对象，而现有渠道很难提前识别并约束这些风险。

### RR5

- 标题：
  - 平台需要有管理员介入处理违规行为、争议与信用纠纷
- 问题说明：
  - 一旦出现骚扰、虚假信息、恶意评价、违规内容、申诉争议，仅依靠用户自我协调难以解决，平台必须提供审核、处理、纠偏和处罚能力。

## 3. 系统特性 SF

系统特性（SF）用于承接 RR，对应的是“系统层面的能力域”。

### SF1 账号准入与身份可信系统

- 对应 RR：
  - RR3
  - RR4
- 目标：
  - 为平台建立基础可信门槛，确保主要用户具备校内身份，降低冒充、骚扰和失联风险。
- 范围：
  - 校园邮箱验证码登录/注册
  - 学号 + 校园邮箱认证
  - 基础资料提交与认证状态回显

### SF2 需求发布与广场发现系统

- 对应 RR：
  - RR1
  - RR2
- 目标：
  - 让用户能够发布结构化搭子需求，并在广场中被有效发现与筛选。
- 范围：
  - 统一发布
  - 场景化字段
  - 先审后发
  - 广场浏览、搜索、筛选、排序

### SF3 匹配联系、邀约确认与站内沟通系统

- 对应 RR：
  - RR1
  - RR2
  - RR3
- 目标：
  - 在发现合适对象之后，提供低社交压力的站内联系与邀约确认机制，而不是让用户一开始就跳转到外部沟通。
- 范围：
  - 发起邀约
  - 双向确认
  - 发起私信
  - 文本消息
  - 图片 / 截图
  - 联系方式卡片
  - 消息举报

### SF4 信用评价与申诉治理系统

- 对应 RR：
  - RR4
  - RR5
- 目标：
  - 让平台对“靠谱不靠谱”有记录、有展示、有申诉、有纠偏。
- 范围：
  - 活动/合作后互评
  - 靠谱分
  - 评价摘要展示
  - 申诉与证据上传
  - 被投诉方回应

### SF5 管理审核与平台治理系统

- 对应 RR：
  - RR5
- 目标：
  - 为平台提供人工审核、案件处理、账号处置和信用干预能力。
- 范围：
  - 审核工作台
  - 用户管理
  - 案件处理
  - 靠谱分干预

## 4. 研发需求 IR

研发需求（IR）建议写成可被后续设计、建模和实现承接的系统行为描述。

### IR1 身份认证

- 标题：
  - 用户可通过校园邮箱验证码完成登录或注册
- 所属 SF：
  - SF1
- 说明：
  - 系统应支持使用校园邮箱接收验证码完成账号准入。

### IR2 实名认证

- 标题：
  - 用户需提交学号及基础校内信息完成身份认证
- 所属 SF：
  - SF1
- 说明：
  - 系统应支持用户提交学号、学院、年级、昵称等信息，并进入认证审核流程。

### IR3 发布需求

- 标题：
  - 用户可以创建搭子需求并填写统一字段与场景字段
- 所属 SF：
  - SF2
- 说明：
  - 系统应支持围绕 5 类场景发布结构化需求。

### IR4 场景分类

- 标题：
  - 发布需求时支持选择五类搭子场景
- 所属 SF：
  - SF2
- 说明：
  - 场景统一为：饭搭子、学习搭子、运动搭子、小组作业队友、大创项目队友。

### IR5 时地人数等公共字段

- 标题：
  - 发布需求时支持设置时间、地点、人数、性别偏好、标签与有效期
- 所属 SF：
  - SF2

### IR6 先审后发

- 标题：
  - 搭子需求提交后需经管理员审核通过方可进入广场
- 所属 SF：
  - SF2
- 说明：
  - 这是当前 V1 的关键约束，不能弱化为普通补充项。

### IR7 广场浏览与筛选

- 标题：
  - 用户可在广场浏览、搜索、筛选和排序已审核通过的需求
- 所属 SF：
  - SF2

### IR8 规则推荐

- 标题：
  - 系统可根据场景、时间、地点、标签等条件给出规则推荐结果
- 所属 SF：
  - SF2 / SF3
- 说明：
  - 推荐为辅，不应替代广场主路径。

### IR9 发起邀约或联系

- 标题：
  - 用户可从需求详情页向目标对象发起邀约或站内联系
- 所属 SF：
  - SF3
- 说明：
  - 为兼容队友当前版本，保留“邀约”表达；在统一口径下，它可通过站内会话或轻量确认方式落地。

### IR10 邀约确认与多类型消息

- 标题：
  - 系统支持邀约确认、文本消息、图片/截图和联系方式卡片
- 所属 SF：
  - SF3
- 说明：
  - 队友已有的“匿名展示、接受后展示联系方式、一键拒绝”等设想，可优先作为该 IR 下的交互细化，而不必单独删掉重建。

### IR11 消息与需求举报

- 标题：
  - 用户可对需求内容或私信消息发起举报
- 所属 SF：
  - SF3 / SF5

### IR12 互评

- 标题：
  - 活动或合作结束后，双方可对彼此进行互评
- 所属 SF：
  - SF4

### IR13 靠谱分展示

- 标题：
  - 用户主页展示靠谱分及近期评价摘要
- 所属 SF：
  - SF4

### IR14 申诉机制

- 标题：
  - 用户可针对评价、投诉处理结果或相关争议提交申诉并上传证据
- 所属 SF：
  - SF4

### IR15 被投诉方回应

- 标题：
  - 被投诉方可查看案件并补充说明与证据回应
- 所属 SF：
  - SF4

### IR16 平台治理

- 标题：
  - 管理员可审核内容、处理案件、干预靠谱分并处置违规账号
- 所属 SF：
  - SF5

## 5. User Story

以下 User Story 已结合队友原有版本改写为更适配当前 V1 方案的版本。

说明：

- 保留了队友版本中“快速找到人”“精准匹配”“信用筛选”“历史复用”“低压力邀约”“双向确认”等有效内容
- 不再直接删掉匿名邀约类内容，而是将其并入“联系与邀约确认”这一更大能力域
- 每条都补了优先级与至少 2 条验收标准，便于你后续录入 CodeArts

---

### US1 饭搭子快速匹配

- 标题：
  - 快速匹配同时间同食堂的饭搭子
- 标准格式：
  - 作为一名想找饭搭子的学生，我希望按时间、食堂和预算快速找到合适对象，这样我不用在群里反复询问却没人回应。
- 对应 IR：
  - IR3、IR4、IR5、IR7
- 优先级：
  - 高
- 验收标准：
  1. 用户发布饭搭子需求时可填写食堂、预算、口味偏好等字段。
  2. 其他用户可在广场中按时间和地点筛选到该需求。

### US2 学习搭子匹配

- 标题：
  - 按学习目标和作息时间匹配固定学伴
- 标准格式：
  - 作为一名备考或自习学生，我希望按学习目标、作息节奏和监督方式筛选学习搭子，这样我能找到节奏一致的人长期配合。
- 对应 IR：
  - IR3、IR4、IR5、IR7
- 优先级：
  - 高
- 验收标准：
  1. 学习搭子需求支持填写学习目标、作息节奏、监督方式。
  2. 用户可基于这些字段筛选学习搭子需求。

### US3 运动搭子筛选

- 标题：
  - 按运动水平和场地筛选球友
- 标准格式：
  - 作为一名想找运动搭子的学生，我希望按运动项目、水平和场地筛选对象，这样可以减少水平差异过大带来的沟通成本。
- 对应 IR：
  - IR3、IR4、IR5、IR7
- 优先级：
  - 中
- 验收标准：
  1. 运动搭子需求支持填写项目、水平、场地/器材信息。
  2. 广场中可按项目或水平进行筛选。

### US4 发布不被淹没

- 标题：
  - 发布的需求不被群聊淹没，能被真正有空的人看到
- 标准格式：
  - 作为一名发布需求的学生，我希望我的需求在审核通过后进入公开广场，这样真正有相同需求的人可以稳定看到，而不是被普通群消息迅速淹没。
- 对应 IR：
  - IR3、IR6、IR7
- 优先级：
  - 高
- 验收标准：
  1. 用户提交需求后，需求状态进入待审核而不是立即公开。
  2. 审核通过后，该需求可在广场中被其他用户浏览到。

### US5 大创技能匹配

- 标题：
  - 按编程等特定技能精准找大创队友
- 标准格式：
  - 作为一名发起大创项目的学生，我希望按项目方向、所需技能和角色筛选队友，这样可以更快找到真正能参与交付的人。
- 对应 IR：
  - IR3、IR4、IR5、IR7、IR8
- 优先级：
  - 高
- 验收标准：
  1. 大创项目需求支持填写方向、技能、角色、周期和交付目标。
  2. 广场或推荐结果中可基于技能与标签进行筛选或排序。

### US6 过滤低质量队友

- 标题：
  - 组队时过滤经常划水拖延的人
- 标准格式：
  - 作为一名需要长期协作的学生，我希望在联系他人前看到其靠谱分和评价摘要，这样我可以尽量避开经常划水、拖延或不交付的人。
- 对应 IR：
  - IR13、IR16
- 优先级：
  - 高
- 验收标准：
  1. 需求详情页或用户主页可展示对方靠谱分。
  2. 需求详情页或用户主页可展示近期评价摘要标签。

### US7 发起低压力邀约与站内联系

- 标题：
  - 在平台内先发起邀约或联系，再决定是否交换联系方式
- 标准格式：
  - 作为一名不想一开始就暴露外部联系方式的学生，我希望先通过低压力邀约或站内联系确认意向，再决定是否交换联系方式，这样联系更有安全感。
- 对应 IR：
  - IR9、IR10
- 优先级：
  - 高
- 验收标准：
  1. 用户可从需求详情页发起邀约、意向确认或站内私信。
  2. 系统至少支持一种不直接暴露外部联系方式的联系路径。

### US8 运动/需求尽快收到响应

- 标题：
  - 发布需求后尽快收到响应而不是石沉大海
- 标准格式：
  - 作为一名发布需求的学生，我希望系统能够通过广场展示和规则推荐提升需求曝光，这样我发布后不至于长期没有回应。
- 对应 IR：
  - IR7、IR8
- 优先级：
  - 中
- 验收标准：
  1. 用户发布成功并审核通过后，需求可进入广场公开展示。
  2. 系统可生成基于规则的推荐结果供其他用户查看。

### US9 互评记录履约表现

- 标题：
  - 活动结束后记录对方是否守时和合作态度
- 标准格式：
  - 作为一名完成活动或合作的学生，我希望在结束后评价对方的守时、失联情况和合作态度，这样平台能够逐步积累可信记录。
- 对应 IR：
  - IR12、IR13
- 优先级：
  - 高
- 验收标准：
  1. 平台能够为已结束活动或合作生成待评价入口。
  2. 评价结果可影响对方靠谱分或评价摘要展示。

### US10 发起申诉纠偏

- 标题：
  - 如果被误评或被误判，可以提交申诉纠正
- 标准格式：
  - 作为一名被误评或被误处理的学生，我希望提交申诉并上传证据，这样管理员可以结合事实纠正不合理结果。
- 对应 IR：
  - IR14、IR15、IR16
- 优先级：
  - 高
- 验收标准：
  1. 用户可针对评价或处理结果发起申诉。
  2. 申诉时支持填写说明并上传证据材料。

### US11 管理员处理违规与纠纷

- 标题：
  - 管理员可查看并处理举报、申诉和违规账号
- 标准格式：
  - 作为系统管理员，我希望查看用户举报、申诉案件和违规账号信息，这样我可以维持平台秩序并处理争议。
- 对应 IR：
  - IR11、IR14、IR15、IR16
- 优先级：
  - 高
- 验收标准：
  1. 管理员可查看待审核内容或案件列表。
  2. 管理员可执行通过、驳回、警告、禁言、封禁等处理动作。

### US12 查看历史合作对象

- 标题：
  - 快速找到之前合作愉快的搭子再次联系
- 标准格式：
  - 作为一名已有历史合作记录的学生，我希望在个人空间中查看历史搭子或合作记录，这样我能再次联系合作体验较好的对象。
- 对应 IR：
  - IR13
- 优先级：
  - 中
- 验收标准：
  1. 用户可以查看近期评价摘要或历史合作相关记录入口。
  2. 历史记录能帮助用户辨识曾合作过的对象。
