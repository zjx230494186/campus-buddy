# 12 Code Generation Constraints V1

本文定义 `校园搭子` 项目进入代码实现阶段时必须遵守的代码生成与测试约束。

## 0. 文档状态

- 当前阶段：execution constraint
- 适用范围：后续所有代码实现线程
- 约束性质：强约束
- 上游依据：
  - `D:\big_homework\AGENTS.md`
  - `D:\big_homework\docs\06_execution_preparation_v1.md`
  - `D:\big_homework\docs\11_detailed_design_plan_v1.md`

## 1. 总原则

代码实现必须逐个模块、逐个功能推进。

不得一次性生成跨越多个模块的大范围代码。不得在测试缺失、测试未运行或测试失败的情况下继续进入下一模块。

后续开发过程采用“迭代增量模型 + 测试先行 + 阶段评审”的混合模型。每个增量必须能回链到需求编号、设计元素和测试记录。

## 2. 实现顺序约束

每个模块或功能必须遵循以下顺序：

1. 明确本次实现的模块或功能边界。
2. 明确对应需求编号，例如 `IR`、`US`、`FR`。
3. 编写测试用例。
4. 在功能尚未实现时运行测试，确认测试失败。
5. 编写最小实现代码。
6. 运行本批测试，确认全部通过。
7. 运行必要的既有回归测试。
8. 对涉及后端、数据库迁移、对象存储、部署配置或公网访问的批次，执行 Ubuntu 24 服务器 smoke test。
9. 留档测试结果和服务器验证结果。
10. 更新交接文档后，才允许进入下一模块或功能。

## 3. 测试先行约束

### 3.1 编码前必须先写测试

每个功能开始实现前，必须先写对应测试用例。

测试至少覆盖：

- 正常路径
- 关键异常路径
- 权限或状态限制
- 与上游/下游模块的关键契约

### 3.2 必须确认红灯状态

在功能尚未实现前，必须运行测试并确认失败。

失败结果需要说明：

- 哪些测试失败
- 失败原因是否符合预期
- 是否证明当前功能确实尚未实现

如果测试在未实现功能前已经通过，必须先修正测试，不得继续编码。

### 3.3 实现后必须确认绿灯状态

功能实现完成后，必须运行测试并确认：

- 本批新增测试全部通过
- 必要的既有回归测试通过
- 没有因本功能引入明显回归

若测试失败，不得进入下一功能。

## 4. 后端测试框架约束

后端测试框架采用 JUnit 5 + Spring Boot Test + Testcontainers。

- 单元测试采用 JUnit 5。
- Spring 组件、接口和安全链路测试采用 Spring Boot Test。
- 数据库集成测试采用 Testcontainers 拉起真实 PostgreSQL 容器。
- 测试数据库迁移通过 Flyway 执行。
- 不使用 H2 替代 PostgreSQL 作为主要数据库集成测试环境。
- 后端测试留档需要区分单元测试、切片测试、集成测试和接口测试。

## 5. Qt 客户端测试框架约束

Qt 桌面客户端测试采用 Qt Test 作为单元测试基础。

- 客户端测试优先覆盖非 UI 逻辑、模型类、表单校验、客户端 API 封装和部分 Widget 行为。
- 客户端实现需要将业务逻辑、API 调用封装和界面展示适度解耦，避免全部逻辑堆在 Widget 事件处理函数中。
- Qt 客户端采用 UI Widget 层、ViewModel/Presenter 层、API Client 层、Domain Model 层的分层结构。
- AI coding 生成客户端代码时必须明确当前修改属于哪一层，不得把页面、网络请求和业务判断混在同一个 Widget 类中。
- Qt 客户端 HTTP 通信使用 `QNetworkAccessManager`，并且只能由 API Client 层封装调用。
- UI Widget 层不得直接调用 `QNetworkAccessManager`。
- Qt 客户端令牌存储采用 `SecureTokenStore` 抽象，Win11 实现优先使用 Windows Credential Manager。
- 禁止明文保存 access token 和 refresh token。
- AI coding 不得绕过 `SecureTokenStore` 直接读写 ini/json/文本文件保存 token。
- Qt 客户端普通配置可使用 `QSettings` 保存 API base URL、超时时间、轮询间隔等非敏感项。
- `QSettings` 不得保存 token、密码、refresh token 或其他敏感凭据。
- 复杂 UI 端到端自动化测试不作为 V1 主测试框架，后续可结合手工测试清单或专门 GUI 自动化方案补充。
- 客户端测试留档需要记录 Qt Test 用例、运行命令和覆盖范围。

## 6. 模块粒度约束

后续实现任务应按如下粒度拆分：

- 一个模块中的一个子闭环
- 一个页面或接口的一个明确功能
- 一个状态流转或一个业务动作

不推荐的任务粒度：

- “实现整个学生端”
- “实现全部管理端”
- “一次性生成完整后端”
- “把认证、发布、审核、消息、评价全部一起做完”

## 6.1 服务器 smoke test 门禁

后续实现批次不再只以 Win11 本地环境通过作为完整闭环。凡涉及后端服务、数据库迁移、对象存储、部署配置、Nginx/公网 IP 访问或服务器运行配置的批次，必须满足：

1. 本地测试通过。
2. Ubuntu 24 服务器部署或重启成功。
3. 服务健康检查通过。
4. 本批关键接口 smoke test 通过。
5. 数据库迁移或连通性验证通过。
6. 如本批涉及文件能力，对象存储连通性验证通过。
7. 敏感配置确认仅存在于服务器私有环境变量或私有配置中，未进入仓库、长期文档、聊天记录或 Qt 客户端。

Qt 纯客户端界面批次可不强制服务器部署，但若该界面依赖新后端接口，应至少联调已部署服务器接口或明确记录未做服务器联调的风险。

## 7. 留档要求

所有测试都需要留档。

建议留档位置：

- `D:\big_homework\docs\validation\`
- 或当前实现线程指定的验证目录

每批测试至少记录：

- 测试批次名称
- 对应模块或功能
- 对应需求编号
- 测试文件路径
- 红灯运行结果
- 实现摘要
- 绿灯运行结果
- 服务器部署 / smoke test 结果
- 未覆盖风险
- 下一步建议

建议文件命名：

```text
YYYYMMDD_module_feature_test_record.md
```

## 9. 进入下一功能的门槛

只有同时满足以下条件，才能进入下一功能：

1. 本功能边界已记录。
2. 本功能测试已先写。
3. 未实现时测试已失败。
4. 实现后测试已通过。
5. 必要回归测试已通过。
6. 测试记录已落盘。
7. 交接文档已更新。

## 10. 接口契约约束

后端 REST 接口实现必须遵守接口先契约化原则。

- 每个接口实现前必须先明确接口路径、HTTP 方法、认证要求、请求体、响应体、错误响应和对应需求编号。
- 后端接口文档采用 OpenAPI 3 + springdoc-openapi。
- Controller 必须具备 OpenAPI 注解或等效接口说明。
- DTO 字段必须有明确说明，不能只依赖字段名猜测含义。
- 所有接口必须声明认证要求。
- 密码登录必须使用 BCrypt 保存密码哈希，不得保存明文密码，不得自研密码哈希方案。
- 所有错误响应必须使用统一错误结构，并在接口文档中说明。
- 统一错误响应结构为 `code`、`message`、`details`、`traceId`。
- 后端日志采用结构化日志，并在每个请求中生成或传播 `traceId`。
- 日志不得输出 token、密码、认证材料、联系方式、对象存储临时访问凭据等敏感信息。
- 管理端关键治理操作必须写入独立审计日志表，不得只依赖普通运行日志。
- 审计日志不得记录不必要的敏感原文；涉及材料、联系方式等内容时应记录引用关系或摘要。
- 附件访问必须经过后端业务权限校验，不得直接向客户端暴露对象存储长期公开地址。
- V1 不引入复杂的临时签名 URL 流程，除非后续设计另行确认。
- V1 附件上传由后端中转处理，客户端不得直传对象存储，也不得持有对象存储访问凭据。
- 后端附件上传接口必须校验登录状态、业务归属、文件类型、大小限制，并写入附件元数据。
- 普通图片支持 `jpg`、`jpeg`、`png`、`webp`，单文件不超过 5 MB。
- 认证材料/投诉证据支持 `jpg`、`jpeg`、`png`、`pdf`，单文件不超过 10 MB。
- 禁止上传可执行文件、压缩包、脚本文件和其他高风险文件。
- 文件校验不得只检查扩展名，也需要校验 MIME 类型。
- 新增或修改接口时，必须同步更新 OpenAPI 文档。
- 接口测试留档时，应引用接口路径、需求编号和 OpenAPI 契约。

## 11. 与 AI 编码的关系

如果后续使用 CodeArts 智能体或其他 AI 工具生成代码，提示词必须明确包含：

- 本次只实现哪个模块/功能
- 不允许修改哪些模块
- 对应需求编号
- 先写哪些测试
- 未实现时预期失败
- 实现后必须通过哪些测试
- 测试记录写入哪里
- 若涉及 REST 接口，必须先定义 OpenAPI 接口契约，并在实现后同步接口文档

AI 输出代码后，必须人工复核并运行测试；不得仅凭 AI 声称测试通过就进入下一步。

## 12. 当前结论

本项目后续编码不以“生成多少代码”为进度标准，而以“一个功能的测试红灯、实现、绿灯、留档闭环”为进度标准。
