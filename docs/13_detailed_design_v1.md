## 1. 详细设计范围与优先级分层

### 1.1 设计范围

本文覆盖 V1 全部能力，但按 MVP 优先级分层书写。

### 1.2 P0 / MVP 主链路

P0 模块需要详写对象、状态、流程、协作和测试入口。

- 账号认证
- 需求发布与审核
- 广场发现
- 低压力联系

### 1.3 P1 支撑能力

P1 模块需要完整定义边界和关键状态，细节深度可低于 P0。

- 评价与信用摘要
- 投诉申诉
- 管理端治理
- 通知与留痕

P1 文档推进顺序按依赖关系确定为：

1. 评价与信用摘要。
2. 投诉申诉与案件。
3. 管理端治理。
4. 通知与留痕。

该顺序的依据是：评价与信用摘要承接低压力联系结束后的结果沉淀，是投诉申诉、管理端治理和通知留痕的上游对象；投诉申诉依赖评价、会话、需求等争议对象；管理端治理依赖投诉案件和信用变化对象；通知与留痕横向承接各模块状态变化和操作记录。

### 1.4 P2 / 扩展能力

P2 能力仅保留扩展点，不进入当前详细设计主体实现范围。

- 复杂推荐
- 多校扩展
- 完整 IM
- 高级数据看板

## 2. 软件工程过程模型

本项目后续采用“迭代增量模型 + 测试先行 + 阶段评审”的混合软件工程模型。

### 2.1 过程约束

- 需求与设计阶段形成阶段性基线。
- 实现阶段按模块、按功能增量推进。
- 每个功能实现前先写测试，确认未实现时测试失败。
- 每个功能实现后确认新增测试和必要回归测试通过。
- 每个增量完成后进行阶段评审，检查需求回链、设计一致性、测试留档和未解决风险。

- 按业务职责划分，不按页面堆叠。

### 4.4 P0-1 账号准入与身份认证

#### 4.4.1 模块定位

账号准入与身份认证模块负责用户进入平台前后的基础身份闭环，覆盖注册/登录、JWT 签发与刷新、校内身份认证资料提交、认证状态回显、退出登录与令牌撤销。

本模块处理学生侧的身份准入、登录态维护和认证状态感知；管理员对认证资料的审核动作归入管理端治理模块，但其审核结果会回写并影响本模块中的用户认证状态。

#### 4.4.2 参与角色

- 未认证用户：可注册、登录、提交校内身份认证资料、查看认证状态。
- 已认证学生：可保持登录态、刷新令牌、退出登录，并获得后续 P0 主链路能力入口。
- 管理员：不直接属于本模块主操作角色，但其审核结果影响用户认证状态。

未认证用户允许浏览广场和需求详情，但不得发布需求、发起联系、解锁联系方式、评价或投诉。发布、联系、解锁联系方式、评价和投诉均要求用户处于 `VERIFIED` 状态；`SUSPENDED` 用户按治理限制处理。

#### 4.4.3 核心对象

- `User`
- `UserProfile`
- `AuthenticationRecord`
- `CampusEmailVerification`
- `Attachment`
- `RefreshTokenRecord`

#### 4.4.4 状态与约束

本模块至少涉及用户认证状态和令牌状态。

用户认证状态采用以下状态集合：

| 状态 | 含义 | 说明 |
|---|---|---|
| `UNVERIFIED` | 未认证 | 用户已注册或可登录，但尚未提交校内身份认证资料 |
| `PENDING_REVIEW` | 待审核 | 用户已提交认证资料，等待管理员审核 |
| `VERIFIED` | 已认证 | 管理员审核通过，用户获得已认证学生能力 |
| `REJECTED` | 已驳回 | 管理员驳回认证申请，用户可根据规则重新提交 |
| `SUSPENDED` | 已限制 | 治理处置导致账号能力受限，不等同于认证失败 |

认证状态流转规则如下：

| 起始状态 | 触发动作 | 目标状态 | 说明 |
|---|---|---|---|
| `UNVERIFIED` | 完成校园邮箱验证并提交校内身份认证资料 | `PENDING_REVIEW` | 用户完成邮箱验证后首次提交认证资料 |
| `PENDING_REVIEW` | 管理员审核通过 | `VERIFIED` | 用户获得已认证学生能力 |
| `PENDING_REVIEW` | 管理员审核驳回 | `REJECTED` | 用户认证申请被驳回 |
| `REJECTED` | 用户重新提交认证资料 | `PENDING_REVIEW` | 允许根据规则重新提交 |
| `VERIFIED` | 管理端治理限制 | `SUSPENDED` | 仅由治理动作触发，不属于普通认证审核 |

普通认证流程不得直接产生 `SUSPENDED` 状态。

令牌状态需要支持签发、刷新、轮换、撤销和过期。

#### 4.4.5 主要流程

1. 用户通过邮箱密码或邮箱验证码注册/登录。
2. 后端签发 access token 与 refresh token。
3. 未认证用户提交个人身份信息与校内身份材料。
4. 用户完成校园邮箱验证。
5. 用户查看认证状态。
6. 管理员审核认证资料并回写认证结果。
7. 用户刷新令牌或退出登录。

V1 身份验证由个人信息、校内身份材料和校园邮箱验证共同构成。手机号可作为账号联系方式或辅助信息保存，但不作为登录凭证，也不作为校内身份认证的证明依据。用户必须先完成校园邮箱验证，才能提交认证资料并进入 `PENDING_REVIEW`；未完成校园邮箱验证的用户仍停留在 `UNVERIFIED`。

校内身份认证资料提交不设置重新提交次数限制或冷却时间。系统应在提交界面明确标注需要提交的材料，并在客户端与后端进行必填项完整性校验；资料不完整时不得打包提交。认证被驳回后，用户可根据管理员驳回原因补充或更换材料后重新提交。

认证资料被驳回后，系统不保留原提交的敏感资料内容，也不提供历史提交材料回看能力。被驳回申请中的个人敏感明细和材料附件应被删除或解除业务引用，避免长期留存学生个人信息。系统仅保留当前认证状态、驳回原因、处理时间等最小必要的非敏感结果信息，用于用户理解驳回原因并重新提交。

认证审核通过后，系统不长期保留学生证、校园卡等认证材料图片或 PDF 原件。审核完成后应删除认证材料原件或解除业务引用，仅保留认证结果、认证时间、认证方式、校园邮箱和必要的脱敏身份摘要。后续日常业务不得依赖回看认证材料原件；若出现身份争议，应通过重新认证或人工补充材料处理。

处于 `PENDING_REVIEW` 状态的认证材料不设置额外的最长保留时间，也不新增“审核逾期”状态。V1 按管理流程要求在较短时间内完成人工审核，通常不超过 1 天；当前不为低概率逾期场景引入定时过期、自动删除和重新提交流程。

校园邮箱验证采用学校邮箱域名白名单 + 邮件验证码方式：系统先校验邮箱域名是否属于允许的学校域名，再向该邮箱发送验证码，用户完成验证码校验后视为完成校园邮箱验证。

校园邮箱验证码有效期为 10 分钟；同一邮箱验证码发送后 60 秒内不得重复发送；同一验证码连续错误 5 次后失效，需要重新发送验证码。

邮箱验证码登录复用同一套验证码安全规则，即 10 分钟有效、60 秒重发间隔、连续错误 5 次失效。登录验证码与校园邮箱认证验证码用途不同，后端需要区分验证码使用场景。

邮箱密码登录要求密码长度至少 8 位。后端使用 BCrypt 存储密码哈希，不保存明文密码，不自研密码哈希方案。

密码登录需要设置失败限制。连续 5 次密码登录失败后，系统对该账号的密码登录临时锁定 15 分钟；锁定期内拒绝继续使用密码登录。密码登录成功后，清零该账号的密码登录失败计数。邮箱验证码登录不使用账号锁定机制，继续按验证码有效期、重发间隔和错误次数失效规则处理。

V1 提供密码重置流程。用户通过邮箱验证码验证后重置密码，验证码规则复用邮箱验证码安全规则；密码重置后新密码仍需满足至少 8 位并使用 BCrypt 哈希存储。

密码重置成功后，后端必须撤销该用户全部已签发的 refresh token。其他设备在 access token 过期后不得继续刷新登录态，需要重新登录。该规则用于降低密码泄露、邮箱验证码找回后旧设备继续持有长期登录态的安全风险。

V1 令牌有效期采用：access token 有效期 30 分钟，refresh token 有效期 14 天。用户刷新登录态时，后端签发新的 access token，并轮换 refresh token；旧 refresh token 在成功刷新后立即失效，不得再次用于续期。服务端需要保存 refresh token 的有效状态、过期时间、撤销时间和轮换关系，以支持退出登录、密码重置、异常 token 复用识别和后续设备管理扩展。

用户普通退出登录时，仅撤销当前设备或当前会话对应的 refresh token，不影响同一账号在其他设备上的有效登录态。系统另设“退出所有设备”能力，用于撤销该账号当前全部有效 refresh token。该规则要求 `RefreshTokenRecord` 能够区分会话或设备标识，并记录 token 签发来源、失效原因和撤销时间。

同一账号允许多设备或多客户端会话同时在线。V1 正式展示与主要实现重心仍以 Win11 桌面 PC 端为主，不要求实现完整设备管理页面，也不将移动端作为 V1 必交付范围；但后端账号、令牌和接口设计需要保留后续移动端接入空间，避免将登录态、会话记录和业务接口绑定为单一 PC 客户端专用实现。

校园邮箱域名白名单 V1 按单校配置，只允许项目目标学校的官方邮箱域名；配置方式应保留后续扩展为多校域名配置表的空间。

校园邮箱作为校内身份的关键依据，V1 强制遵循一人一校园邮箱一账号原则。用户更换校园邮箱后，原账号信息保留，但必须重新按照注册/认证流程完成校园邮箱验证、认证资料提交和管理员审核；重新认证完成前不得获得已认证学生互动能力。

同一个校园邮箱不得重复注册多个账号。注册、校园邮箱验证和邮箱更换流程均需要校验校园邮箱唯一性；若用户遗忘历史账号或密码，应通过账号找回或密码重置流程恢复原账号访问，而不是使用同一校园邮箱创建新账号。该约束用于保持认证记录、信用摘要、投诉治理和历史业务记录的连续性。

用户更换校园邮箱后，认证状态回退到 `UNVERIFIED`。系统保留其历史发布、会话、评价和信用摘要，但冻结发布、发起联系、解锁联系方式、评价和投诉等互动能力，直到重新完成认证并回到 `VERIFIED` 状态。

邮箱更换后的重新认证期间，历史发布继续对外展示，但需要标记账号认证状态变化；其他用户不得基于这些历史发布发起新的联系或解锁联系方式。已存在的历史会话和评价链路保留。

#### 4.4.6 接口与客户端关系

Qt 客户端通过 API Client 层调用账号与认证相关 REST 接口。令牌保存通过 `SecureTokenStore` 完成，普通配置不得保存 token。认证失败、令牌过期和权限不足均按统一错误响应结构处理。

#### 4.4.7 测试与追踪入口

后续测试至少覆盖注册/登录、校园邮箱唯一性、密码登录失败锁定、多会话登录、令牌刷新、refresh token 轮换、旧 refresh token 复用失败、普通退出当前会话、退出所有设备、认证资料提交、认证资料完整性校验、认证驳回后敏感资料删除、认证通过后材料原件删除、认证状态查询、认证状态权限限制、令牌撤销和密码重置后 refresh token 全部失效。当前不设计认证审核逾期状态和待审核材料自动过期测试。测试用例需要回链到需求编号、接口契约和状态流转。

账号准入与身份认证模块第一轮正文补全已收束。当前已确认登录方式、校园邮箱认证、用户认证状态、邮箱更换与重新认证、令牌生命周期、退出登录、多会话、安全限制、认证资料提交、材料留存和隐私边界。后续若接口契约、管理端审核或实现验证发现冲突，再回到本模块补充修订。

### 4.5 P0-2 需求发布与审核

#### 4.5.1 模块定位

需求发布与审核模块承接已认证学生发布搭子需求的完整发布物生命周期，覆盖草稿保存、提交审核、管理员审核、审核通过后进入广场展示、审核驳回后修改重提、发布者主动下架等能力。

本模块聚焦“发布物从创建到可见或不可见”的状态与审核闭环，不覆盖广场推荐排序、发起联系、联系方式解锁、评价与信用摘要变化。广场展示后的发现与排序归入“广场发现与推荐”模块；基于需求发起邀约、站内消息和联系方式交换归入“低压力联系”模块；活动结束后的评价与信用变化归入“评价与信用摘要”模块。

第一轮核心对象包括：

- `PartnerPost`
- `Attachment`
- `ModerationRecord`

#### 4.5.2 参与角色

- 已认证学生：可新建草稿、编辑草稿、提交审核、查看自己的审核结果、修改被驳回需求并重新提交、主动下架自己的已发布需求。
- 管理员：负责审核待审核需求，给出通过或驳回结果；具体治理操作归入管理端治理模块，但审核结果会回写发布物状态。
- 未认证用户：不可新建草稿、提交审核或发布需求。

只有认证状态为 `VERIFIED` 的用户可以新建草稿、提交审核、修改草稿和下架自己的已发布需求。`UNVERIFIED`、`PENDING_REVIEW`、`REJECTED`、`SUSPENDED` 用户不得发布需求或提交审核。客户端应根据认证状态控制发布入口可见性和可操作性；后端接口必须进行认证状态校验，不能只依赖客户端隐藏入口。

未认证但已登录用户不得创建或保存服务端 `PartnerPost` 草稿。P0-2 不提供“先写草稿，认证通过后再提交”的服务端发布入口；用户必须先完成校内身份认证并处于 `VERIFIED` 状态，才能调用学生侧我的发布接口创建草稿。该规则用于保持发布资格、草稿归属和审核入口一致，避免未认证用户在服务端留下不可提交的发布物。

#### 4.5.3 `PartnerPost` 字段模型

`PartnerPost` 采用“公共字段 + 场景扩展字段”的单对象模型。五类搭子需求不拆分为五个核心领域对象，也不为每类需求单独设计一套发布、审核、广场和联系接口。发布、审核、广场发现和低压力联系主链路统一围绕 `PartnerPost` 协作。

`PartnerPost` 公共字段用于承载所有场景共用的生命周期、审核、展示和检索信息，建议字段如下：

| 字段 | 含义 | 说明 |
|---|---|---|
| `id` | 需求 ID | 系统生成的唯一标识 |
| `publisherId` | 发布者 ID | 指向发布该需求的用户 |
| `sceneType` | 搭子类型 | 饭搭子、学习搭子、运动搭子、小组作业队友、大创项目队友 |
| `status` | 发布状态 | `DRAFT`、`PENDING_REVIEW`、`PUBLISHED`、`REJECTED` |
| `title` | 标题 | 列表、详情和审核队列的主展示字段 |
| `description` | 正文描述 | 需求完整说明，广场列表只展示摘要 |
| `timeMode` | 时间表达方式 | 用于区分明确时间、时间范围或文字化时间偏好 |
| `timeText` | 时间文字 | 面向用户展示的时间描述 |
| `startAt` | 开始时间 | 明确时间或时间范围场景使用，可为空 |
| `endAt` | 结束时间 | 明确时间范围场景使用，可为空 |
| `locationText` | 地点或地点偏好 | 可用于广场地点关键词筛选 |
| `participantCount` | 期望人数 | 表达还需要几人或目标人数 |
| `targetRequirement` | 人数/对象要求 | 性别偏好、年级偏好、能力要求等面向对象的约束摘要 |
| `contactPreference` | 联系方式交换偏好 | 仅表达发布者对联系节奏的偏好，不直接暴露联系方式 |
| `tags` | 标签集合 | 用于展示、筛选和后续推荐扩展 |
| `attachmentIds` | 附件 ID 集合 | 仅保存发布图片附件引用，不保存对象存储裸地址 |
| `scenePayload` | 场景扩展字段 | 按 `sceneType` 进行结构化校验 |
| `rejectReason` | 最近一次驳回原因 | 仅保存最近一次审核驳回原因 |
| `reviewedBy` | 审核管理员 ID | 审核通过或驳回后记录最近一次审核人 |
| `reviewedAt` | 审核时间 | 审核通过或驳回后记录 |
| `publishedAt` | 发布时间 | 进入 `PUBLISHED` 时记录，用于广场默认排序 |
| `createdAt` | 创建时间 | 首次创建草稿时记录 |
| `updatedAt` | 更新时间 | 内容或状态变化时更新 |

`scenePayload` 不应被理解为无约束的任意 JSON。后端提交审核时必须根据 `sceneType` 执行场景字段校验，Qt 客户端也需要按 `sceneType` 切换发布表单。草稿保存阶段允许 `scenePayload` 不完整；提交审核阶段必须同时校验公共必填字段和场景必填字段。

五类场景扩展字段建议如下，后续接口契约和测试批次可在此基础上继续收敛：

| `sceneType` | 场景扩展字段 | 说明 |
|---|---|---|
| `MEAL` | `canteen`、`budgetText`、`tastePreference`、`mealType` | 承接饭搭子中的食堂、预算、口味和餐别 |
| `STUDY` | `studyGoal`、`studyRhythm`、`supervisionMode`、`subjectText` | 承接学习目标、作息节奏、监督方式和学习内容 |
| `SPORT` | `sportType`、`skillLevel`、`venueText`、`equipmentRequirement` | 承接运动项目、水平、场地和器材要求 |
| `COURSE_TEAM` | `courseName`、`assignmentType`、`neededRoles`、`deadlineAt` | 承接课程、小组作业类型、所需角色和截止时间 |
| `INNOVATION_PROJECT` | `projectDirection`、`neededSkills`、`neededRoles`、`cycleText`、`deliverableText` | 承接大创方向、技能、角色、周期和交付目标 |

公共字段用于支撑发布审核主链路和广场基础筛选；场景扩展字段用于提升具体场景的表达精度。广场 V1 的硬筛选优先使用 `sceneType`、`timeText/startAt/endAt`、`locationText`、`participantCount/targetRequirement` 和 `tags`，不要求第一版对所有 `scenePayload` 字段都提供筛选入口。后续如需要按技能、运动水平、预算等字段筛选，应在广场发现模块中逐项确认，不在本模块提前扩大筛选范围。

联系方式值不得写入 `PartnerPost`。`contactPreference` 只表达发布者对联系方式交换节奏的偏好，例如“先站内沟通”“确认后交换联系方式”等；真实微信号、手机号、QQ 号由低压力联系模块中的 `ContactCard` 管理，并在双方满足解锁条件后受控返回。

#### 4.5.4 状态与约束

需求发布状态采用以下状态集合：

| 状态 | 含义 | 说明 |
|---|---|---|
| `DRAFT` | 草稿 | 发布者尚未提交审核，可继续编辑 |
| `PENDING_REVIEW` | 待审核 | 发布者已提交，等待管理员审核 |
| `PUBLISHED` | 已发布 | 审核通过，进入广场展示范围 |
| `REJECTED` | 已驳回 | 审核未通过，发布者可修改后重新提交 |

P0-2 当前不保留 `WITHDRAWN` 状态，也不引入 `EXPIRED`、`CLOSED`、`ARCHIVED` 等状态。活动是否结束、评价是否生成、历史是否归档等问题后续分别在低压力联系、评价与信用摘要或扩展性设计中确认，不提前混入需求发布与审核模块。

需求发布状态流转规则如下：

| 起始状态 | 触发动作 | 目标状态 | 说明 |
|---|---|---|---|
| 无记录 | 发布者显式保存草稿 | `DRAFT` | 创建新的 `PartnerPost` 草稿 |
| `DRAFT` | 发布者编辑并保存草稿 | `DRAFT` | 草稿可反复编辑保存 |
| `DRAFT` | 发布者提交审核且校验通过 | `PENDING_REVIEW` | 内容冻结并进入管理员审核队列 |
| `PENDING_REVIEW` | 发布者撤回审核 | `DRAFT` | 发布者发现内容需修改时主动撤回 |
| `PENDING_REVIEW` | 管理员审核通过 | `PUBLISHED` | 进入广场可展示范围 |
| `PENDING_REVIEW` | 管理员审核驳回 | `REJECTED` | 需填写驳回原因 |
| `REJECTED` | 发布者开始编辑 | `DRAFT` | 返回草稿后可修改并重提 |
| `PUBLISHED` | 发布者主动下架 | `DRAFT` | 下架后不得直接恢复发布，需重新审核 |

除上表外的状态跳转均视为非法操作。后端接口必须以当前服务端状态为准进行校验，客户端状态只能用于交互提示，不能作为最终判断依据。

草稿与正式需求不拆分为两个独立核心领域对象。P0-2 使用 `PartnerPost` 统一承载需求发布物，草稿通过 `PartnerPost.status = DRAFT` 表达。草稿阶段允许部分字段暂未完整；提交审核时必须执行完整性校验，校验通过后状态进入 `PENDING_REVIEW`。文档中不再将 `PostDraft` 作为独立核心领域对象，仅将“草稿”作为 `PartnerPost` 的状态或客户端草稿视图表达。

`PENDING_REVIEW` 状态下发布内容冻结，发布者不得直接编辑需求内容。若发布者需要修改内容，必须先撤回审核，使 `PartnerPost.status` 从 `PENDING_REVIEW` 回到 `DRAFT`；修改完成后再次提交审核，并重新进入审核队列。后端编辑接口必须拒绝对 `PENDING_REVIEW` 状态内容的直接修改。

撤回审核是 P0-2 的正式业务动作。只有发布者本人可以撤回自己处于 `PENDING_REVIEW` 状态的需求；管理员、其他学生和未登录用户不得代替发布者撤回。撤回成功后，`PartnerPost.status` 回到 `DRAFT`，内容保持提交时的最新版本，发布者可继续编辑。撤回审核不要求填写原因，但系统应记录必要操作留痕，例如发布物、操作者、操作类型和操作时间。

撤回审核需要处理与管理员审核的状态并发。若发布者发起撤回时，需求已经被管理员审核通过或驳回，则撤回失败，后端返回状态冲突错误，例如 `POST_STATUS_CONFLICT`；客户端应刷新需求详情或我的发布列表，展示最新状态。管理员审核接口同样必须在提交裁定时校验需求仍处于 `PENDING_REVIEW`，若发布者已撤回，则审核失败并返回状态冲突，避免对已回到草稿的内容继续裁定。

`PUBLISHED` 状态下发布者不得直接编辑已公开内容。若发布者需要修改已发布需求，应先主动下架，使 `PartnerPost.status` 从 `PUBLISHED` 进入 `DRAFT`，修改完成后重新提交审核；审核通过后才能再次进入 `PUBLISHED`。该规则保证所有对外公开内容均经过审核，避免已发布内容绕过先审后发机制。

`REJECTED` 状态表示上一版审核未通过的结果态。发布者开始编辑被驳回需求时，`PartnerPost.status` 应从 `REJECTED` 回到 `DRAFT`；修改完成后重新提交审核并进入 `PENDING_REVIEW`。系统应保留上一版驳回原因用于用户理解修改方向，但不得将“已驳回且内容已修改”混合为同一状态。

管理员审核需求发布时，V1 只提供“通过”和“驳回”两种审核结果。审核通过后，`PartnerPost.status` 从 `PENDING_REVIEW` 进入 `PUBLISHED`；审核驳回后，`PartnerPost.status` 从 `PENDING_REVIEW` 进入 `REJECTED`，且管理员必须填写驳回原因。P0-2 不设置“打回补充但仍保持待审核”“转交其他管理员”“暂缓审核”等复杂审核结果。

P0-2 不设置独立的发布审核记录对象。审核结果、审核时间、驳回原因等仅作为 `PartnerPost` 当前审核状态所需的必要字段保存，用于状态回显和发布者修改。审核动作不额外形成完整历史留痕；如后续管理端治理模块需要审计管理员操作，再在治理模块中统一确认，不在 P0-2 中提前引入。

在不设置独立审核记录的前提下，`PartnerPost` 只保留最近一次驳回原因。发布者开始编辑并回到 `DRAFT` 后，仍可在本次修改界面查看最近一次驳回原因；再次提交审核后，客户端可隐藏该原因，后端可清空或覆盖该字段。系统不提供历史驳回原因列表。

发布需求审核通过后，`PartnerPost.status` 立即进入 `PUBLISHED`，并进入广场可展示范围。P0-2 只负责发布物是否可见的状态开关，不负责广场列表中的排序、筛选、推荐权重和展示策略；这些内容后续归入“广场发现与推荐”模块。广场列表默认只读取 `PUBLISHED` 状态的需求。

发布者主动下架已发布需求后，不允许直接恢复发布。主动下架使 `PartnerPost.status` 从 `PUBLISHED` 回到 `DRAFT`；如需重新展示，发布者必须重新提交审核，审核通过后才能再次进入 `PUBLISHED`。系统不保留 `WITHDRAWN` 状态，主动下架作为操作记录或审核相关留痕保存。主动下架不要求发布者填写下架原因，系统仅记录必要操作留痕，例如发布物、操作者、操作类型和操作时间。

同一用户允许同时发布多个 `PUBLISHED` 需求，但 V1 设置宽松数量上限：同一用户最多同时保留 10 条 `PUBLISHED` 需求。达到上限后，用户可以继续保存草稿，但不得提交新的需求进入审核；管理员审核通过前也需要校验该用户当前已发布数量，避免审核通过后突破上限。

需求发布中的附件图片作为发布内容的一部分进入审核。若 `PartnerPost` 携带附件图片，管理员审核页需要提供附件预览；审核通过后附件随需求进入可展示范围，审核驳回后附件不对外展示。附件可见性跟随 `PartnerPost.status`，避免通过图片绕过先审后发机制。

`PartnerPost` 提交审核时至少需要满足以下必填字段完整性：

- 标题。
- 搭子类型。
- 活动时间或时间偏好。
- 地点或地点偏好。
- 人数/对象要求。
- 正文描述。
- 联系方式交换偏好。

附件图片作为选填内容，不作为所有搭子需求的必填项。客户端需要在提交审核前进行表单校验，后端提交审核接口需要再次校验必填字段，避免绕过客户端校验提交无效需求。

需求发布表单需要支持自动保存草稿，但自动保存限定在 Qt 客户端本地缓存中完成，不因每次输入变更立即同步到后端。用户显式点击“保存草稿”或“提交审核”时，客户端才调用后端接口写入或更新 `PartnerPost`。后端只处理显式保存草稿和提交审核，不将客户端本地自动保存作为服务端核心流程。

客户端本地自动保存草稿不要求加密存储，但不得以 txt、docx、明文 json 等容易被普通文本查看器直接读取和误修改的格式保存。Qt 客户端应使用专用本地缓存格式，并加入基础完整性校验或版本标识，降低误修改和误读取风险。草稿本地缓存不得使用 `QSettings` 保存，且不应包含 access token、refresh token、密码等敏感凭据。

#### 4.5.5 接口契约草案

需求发布与审核接口按“学生侧我的发布接口”和“管理侧审核接口”两组拆分。学生侧接口只处理当前登录用户自己的发布物生命周期；管理侧接口只处理管理员审核队列、审核详情和审核裁定。两组接口不共用万能写接口，避免学生操作和管理员操作混在同一权限边界中。

学生侧接口草案如下：

| 接口 | 用途 | 认证要求 | 关键规则 |
|---|---|---|---|
| `POST /api/me/partner-posts` | 创建草稿或显式保存新草稿 | 登录且 `VERIFIED` | 初始状态为 `DRAFT`，允许字段不完整 |
| `PUT /api/me/partner-posts/{postId}` | 更新我的草稿 | 登录且 `VERIFIED`，且为发布者本人 | 仅允许更新 `DRAFT` 状态；`PENDING_REVIEW`、`PUBLISHED` 不允许直接编辑 |
| `POST /api/me/partner-posts/{postId}/submit-review` | 提交审核 | 登录且 `VERIFIED`，且为发布者本人 | 仅允许 `DRAFT -> PENDING_REVIEW`；提交前校验公共必填字段和场景必填字段 |
| `POST /api/me/partner-posts/{postId}/withdraw-review` | 撤回审核 | 登录且 `VERIFIED`，且为发布者本人 | 仅允许 `PENDING_REVIEW -> DRAFT`；若已被管理员裁定，返回 `POST_STATUS_CONFLICT` |
| `POST /api/me/partner-posts/{postId}/unpublish` | 主动下架 | 登录且 `VERIFIED`，且为发布者本人 | 仅允许 `PUBLISHED -> DRAFT`；下架后重新展示必须再次提交审核 |
| `GET /api/me/partner-posts` | 获取我的发布列表 | 登录用户 | 支持按状态分页查询，返回可执行动作和最近驳回原因摘要 |
| `GET /api/me/partner-posts/{postId}` | 获取我的发布详情 | 登录用户，且为发布者本人 | 返回完整编辑所需字段和状态动作，不返回对象存储裸地址 |

管理侧接口草案如下：

| 接口 | 用途 | 认证要求 | 关键规则 |
|---|---|---|---|
| `GET /api/admin/partner-posts/review-queue` | 获取需求审核队列 | `ROLE_ADMIN` | 默认只返回 `PENDING_REVIEW`，按提交时间或风险摘要排序 |
| `GET /api/admin/partner-posts/{postId}` | 获取待审核需求详情 | `ROLE_ADMIN` | 返回审核所需公共字段、场景字段、附件受控预览入口和发布者必要摘要 |
| `POST /api/admin/partner-posts/{postId}/review` | 提交审核裁定 | `ROLE_ADMIN` | 仅允许裁定 `PENDING_REVIEW`；通过进入 `PUBLISHED`，驳回进入 `REJECTED` 且必须填写原因 |

学生侧写接口请求体围绕 `PartnerPost` 公共字段和 `scenePayload` 组织。草稿保存接口允许字段缺失；提交审核接口不额外接收整份发布内容，只对当前已保存的 `PartnerPost` 执行完整性校验和状态流转。该规则避免“保存草稿”和“提交审核”两条写入路径各自携带不同版本内容。

管理侧审核接口请求体只允许提交审核结果，不允许同时修改发布内容。审核请求体至少包含 `decision` 和可选或必填的 `reason`：当 `decision = REJECT` 时，`reason` 必填；当 `decision = APPROVE` 时，`reason` 可选。管理员不得在审核通过前替发布者改标题、正文、场景字段或附件。

| `status` | `DRAFT`、`PENDING_REVIEW`、`PUBLISHED`、`REJECTED` | P0-2 发布物四态 |

字段长度和值域约束：

| 字段 | 约束 | 说明 |
|---|---|---|
| `title` | 1 到 40 个中文字符或等效长度字符 | 提交审核时必填；草稿可为空，但若填写则不得超过上限 |
| `description` | 1 到 500 个中文字符或等效长度字符 | 提交审核时必填；广场列表摘要另按展示规则截断 |
| `timeText` | 不超过 60 个中文字符或等效长度字符 | `TEXT_PREFERENCE` 时提交审核必填 |
| `locationText` | 1 到 80 个中文字符或等效长度字符 | 提交审核时必填 |
| `targetRequirement` | 1 到 120 个中文字符或等效长度字符 | 提交审核时必填 |
| `contactPreference` | 1 到 80 个中文字符或等效长度字符 | 提交审核时必填，不得包含手机号、微信号、QQ 号等联系方式值 |
| `tags` | 最多 8 个，每个 1 到 12 个中文字符或等效长度字符 | 重复标签应去重或拒绝 |
| `participantCount` | 1 到 20 | 表达期望人数或还需人数，P0 不支持超大规模活动 |
| `rejectReason` / `reason` | 驳回时 1 到 200 个中文字符或等效长度字符 | 管理员驳回时必填 |
| `attachmentIds` | 最多 6 个 | 仅允许发布图片附件引用，附件类型和大小沿用项目附件规则 |

时间字段校验规则：

- `timeMode = EXACT_TIME` 时，`startAt` 必填，`endAt` 可为空。
- `timeMode = TIME_RANGE` 时，`startAt` 和 `endAt` 均必填，且 `endAt` 必须晚于 `startAt`。
- `timeMode = TEXT_PREFERENCE` 时，`timeText` 必填，`startAt` 和 `endAt` 可为空。
- 提交审核时不得提交明显早于当前时间的明确活动时间；草稿保存阶段可以暂不执行该强校验，但不得接受无法解析的日期时间格式。

场景扩展字段的字符串长度默认不超过 120 个中文字符或等效长度字符；技能、角色等列表型字段默认最多 10 项，每项不超过 20 个中文字符或等效长度字符。若某个场景后续需要更精细限制，应在对应场景字段表中追加，而不是在实现中隐式扩大或缩小。

两组接口统一使用项目错误响应结构 `code`、`message`、`details`、`traceId`。P0-2 至少需要约定以下错误码，后续实现可在不改变语义的前提下细化：

| 错误码 | 适用场景 |
|---|---|
| `UNAUTHORIZED` | 未登录访问需要登录的接口 |
| `FORBIDDEN` | 非发布者操作他人发布物，或非管理员访问管理接口 |
| `AUTHENTICATION_STATUS_REQUIRED` | 登录用户不是 `VERIFIED`，不得创建、编辑、提交、撤回或下架需求 |
| `POST_NOT_FOUND` | 发布物不存在，或当前用户无权感知该发布物 |
| `POST_STATUS_CONFLICT` | 当前状态不允许该动作，或撤回/审核发生状态并发冲突 |
| `VALIDATION_FAILED` | 公共字段或场景字段校验失败 |
| `PUBLISHED_POST_LIMIT_EXCEEDED` | 同一用户已达到 10 条 `PUBLISHED` 需求上限 |

公共字段或场景字段校验失败统一返回 `VALIDATION_FAILED`，不为每个字段单独设计错误码。错误响应中的 `details` 应提供字段路径到错误信息的映射，便于 Qt 客户端将错误定位到具体表单项。字段路径可使用公共字段名或 `scenePayload` 下的嵌套路径，例如 `title`、`startAt`、`scenePayload.neededSkills`、`scenePayload.deadlineAt`。

提交审核接口执行强校验：公共必填字段和当前 `sceneType` 对应的场景必填字段缺失、格式错误、时间范围非法、人数要求非法、标签数量或长度超限时，均返回 `VALIDATION_FAILED`。保存草稿接口执行弱校验：允许业务字段不完整，但仍应拒绝明显非法的数据形态，例如无法识别的 `sceneType`、超长标题、超长正文、非法附件引用或不符合结构的 `scenePayload`。

`VALIDATION_FAILED` 的 `details` 结构建议如下，具体字段文案可在实现时按客户端展示需要调整：

```json
{
  "code": "VALIDATION_FAILED",
  "message": "提交内容校验失败",
  "details": {
    "title": "标题不能为空",
    "locationText": "地点或地点偏好不能为空",
    "scenePayload.neededSkills": "请至少填写一项所需技能"
  },
  "traceId": "..."
}
```

Qt 客户端 API Client 层应按上述两组接口分别封装学生侧发布服务和管理侧审核服务。学生端页面不得调用管理侧接口；管理员审核 UI 不应复用学生侧编辑接口修改发布物内容。未认证用户的发布入口应直接禁用或引导其先完成校内身份认证，不向后端发送创建草稿请求。

### 4.6 P0-3 广场发现与推荐

#### 4.6.1 模块定位

广场发现与推荐模块承接已发布需求的广场列表、筛选、搜索、需求详情查看和基础排序/推荐入口；其输入以前序模块中 `PUBLISHED` 状态的 `PartnerPost` 为准。

本模块负责让用户发现需求，并基于标题、类型、时间、地点、对象要求、发布者摘要和信用摘要等信息判断是否感兴趣；不覆盖发起邀约、站内消息、联系方式卡片或联系方式解锁。需求详情页可以提供联系入口，但入口触发后的业务动作归入 P0-4 “低压力联系”模块。

未认证用户允许查看广场列表和需求详情，但只能查看 `PUBLISHED` 需求的公开信息。未认证用户在需求详情页看到的联系入口应处于禁用或引导认证状态，不得发起邀约、站内消息、联系方式解锁或其他真实互动。后端需要区分公开浏览权限和互动权限，不能仅依赖客户端隐藏联系入口。

广场列表和需求详情不对未登录游客开放。V1 要求用户至少完成登录后才能浏览广场；登录但未认证用户可以浏览 `PUBLISHED` 需求的公开信息，只有 `VERIFIED` 用户可以进入后续互动链路。该规则将“登录”作为浏览门槛，将“校内身份认证”作为互动门槛。

广场列表 V1 基础筛选条件包括搭子类型、时间偏好、地点关键词、人数/对象要求。发布者信用摘要在列表项和详情页中展示，作为用户判断参考，但不作为 V1 硬筛选条件；P0-3 不提供按信用摘要区间筛选的能力。

广场搜索范围限定在需求标题和正文描述。`PostSearchCriteria.keyword` 仅匹配 `PartnerPost.title` 与 `PartnerPost.description`；搭子类型、时间偏好、地点关键词、人数/对象要求通过结构化筛选表达，不混入全文搜索语义。

广场默认排序采用发布时间倒序。V1 默认按 `PUBLISHED` 需求的发布时间从新到旧展示，`PostSortStrategy` 默认值为 `LATEST`；同一发布时间下使用稳定次序字段补充排序，例如 `postId` 或创建时间。P0-3 暂不引入复杂个性化推荐算法，但保留排序策略对象用于后续扩展推荐排序。

广场列表需要分页加载。后端列表接口采用分页查询，默认每页 20 条，并明确 `page`、`size`、`sort` 等请求参数；Qt 客户端交互明确采用滚动加载，在用户接近列表底部时加载下一页。筛选、搜索和排序条件变化时，客户端应重置分页状态并从第一页重新加载。

需求详情页不展示发布者完整个人资料。`PostDetailView` 只展示发布者公开摘要，包括昵称/展示名、认证状态、信用摘要，以及必要的学院、年级等脱敏摘要；不得展示真实姓名、学号、校园邮箱、手机号等敏感信息。后端详情接口不得直接返回 `UserProfile` 全量字段，联系方式交换应留给 P0-4 “低压力联系”模块处理。

需求详情页可以展示附件图片原图，但仅限经过审核且随 `PUBLISHED` 需求公开的附件。附件访问仍需通过后端权限校验和后端中转，不直接暴露对象存储长期公开地址。详情接口可以返回附件引用或受控访问入口，不返回对象存储裸 URL；未登录用户、非 `PUBLISHED` 状态需求或无权限场景不得访问附件。

广场列表项可以展示第一张附件缩略图，但不直接加载附件原图。缩略图仅来自已审核通过且随 `PUBLISHED` 需求公开的附件，并通过后端受控访问入口加载；无附件时使用统一占位样式。该规则用于提高列表浏览效率，同时避免滚动加载时大量加载原图。

广场列表项不展示完整正文描述。`PostListItemView` 展示正文摘要，默认取正文描述前 30 个中文字符或等效长度字符；完整正文仅在 `PostDetailView` 中展示。正文摘要用于提升列表浏览的信息密度，避免每个列表项过于单薄，同时控制滚动列表高度和阅读负担。

广场列表不提供“只看自己发布的需求”筛选。用户自己的 `PUBLISHED` 需求如果出现在广场列表中，可以像其他需求一样被点开查看详情；但编辑、下架、重新提交审核等管理动作不在广场筛选中承载，应通过“我的发布”或发布管理入口处理。

用户不能在广场对自己发布的需求发起联系。自己的需求详情页可以正常查看，但联系入口应禁用或显示“这是你发布的需求”等提示。P0-3 详情视图需要识别 `ownPost` 展示状态；P0-4 低压力联系接口后续也必须校验发起者不能等于发布者。

V1 不实现收藏/保存需求功能。P0-3 不设计收藏按钮、我的收藏列表、`FavoritePost` 对象或收藏接口；但广场列表项和详情视图应避免将收藏能力排除在布局和对象扩展之外，后续可通过新增 `FavoritePost` 或同类用户-需求关系对象扩展收藏能力。

### 4.7 P0-4 低压力联系

#### 4.7.1 模块定位

低压力联系模块承接用户基于已发布需求发起邀约或站内联系、建立会话、发送站内消息、交换联系方式卡片和联系方式解锁等能力。该模块负责从需求详情页产生受控互动，并在双方具备意愿后交换联系方式。

本模块不扩展为完整即时聊天系统。V1 站内消息作为轻量沟通载体存在，不设计在线状态、已读回执、输入中、实时推送等完整 IM 能力；消息传输方式延续既有技术约束，V1 采用 REST 轮询，并为后续 WebSocket 演进预留边界。

第一轮核心对象包括：

- `Conversation`
- `Message`
- `ContactCard`
- `ContactUnlockRecord`
- `PartnerPost`

#### 4.7.2 参与角色与前置条件

发起低压力联系时，发起者必须处于 `VERIFIED` 状态，需求发布者在发起时也必须仍处于 `VERIFIED` 状态。若发布者因校园邮箱更换、重新认证、治理限制等原因不再具备已认证学生能力，则不得基于其历史发布发起新的联系。

发起联系还需要满足以下前置条件：目标需求处于 `PUBLISHED` 状态；发起者不能等于发布者；发起者与发布者均未处于阻断该互动的治理限制状态。后端发起联系接口必须执行上述校验，客户端禁用入口不能替代后端校验。

发起联系成功后，系统立即创建一个绑定目标 `PartnerPost` 的 `Conversation`，并发送一条初始邀约消息。该会话参与者为发起者和需求发布者；初始邀约消息用于表达联系意图。联系方式卡片在会话创建时保持未解锁状态，后续需要按联系方式解锁规则处理。

同一两个用户之间只能存在一个有效 `Conversation`。用户基于某个 `PartnerPost` 发起联系时，系统应先按参与双方查找是否已有有效会话；若已有会话，则不重复创建新会话，而是在既有会话中追加本次邀约上下文和初始邀约消息，保留先前聊天记录。该规则不按 `postId + initiatorId` 限制会话唯一性，而按用户双方关系限制有效会话唯一性。

P0-4 不单独设计 `Invitation` 对象。根据具体 `PartnerPost` 发起的邀约上下文直接由 `Conversation` 及其消息记录共同维护，不再要求划分每一条消息归属于哪一次邀约。后续如同一双方因新的需求再次沟通，系统复用既有会话并追加新的初始邀约消息即可。

初始邀约消息允许用户完全自由输入，不采用固定系统模板或“模板 + 附言”模式。为控制消息质量和界面展示，初始邀约消息与普通站内文本消息统一采用短消息约束：消息去除首尾空白后必须非空，长度上限为 30 个中文字符或等效长度字符。V1 不引入图片、附件、表情等非纯文本消息类型。

#### 4.7.3 消息边界

P0-4 的站内消息定位为低压力联系入口，不承担完整即时通讯系统职责。消息内容以简短文本为主，初始邀约消息和普通站内文本消息共用同一套非空校验、长度校验和服务端约束。Qt 客户端应在输入框中给出剩余字数或超限提示；后端接口必须对空消息、全空白消息和超长消息进行拒绝，客户端校验不能替代后端校验。

`Conversation` 设置 `ACTIVE` 与 `CLOSED` 两种业务状态。会话创建后默认为 `ACTIVE`；任一参与方可以主动关闭会话。`CLOSED` 状态下不允许继续发送普通消息或解锁联系方式，但历史消息保留，用于用户回看和必要治理。若同一双方后续因新的 `PartnerPost` 再次发起联系，系统复用既有会话并通过新的初始邀约消息重新激活为 `ACTIVE`，不得在没有新需求和新初始消息的情况下静默恢复会话。

联系方式卡片默认处于锁定状态。只有当双方都在同一 `ACTIVE` 会话中明确确认愿意交换联系方式后，系统才解锁双方联系方式；单方确认不能查看对方联系方式。联系方式解锁结果由 `ContactUnlockRecord` 记录双方确认状态和最终解锁时间，后端返回联系方式前必须校验会话状态、参与关系和双方确认结果。

V1 的 `ContactCard` 仅支持用户主动填写并确认可交换的微信号、手机号和 QQ 号。用户可以填写其中一种或多种，但至少需要维护一种可交换联系方式。校园邮箱不作为联系方式卡片内容，其用途限定为账号注册、登录和校内身份认证，不默认进入社交交换链路。联系方式解锁后，系统只返回对方已维护且允许交换的联系方式项。

用户发起联系前必须已经维护至少一种有效的可交换联系方式，否则不得发起联系，应先进入联系方式维护流程。需求发布者可以在尚未维护联系方式的情况下发布需求并接收邀约、进行站内沟通，但在双方联系方式解锁前也必须补全联系方式卡片。联系方式解锁接口需要同时校验双方均存在有效 `ContactCard`。

联系方式解锁记录只保存双方已完成解锁的事实，不保存联系方式值的历史快照。用户后续修改联系方式后，已解锁会话中对方再次查看时应读取最新 `ContactCard` 内容；若用户删除到没有任何可交换联系方式，则已解锁会话中对方暂时看不到联系方式，并提示对方暂未维护可交换联系方式。该规则避免旧手机号、旧 QQ 号或旧微信号在系统中长期残留。

V1 不提供站内消息撤回或删除能力。`Message` 一经发送即作为会话历史保留，客户端不提供撤回、删除单条消息或删除服务端会话记录的入口。该约束用于保持会话上下文完整，并为后续投诉、申诉和管理员治理提供必要依据。

站内消息链路需要提供轻量举报入口。V1 按会话维度举报，不提供逐条消息举报；用户可在会话页发起“举报该会话”，选择举报原因并填写补充说明。P0-4 只负责提供入口和上下文挂接，系统应将会话、双方用户、关联需求和最近消息上下文提交给后续投诉/申诉模块处理，具体案件状态、证据补充和管理员处置流程归入 P1 “投诉申诉与案件”。

V1 站内消息采用 REST 轮询，并支持会话列表未读数和总未读数。系统需要记录每个用户在每个会话中的已读位置，例如 `lastReadMessageId` 或等价字段；用户进入会话详情后，将该会话中对方发送且尚未计入已读位置的消息标记为已读。该机制仅用于当前用户自己的未读提示，不向对方展示“已读”状态，不形成已读回执。

V1 站内消息轮询频率统一按 1 秒一次设计。Qt 客户端在会话列表和会话详情页均可按 1 秒间隔查询消息更新、会话未读数和总未读数；网络异常时应避免阻塞界面，并在恢复后重新拉取最新数据。该方案以桌面端接近即时的使用体验为优先，当前规模下不认为 1 秒轮询会造成不可接受的资源占用；后续如进入更高并发或移动端常驻场景，可在保持业务接口边界稳定的前提下演进到 WebSocket 或其他推送机制。

会话详情页消息查询采用增量拉取与历史分页结合的方式。进入会话时先加载最近一页历史消息，随后客户端维护当前最大 `messageId`，每 1 秒通过 `afterMessageId` 拉取新增消息；当用户向上滚动查看历史时，再按分页参数加载更早消息。消息查询接口需要支持最近一页、更早一页和某消息之后新增消息三类查询场景，避免轮询时重复拉取完整消息列表。

会话关闭后仍保留历史消息记录，但不再提供联系方式查看能力。`CLOSED` 表示一段联系关系已经终结，即使关闭前双方已经完成联系方式解锁，系统也不应在关闭后的会话中继续展示对方联系方式。关闭后不得发送新消息、不得发起新的联系方式解锁确认，也不得通过既有解锁记录继续访问联系方式；如同一双方后续因新的 `PartnerPost` 再次发起联系，需要通过新的初始邀约消息重新激活会话，并重新完成双方联系方式解锁确认。

关闭后的会话不能由任一方在旧会话中直接手动重新打开。会话重新激活必须通过新的 `PartnerPost` 邀约触发，并重新满足发起联系前置条件、联系方式卡片要求和初始邀约消息校验。为支持后续复联场景，邀约帖子可以在发布或发布后由发布者手动指定，定向推送给先前有过沟通记录的用户；该定向推送仅作为新需求邀约入口，不绕过对方是否接受沟通、会话状态和联系方式重新解锁规则。

定向推送的接收方范围限定为与发布者存在历史 `Conversation` 的用户。V1 不允许发布者按学号、手机号、邮箱、昵称或其他个人标识搜索任意用户进行定向推送。后端需要校验发布者与接收方存在历史会话记录，且双方当前不存在阻断互动的治理限制；通知模块后续负责承接定向推送通知。

P0-4 需要支持用户拉黑功能。任一用户拉黑对方后，系统应立即阻断双方后续互动：拉黑方不再收到被拉黑方发送的站内消息、定向推送或新的邀约入口；被拉黑方也不得继续查看拉黑方在该关系链路中的消息内容、联系方式或后续定向推送信息。拉黑关系应作为会话、发起联系、消息发送、联系方式查看和定向推送接口的统一前置校验条件。

用户可以主动解除拉黑，但解除拉黑只移除阻断关系，不自动恢复旧会话互动。若旧会话已关闭，则仍保持 `CLOSED`，历史联系方式仍不可见；双方后续如需再次沟通，仍需通过新的 `PartnerPost` 邀约、发送新的初始邀约消息，并重新完成联系方式解锁确认。解除拉黑不应自动通知对方，也不应自动恢复会话、消息可见性或联系方式可见性。

拉黑后，该关系链路中的历史会话消息在双方普通客户端视图中均不可见。客户端会话列表不再展示该会话，普通会话详情接口也不得向任一方返回历史消息内容；但服务端仍保留消息记录，用于投诉、申诉和管理员治理。解除拉黑后，也不自动恢复历史消息可见性；若双方后续通过新的 `PartnerPost` 邀约重新建立联系关系，再按新的会话激活和消息展示规则处理。

拉黑动作触发后，当前双方 `Conversation` 应立即进入 `CLOSED`，并记录关闭原因或阻断来源为 `BLOCKED`。会话业务状态仍只保留 `ACTIVE` 与 `CLOSED`，不额外增加 `BLOCKED` 状态；拉黑语义由 `BlockRelation` 或等价黑名单关系对象表达。发消息、联系方式解锁、定向推送和重新激活会话时，应同时校验 `Conversation.status` 与拉黑关系。

低压力联系模块第一轮正文补全已收束。当前已确认模块边界、参与角色、发起联系前置条件、会话唯一性、会话关闭与重新激活、短文本消息规则、联系方式卡片与双方解锁机制、关闭后联系方式可见性、定向推送范围、会话举报、未读数、REST 轮询频率、增量消息拉取、拉黑与解除拉黑规则。后续若评价信用、投诉申诉、管理端治理或接口契约发现冲突，再回到本模块补充修订。

第一轮核心对象和设计元素包括：

- `PartnerPost`
- `PostSearchCriteria`
- `PostSortStrategy`
- `PostListItemView`
- `PostDetailView`

### 4.8 P1-1 评价与信用摘要

#### 4.8.1 模块定位

评价与信用摘要模块承接用户在发生站内联系后的主观反馈、星级评价、信用摘要展示和信用分值沉淀。该模块不负责判断线下活动是否真实完成，也不强制要求用户评价；评价是用户主动发起的补充反馈能力，信用摘要则需要在用户不评价时仍能形成稳定、温和的默认统计口径。

本模块上游依赖低压力联系模块中的 `Conversation`、`ContactUnlockRecord` 和参与双方关系；下游为投诉申诉、管理端治理、通知与留痕提供评价记录、信用摘要、分值变化和争议对象。

第一轮核心对象包括：

- `Review`
- `CreditSummary`
- `CreditChangeLog`
- `Conversation`
- `ContactUnlockRecord`

#### 4.8.2 评价触发与任务边界

V1 不采用“会话关闭后自动生成评价任务”的规则。搭子活动或沟通结束后，双方很可能不会主动关闭对话；如果评价入口依赖会话关闭，互评链路会难以展开。因此，V1 将评价定位为用户主动发起的额外功能。

用户可以在与对方存在有效 `Conversation` 后主动发起评价。系统不强制双方互评，不要求双方都提交评价，也不要求会话必须关闭。评价入口可以出现在会话详情、历史联系记录或用户信用卡片相关入口中，但后端必须校验评价者与被评价者之间存在可评价的会话关系。

有效对话定义为双方在同一 `Conversation` 中各自至少发送过 2 条用户消息。仅创建会话、只有初始邀约、只有单方连续消息、系统消息或纯状态变化都不计入有效对话。只有达到有效对话门槛后，该会话才可以计入默认 4 星、总对话次数和主动评价入口。

拉黑导致关闭的会话不应鼓励继续评价。若双方已进入拉黑关系，普通客户端不提供新的评价入口；如已经存在评价争议，后续由投诉申诉与管理端治理处理。

#### 4.8.3 星级与信用摘要计算口径

信用摘要以“对话平均星数”为核心展示口径。每个用户的评价分按照其参与过的对话记录进行平均计算，而不是只统计主动填写评价的记录。该规则用于避免“没有人评价就没有信用数据”的冷启动问题。

V1 采用以下计分规则：

| 对话类型 | 默认或可评分上限 | 说明 |
|---|---|---|
| 无显式评价的有效对话 | 默认 4 星 | 双方没有主动评价时，该对话按 4 星计入信用摘要 |
| 普通显式评价 | 最高 5 星 | 存在有效会话但未完成联系方式交换时，评价者最高可给 5 星 |
| 已交换联系方式后的显式评价 | 最高 6 星 | 同一会话中双方完成联系方式解锁后，评价者最高可给 6 星 |

新账号初始化时不以 0 次对话、0 分开始，而是带有平台默认的虚拟评分基线：默认视为已经存在 3 次 3 星评分样本和 3 次 4 星评分样本。初始信用摘要因此从 6 个虚拟评分样本、平均 3.5 星开始，后续随着真实有效对话次数和评价质量逐步变化。

用户基本信息信用卡片中需要展示总对话次数。总对话次数只统计真实有效对话，不包含初始的 3 个 3 星虚拟评分样本和 3 个 4 星虚拟评分样本。该字段用于让其他用户理解信用摘要的数据厚度：同样的平均星数下，真实有效对话次数更多的用户可信度更稳定。信用卡片至少展示：

- 平均星数。
- 总对话次数。
- 是否包含平台默认初始评分基线的说明或弱提示。

显式评价按评价者给出的星数计入被评价者的信用摘要。若某个真实有效对话没有显式评价，则按默认 4 星计入双方的信用摘要。若一方评价另一方，而另一方没有评价回来，则有评价的一侧按显式星数计入被评价者，未评价的一侧仍按默认 4 星计入对方。

#### 4.8.4 评价字段与标签展示

P1 V1 的评价字段采用“星级必填 + 预设标签可选”的结构，不开放自由文本短评。

`Review` 至少包含：

| 字段 | 含义 | 说明 |
|---|---|---|
| `id` | 评价 ID | 系统生成 |
| `conversationId` | 关联会话 ID | 评价必须绑定真实有效对话 |
| `reviewerId` | 评价者 ID | 当前登录用户 |
| `revieweeId` | 被评价者 ID | 会话另一方 |
| `rating` | 星级 | 必填；普通评价最高 5 星，联系方式解锁后最高 6 星 |
| `reviewTags` | 评价标签 | 可选；只能从预设标签中选择 |
| `createdAt` | 创建时间 | 评价提交时间 |
| `updatedAt` | 更新时间 | 后续如允许修改评价时使用 |

V1 暂不支持自由短评，原因是自由文本会引入内容审核、辱骂、人身攻击、隐私泄露和申诉成本。预设标签足以支撑信用摘要解释、统计展示和基础治理。

预设标签分为正向标签和负向标签，第一版候选如下：

| 标签类型 | 标签候选 |
|---|---|
| 正向标签 | `守时`、`沟通顺畅`、`配合度高`、`认真负责`、`体验很好` |
| 负向标签 | `迟到`、`失联`、`临时变卦`、`配合度低`、`体验不佳` |

用户信用卡片中可以展示被贴过的高频标签，格式采用“标签名*次数”。例如有 3 个不同评价者给某用户贴过 `配合度高` 标签，则信用卡片展示为 `配合度高*3`。标签次数只统计显式评价中的标签，不从默认 4 星对话中推导标签。

标签展示应优先展示次数较高的标签；同次数时可按最近出现时间或固定标签顺序排序。V1 信用卡片最多展示 5 个标签，避免标签区过长影响页面密度。后续如需要区分正向/负向标签展示，可在信用摘要视图中分组呈现，但当前先保留统一标签列表。

#### 4.8.5 评价修改规则

同一会话中，评价者对同一被评价者最多只能保留一条有效 `Review`。评价提交后允许在 24 小时内修改一次，修改内容可以包括星级和标签；超过 24 小时后不得再修改；已经修改过一次后也不得再次修改。

`Review` 需要记录修改控制字段：

| 字段 | 含义 | 说明 |
|---|---|---|
| `updatedAt` | 最近修改时间 | 初次提交时可与 `createdAt` 相同 |
| `modifiedOnce` | 是否已修改过 | 初次提交为 `false`，修改成功后为 `true` |

评价修改成功后，信用摘要不能简单叠加新评分和新标签。系统必须撤销或覆盖旧评价对被评价者平均星数和标签计数的影响，再按新评价重新计算。实现上可以采用实时重算，也可以维护 `CreditChangeLog` 并在更新时回滚旧记录；但对外结果必须等价于“同一会话只保留最新有效评价”。

评价修改失败场景统一纳入异常分支：

| 场景 | 错误码建议 | 说明 |
|---|---|---|
| 非评价者本人修改 | `FORBIDDEN` | 只能修改自己提交的评价 |
| 评价不存在 | `REVIEW_NOT_FOUND` | 当前用户无权感知时也可返回该错误或不可感知结果 |
| 超过 24 小时 | `REVIEW_MODIFICATION_EXPIRED` | 修改期限已过 |
| 已修改过一次 | `REVIEW_MODIFICATION_LIMIT_EXCEEDED` | 不允许重复修改 |
| 新星级或标签非法 | `VALIDATION_FAILED` | 通过 `details` 返回字段级错误 |

#### 4.8.6 评价争议与申诉挂接

评价争议不在评价与信用摘要模块内直接裁定。普通用户不能自行删除他人评价、屏蔽标签或移除评价对信用摘要的影响；争议通过 P1 “投诉申诉与案件”模块挂接处理，管理员裁定后再影响评价状态和信用摘要。

`Review` 需要具备争议状态：

| 状态 | 含义 | 说明 |
|---|---|---|
| `ACTIVE` | 生效中 | 默认状态，计入信用摘要和标签统计 |
| `DISPUTED` | 争议中 | 被评价方已发起申诉，管理员尚未裁定；仍暂时计入信用摘要 |
| `INVALIDATED` | 已失效 | 管理员裁定该评价不应继续影响信用摘要 |

用户可针对收到的某条 `Review` 发起申诉，案件挂接关系为：

| 字段 | 值 |
|---|---|
| `targetType` | `REVIEW` |
| `targetId` | `reviewId` |
| `complainantId` | 被评价方用户 ID |
| `respondentId` | 评价者用户 ID |

发起申诉后，`Review.status` 从 `ACTIVE` 进入 `DISPUTED`。`DISPUTED` 状态下，该评价仍暂时计入信用摘要，避免用户通过大量申诉立即抹除负面评价。信用卡片或评价详情可弱提示“部分评价存在争议”，但 V1 不要求在普通信用卡片中展开案件详情。

管理员处理评价争议后，可执行以下裁定结果：

| 裁定结果 | 对评价的影响 | 对信用摘要的影响 |
|---|---|---|
| 维持评价 | `DISPUTED -> ACTIVE` | 继续按原评价计入 |
| 隐藏评价标签 | `DISPUTED -> ACTIVE`，但移除或隐藏部分标签 | 星级继续计入，相关标签不再展示或计数 |
| 撤销评价影响 | `DISPUTED -> INVALIDATED` | 星级和标签均不再计入信用摘要 |
| 标记恶意评价 | `DISPUTED -> INVALIDATED`，并产生治理线索 | 星级和标签均不再计入，可交由管理端治理处理评价者 |

评价被裁定为 `INVALIDATED` 后，系统必须重新计算被评价者的平均星数和标签计数。被撤销的评价不计入平均星数、不计入标签次数，也不计入显式评价数量；但关联案件、原评价内容和管理员裁定记录应作为治理留痕保留，不从系统中物理删除。

#### 4.8.7 接口契约草案

评价与信用摘要接口按“评价操作接口”和“信用摘要查询接口”两组拆分。评价操作接口围绕当前登录用户对有效对话和对方的评价动作；信用摘要查询接口围绕用户公开信用卡片和本人信用详情读取。

评价操作接口草案如下：

| 接口 | 用途 | 认证要求 | 关键规则 |
|---|---|---|---|
| `POST /api/me/reviews` | 提交评价 | 登录且 `VERIFIED` | 只能评价与自己存在真实有效对话的对方；同一会话同一被评价者只能有一条有效评价 |
| `PUT /api/me/reviews/{reviewId}` | 修改评价 | 登录且为评价者本人 | 仅允许提交后 24 小时内修改一次 |
| `GET /api/me/reviews/given` | 查看我发出的评价 | 登录用户 | 分页返回当前用户提交的评价 |
| `GET /api/me/reviews/received` | 查看我收到的评价 | 登录用户 | 分页返回他人对当前用户的评价，可显示争议状态 |

信用摘要查询接口草案如下：

| 接口 | 用途 | 认证要求 | 关键规则 |
|---|---|---|---|
| `GET /api/users/{userId}/credit-summary` | 获取指定用户公开信用摘要 | 登录用户 | 返回平均星数、真实有效总对话次数、高频标签，不返回评价者敏感信息 |
| `GET /api/me/credit-summary` | 获取我的信用摘要 | 登录用户 | 可返回比公开摘要更多的本人统计信息，如收到评价数量、争议中评价数量 |

评价提交请求 `CreateReviewRequest` 至少包含：

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `conversationId` | string | 是 | 评价必须绑定有效对话 |
| `revieweeId` | string | 是 | 被评价者必须是会话另一方 |
| `rating` | number | 是 | 普通评价最高 5 星，联系方式解锁后最高 6 星 |
| `reviewTags` | string[] | 否 | 预设标签集合，不支持自由短评 |

评价修改请求 `UpdateReviewRequest` 至少包含：

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `rating` | number | 是 | 修改后的星级 |
| `reviewTags` | string[] | 否 | 修改后的标签集合，以新集合覆盖旧集合 |

公开信用摘要响应 `CreditSummaryResponse` 至少包含：

| 字段 | 类型 | 说明 |
|---|---|---|
| `userId` | string | 用户 ID |
| `averageRating` | number | 平均星数，包含初始虚拟评分基线和真实有效对话评分 |
| `realConversationCount` | integer | 真实有效对话次数，不包含初始虚拟评分基线 |
| `ratingSampleCount` | integer | 参与平均星数计算的样本数，包含虚拟评分样本和真实有效对话样本 |
| `topTags` | object[] | 高频标签，按 `tagName` 和 `count` 返回，客户端展示为 `标签名*次数` |
| `disputedReviewCount` | integer | 争议中评价数量，可用于弱提示 |
| `updatedAt` | datetime | 信用摘要最近更新时间 |

两组接口统一使用项目错误响应结构 `code`、`message`、`details`、`traceId`。P1-1 至少需要约定以下错误码：

| 错误码 | 适用场景 |
|---|---|
| `UNAUTHORIZED` | 未登录访问需要登录的接口 |
| `FORBIDDEN` | 非评价者本人修改评价，或无权查看相关评价 |
| `AUTHENTICATION_STATUS_REQUIRED` | 当前用户不是 `VERIFIED`，不得提交评价 |
| `CONVERSATION_NOT_REVIEWABLE` | 会话不存在、用户不是参与者、未达到有效对话条件或处于拉黑阻断关系 |
| `REVIEW_ALREADY_EXISTS` | 同一会话同一评价者已对同一被评价者提交过评价 |
| `REVIEW_NOT_FOUND` | 评价不存在或当前用户无权感知 |
| `REVIEW_MODIFICATION_EXPIRED` | 评价修改超过 24 小时 |
| `REVIEW_MODIFICATION_LIMIT_EXCEEDED` | 评价已修改过一次 |
| `VALIDATION_FAILED` | 星级、标签或请求字段校验失败 |

Qt 客户端 API Client 层应分别封装 `ReviewApiService` 和 `CreditSummaryApiService`。会话详情、用户信用卡片和个人中心可以调用信用摘要查询接口；评价提交和修改入口只能调用评价操作接口。

#### 4.8.8 测试入口与后续实现批次

评价与信用摘要模块后续实现按 3 个批次推进，每批必须遵守测试先行、红灯确认、最小实现、绿灯确认、回归测试和测试留档规则。评价争议挂接虽然已在本模块定义边界，但完整案件流程归入后续 P1 “投诉申诉与案件”模块。

| 批次 | 范围 | 主要测试入口 | 阶段闭环 |
|---|---|---|---|
| Batch 1 | 后端评价与信用摘要核心 | JUnit 5 + Spring Boot 接口/安全测试 | 评价提交/修改、有效对话判定、默认 4 星、最高 5/6 星、标签聚合、信用摘要查询 |
| Batch 2 | Qt 信用卡片与评价入口 | Qt Test + 必要手工验证清单 | 信用卡片展示平均星数、真实总对话次数、高频标签；会话中可提交/修改评价 |
| Batch 3 | 评价争议挂接到投诉申诉 | JUnit 5 + Spring Boot 接口/安全测试 + 后续案件流程验证 | 评价可作为 `targetType=REVIEW` 的案件目标，`Review.status` 可进入 `DISPUTED/INVALIDATED` 并触发信用重算 |

Batch 1 的后端测试至少覆盖：

- 双方各自至少发送 2 条用户消息后，会话被识别为真实有效对话。
- 只有单方消息、双方各不足 2 条消息、系统消息或拉黑阻断关系时，提交评价返回 `CONVERSATION_NOT_REVIEWABLE`。
- 普通有效对话提交 1 到 5 星评价成功，提交 6 星返回 `VALIDATION_FAILED`。
- 已完成联系方式解锁的有效对话允许提交 1 到 6 星评价。
- 同一评价者不能对同一会话同一被评价者重复提交评价，重复提交返回 `REVIEW_ALREADY_EXISTS`。
- 评价提交后 24 小时内允许修改一次，超过期限或第二次修改被拒绝。
- 修改评价后，平均星数和标签计数按最新有效评价重算。
- 无显式评价的真实有效对话按默认 4 星计入信用摘要。
- 新账号信用摘要包含 3 个 3 星虚拟评分样本和 3 个 4 星虚拟评分样本，但 `realConversationCount` 为 0。
- 信用摘要高频标签按 `tagName/count` 返回，默认 4 星对话不产生标签。

Batch 2 的 Qt 客户端测试与验证至少覆盖：

- 用户信用卡片展示平均星数、真实总对话次数和 `标签名*次数`。
- 信用卡片总对话次数不包含虚拟评分基线。
- 会话满足有效对话条件后展示评价入口。
- 普通会话评价星级上限为 5，联系方式解锁后上限为 6。
- 评价提交或修改失败时，客户端能展示字段错误或状态错误。

Batch 3 的测试至少覆盖：

- 被评价方可针对收到的 `Review` 创建 `targetType=REVIEW` 的案件。
- 发起申诉后 `Review.status` 从 `ACTIVE` 进入 `DISPUTED`，且评价暂时仍计入信用摘要。
- 管理员裁定维持评价后，`DISPUTED -> ACTIVE`。
- 管理员裁定撤销评价影响后，`DISPUTED -> INVALIDATED`，平均星数和标签计数重算。
- 管理员裁定隐藏标签后，星级继续计入，相关标签不再展示或计数。

本模块后续测试留档应写入 `docs/validation/`，每批至少记录对应需求编号、设计入口、测试文件、红灯结果、实现摘要、绿灯结果、回归范围和未覆盖风险。

#### 4.8.9 第一轮待继续确认事项

本模块后续仍需继续确认：

- 信用摘要是否展示最近评价标签、分布或仅展示平均星数与总对话次数。

### 4.9 P1-2 投诉申诉与案件

#### 4.9.1 模块定位

投诉申诉与案件模块承接用户对评价、会话、需求、用户和管理处理结果产生争议时的统一案件入口。该模块负责创建案件、绑定争议目标、上传证据、记录双方说明、推动案件进入管理端处理，并为后续管理端治理、通知与留痕提供案件对象和状态来源。

本模块不直接执行最终治理裁定。管理员裁定、账号处置、信用干预和审计日志归入 P1 “管理端治理”模块；但本模块需要定义案件状态、案件目标、证据与回应边界，使管理端有稳定输入。

第一轮核心对象包括：

- `CaseTicket`
- `CaseEvidence`
- `CaseResponse`
- `Attachment`
- `Review`
- `Conversation`
- `PartnerPost`
- `User`

#### 4.9.2 统一案件对象

投诉、申诉、举报和处理结果争议统一由 `CaseTicket` 承载。V1 不为评价申诉、会话举报、需求举报、用户投诉和处理结果申诉分别建立独立案件对象，避免多套状态机和多套管理端队列。

`CaseTicket` 至少包含：

| 字段 | 含义 | 说明 |
|---|---|---|
| `id` | 案件 ID | 系统生成 |
| `caseType` | 案件类型 | 区分投诉、申诉、举报、处理结果争议 |
| `targetType` | 目标类型 | `REVIEW`、`CONVERSATION`、`PARTNER_POST`、`USER`、`ADMIN_DECISION` |
| `targetId` | 目标 ID | 指向具体业务对象 |
| `complainantId` | 发起人 ID | 发起投诉、申诉或举报的用户 |
| `respondentId` | 被投诉/被申诉关联方 ID | 可为空，按目标对象推导或后续补充 |
| `status` | 案件状态 | 后续状态机确认 |
| `reasonCode` | 原因类型 | 从预设原因中选择 |
| `description` | 补充说明 | 用户填写的案件说明 |
| `evidenceIds` | 证据附件 ID 集合 | 关联 `CaseEvidence` 或附件 |
| `createdAt` | 创建时间 | 案件创建时间 |
| `updatedAt` | 更新时间 | 案件状态或材料变化时间 |

`targetType` 第一版支持：

| `targetType` | 目标说明 | 典型来源 |
|---|---|---|
| `REVIEW` | 某条评价 | 被误评、恶意评价、标签争议 |
| `CONVERSATION` | 某个会话 | 骚扰、失联、沟通争议、会话举报 |
| `PARTNER_POST` | 某条搭子需求 | 虚假需求、违规内容、误导性信息 |
| `USER` | 某个用户 | 持续骚扰、冒充、恶意行为 |
| `ADMIN_DECISION` | 某个管理处理结果 | 用户不认可平台处理结果，发起申诉 |

统一 `CaseTicket` 使用 `targetType + targetId` 绑定业务对象。后端创建案件时必须校验发起人是否有权针对该目标创建案件，例如评价申诉必须是被评价方，会话举报必须是会话参与者，需求举报必须至少是登录用户，处理结果申诉必须与原处理结果有关。

案件证据通过 `CaseEvidence` 或等价证据对象承载，附件存储沿用项目附件规则。证据附件不得直接暴露对象存储裸地址，管理端读取证据时也必须经过后端权限校验。

#### 4.9.3 案件状态机

`CaseTicket.status` 采用三态模型：

| 状态 | 含义 | 说明 |
|---|---|---|
| `SUBMITTED` | 提交态 | 用户已提交案件，案件进入管理端待查看队列 |
| `WAITING_FOR_UPDATE` | 待回应态 | 管理员查看后认为需要发起人补充证据，或需要对方回应、更新说明 |
| `CLOSED` | 完成态 | 管理员确认案件已处理完成，并已做好相应裁定或处置 |

三态流转规则如下：

| 起始状态 | 触发动作 | 目标状态 | 说明 |
|---|---|---|---|
| 无记录 | 用户创建案件 | `SUBMITTED` | 进入管理端待查看队列 |
| `SUBMITTED` | 管理员要求补充证据或对方回应 | `WAITING_FOR_UPDATE` | 通过原因字段区分需要谁更新、更新什么 |
| `SUBMITTED` | 管理员直接裁定完成 | `CLOSED` | 材料足够时可直接完成 |
| `WAITING_FOR_UPDATE` | 发起人或对方补充材料/回应 | `SUBMITTED` | 更新后重新进入管理端待查看队列 |
| `WAITING_FOR_UPDATE` | 管理员认为无需继续等待并裁定 | `CLOSED` | 例如逾期未回应或已有足够材料 |

`WAITING_FOR_UPDATE` 不再拆分为“待补证据”和“待对方回应”两个独立状态。系统通过辅助字段表达待更新原因和责任方：

| 字段 | 含义 | 示例 |
|---|---|---|
| `waitingForRole` | 当前等待谁更新 | `COMPLAINANT`、`RESPONDENT`、`BOTH` |
| `updateReason` | 需要更新的原因 | `MORE_EVIDENCE_REQUIRED`、`RESPONSE_REQUIRED`、`CLARIFICATION_REQUIRED` |
| `updateNote` | 管理员说明 | 说明需要补什么材料或回应什么问题 |

完成态 `CLOSED` 必须带有裁定结果或处理说明。具体处理动作，例如维持评价、撤销评价影响、隐藏评价标签、警告、禁言、封禁、驳回申诉等，不作为 `CaseTicket.status`，而作为管理端治理模块中的裁定结果和操作留痕记录。

#### 4.9.4 创建权限与恶意举报治理

不同 `targetType` 的案件创建权限按目标关系控制：

| `targetType` | 谁可以创建 | 说明 |
|---|---|---|
| `REVIEW` | 被评价方 | 只能对自己收到的评价发起申诉 |
| `CONVERSATION` | 会话参与者 | 只能举报自己参与过的会话 |
| `PARTNER_POST` | 登录用户 | 仅针对已发布或可被当前用户感知的需求 |
| `USER` | 登录用户 | 必须填写原因和说明，用于投诉持续骚扰、冒充、恶意行为等 |
| `ADMIN_DECISION` | 原案件相关方 | 只能对与自己相关的处理结果发起申诉 |

后端创建案件时必须校验目标对象可感知性和发起人与目标的关系。无权创建案件时返回 `FORBIDDEN` 或不可感知的 `CASE_TARGET_NOT_FOUND`；目标对象状态不允许创建案件时返回 `CASE_TARGET_NOT_REPORTABLE`。

举报与投诉入口开放不代表允许滥用。若管理员认定某个案件属于恶意举报或恶意申诉，例如捏造证据、伪造材料、故意歪曲上下文、组织多人针对同一用户批量举报、反复提交明显无事实依据的案件等，系统应将该行为作为治理线索记录，并视情况对违规账号做出处罚。

恶意举报治理归入 P1 “管理端治理”模块，但本模块需要保留必要挂接点：

| 字段 / 关系 | 含义 |
|---|---|
| `maliciousReportFlag` | 管理员是否标记为恶意举报或恶意申诉 |
| `maliciousReportReason` | 恶意举报认定原因 |
| `relatedReporterIds` | 批量恶意举报中涉及的举报人集合，可由管理端分析或人工补充 |
| `governanceActionRef` | 后续账号处罚、警告或限制的治理记录引用 |

对于被认定为恶意举报的账号，管理端可按情节轻重执行提醒、警告、临时禁言、限制举报频率、封禁等处置。具体处罚梯度、处置权限和审计日志在 P1 “管理端治理”模块中继续细化。

#### 4.9.5 证据附件规则

案件证据附件沿用项目投诉证据附件规则，并补充案件级数量限制：

| 项目 | 规则 |
|---|---|
| 支持类型 | `jpg`、`jpeg`、`png`、`pdf` |
| 单文件大小 | 不超过 10 MB |
| 数量上限 | 每个 `CaseTicket` 最多 10 个证据附件 |
| 上传方式 | 后端中转上传对象存储 |
| 访问方式 | 后端业务权限校验，不暴露对象存储裸 URL |

用户创建案件时可以上传 0 到 10 个证据附件。某些案件允许先提交说明、后补证据；管理员认为证据不足时，可将案件状态从 `SUBMITTED` 调整为 `WAITING_FOR_UPDATE`，并设置 `waitingForRole = COMPLAINANT`、`updateReason = MORE_EVIDENCE_REQUIRED`。发起人补充证据后，案件回到 `SUBMITTED`，重新进入管理端待查看队列。

补充证据不得突破每个案件 10 个附件的总上限。若用户需要提交更多材料，应优先合并为 PDF 后上传。证据附件不得包含可执行文件、压缩包、脚本文件或其他高风险格式；文件校验不得只检查扩展名，也需要校验 MIME 类型。

`CaseEvidence` 至少包含：

| 字段 | 含义 | 说明 |
|---|---|---|
| `id` | 证据 ID | 系统生成 |
| `caseId` | 所属案件 ID | 指向 `CaseTicket` |
| `uploaderId` | 上传人 ID | 发起人、回应方或管理员 |
| `attachmentId` | 附件 ID | 指向统一附件元数据 |
| `description` | 证据说明 | 可选，说明该证据证明什么 |
| `createdAt` | 上传时间 | 用于案件时间线 |

管理端查看证据时应展示附件摘要、上传人、上传时间和证据说明。普通用户只能查看自己上传的证据和案件中允许其查看的对方回应材料；证据可见性细节后续在接口契约中继续确认。

#### 4.9.6 回应规则与期限

被投诉方或被申诉关联方不强制必须回应。管理员认为需要对方回应、发起人补充证据或双方补充说明时，可将案件状态置为 `WAITING_FOR_UPDATE`，并设置 `waitingForRole`、`updateReason`、`updateNote` 和 `responseDueAt`。

回应窗口统一为 72 小时：

| 场景 | `waitingForRole` | 说明 |
|---|---|---|
| 需要发起人补充证据 | `COMPLAINANT` | 发起人可追加证据或补充说明 |
| 需要被投诉方回应 | `RESPONDENT` | 被投诉方可提交 `CaseResponse` 和证据 |
| 需要双方补充 | `BOTH` | 双方均可提交补充材料 |

72 小时内未回应，不自动判定回应方违规，也不自动判定案件成立。管理员可以在到期后基于现有材料裁定，并可将“未回应”作为判断时的参考因素。该规则避免案件无限悬置，同时不把未回应本身等同为违规事实。

`CaseResponse` 至少包含：

| 字段 | 含义 | 说明 |
|---|---|---|
| `id` | 回应 ID | 系统生成 |
| `caseId` | 所属案件 ID | 指向 `CaseTicket` |
| `responderId` | 回应人 ID | 发起人、被投诉方或其他案件相关方 |
| `responseText` | 回应说明 | 说明事实、补充上下文或反驳 |
| `evidenceIds` | 回应证据 ID 集合 | 可为空，计入案件总证据附件上限 |
| `createdAt` | 回应时间 | 用于案件时间线 |

当 `WAITING_FOR_UPDATE` 状态下收到符合要求的补充证据或回应后，案件状态回到 `SUBMITTED`，重新进入管理端待查看队列。若只是部分责任方回应，管理员可以继续保持 `WAITING_FOR_UPDATE`，或在已有材料足够时直接裁定为 `CLOSED`。

通知模块后续需要承接 `WAITING_FOR_UPDATE` 的通知与到期提醒，但 P1-2 只定义业务字段和状态，不在本模块展开通知投递策略。

#### 4.9.7 裁定结果与业务回写边界

P1-2 案件模块只定义案件裁定结果类型和业务对象回写边界；具体账号处罚、信用人工干预、审计日志和治理权限归入 P1 “管理端治理”模块继续细化。

`CaseTicket` 完成态需要记录裁定结果：

| 字段 | 含义 | 说明 |
|---|---|---|
| `decisionResult` | 裁定结果 | 管理员对案件的结论 |
| `decisionNote` | 裁定说明 | 面向案件相关方展示的处理说明 |
| `decidedBy` | 裁定管理员 ID | 用于管理留痕 |
| `decidedAt` | 裁定时间 | 案件进入 `CLOSED` 的时间 |

V1 案件裁定结果类型如下：

| 裁定结果 | 含义 | 典型回写 |
|---|---|---|
| `REJECT_CASE` | 驳回案件 | 案件关闭，不改变目标对象 |
| `KEEP_ORIGINAL` | 维持原对象或原处理 | 案件关闭，目标对象保持原状态 |
| `INVALIDATE_REVIEW` | 撤销评价影响 | `Review.status -> INVALIDATED`，信用摘要重算 |
| `HIDE_REVIEW_TAGS` | 隐藏评价标签 | 星级继续计入，相关标签不再展示或计数 |
| `REMOVE_POST` | 下架需求 | 目标 `PartnerPost` 不再进入公开展示范围，具体状态回写后续与发布模块对齐 |
| `CLOSE_CONVERSATION` | 关闭会话 | 目标 `Conversation.status -> CLOSED` |
| `MARK_MALICIOUS` | 标记恶意举报或恶意申诉 | 设置 `maliciousReportFlag`，形成治理线索 |
| `ESCALATE_GOVERNANCE` | 转入账号治理 | 案件关闭或保持关联，后续由管理端治理执行账号处置 |

裁定结果可以组合使用，但 V1 后端接口可先采用单一主裁定结果 + 可选治理升级标记，避免一次案件处理携带过多复合动作。若后续管理端需要同时撤销评价、标记恶意评价并处罚账号，应由管理端治理模块通过多个治理操作记录表达，而不是把所有动作塞进 `CaseTicket.status`。

案件进入 `CLOSED` 后，普通用户不得继续补充证据或回应。若用户不认可处理结果，应基于原管理处理结果创建 `targetType = ADMIN_DECISION` 的新申诉案件，而不是重新打开旧案件。

#### 4.9.8 接口契约草案

投诉申诉与案件接口按“用户侧案件接口”和“管理侧案件接口”两组拆分。用户侧接口负责创建案件、补充证据、提交回应和查看进度；管理侧接口负责查看案件队列、查看详情、要求补充或回应、裁定结案。

用户侧接口草案如下：

| 接口 | 用途 | 认证要求 | 关键规则 |
|---|---|---|---|
| `POST /api/me/cases` | 创建案件 | 登录用户 | 校验 `targetType + targetId` 可感知性和创建权限 |
| `POST /api/me/cases/{caseId}/evidence` | 补充证据 | 案件相关方 | 仅允许在非 `CLOSED` 状态补充，总附件数不超过 10 |
| `POST /api/me/cases/{caseId}/responses` | 提交回应 | 案件相关方 | 通常用于 `WAITING_FOR_UPDATE`，但非 `CLOSED` 状态也可补充说明 |
| `GET /api/me/cases` | 查看我的案件列表 | 登录用户 | 返回我发起或与我相关的案件 |
| `GET /api/me/cases/{caseId}` | 查看我的案件详情 | 案件相关方 | 返回案件状态、时间线、本人可见证据和回应 |

管理侧接口草案如下：

| 接口 | 用途 | 认证要求 | 关键规则 |
|---|---|---|---|
| `GET /api/admin/cases` | 查看案件队列 | `ROLE_ADMIN` | 支持按状态、目标类型、案件类型筛选 |
| `GET /api/admin/cases/{caseId}` | 查看案件详情 | `ROLE_ADMIN` | 返回完整案件材料、双方回应、证据摘要和目标对象摘要 |
| `POST /api/admin/cases/{caseId}/request-update` | 要求补充或回应 | `ROLE_ADMIN` | 将案件置为 `WAITING_FOR_UPDATE`，设置等待对象、原因、说明和 72 小时期限 |
| `POST /api/admin/cases/{caseId}/decide` | 裁定结案 | `ROLE_ADMIN` | 设置 `decisionResult`，案件进入 `CLOSED`，触发必要业务回写 |

`CreateCaseRequest` 至少包含：

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `caseType` | string | 是 | 投诉、申诉、举报或处理结果争议 |
| `targetType` | string | 是 | `REVIEW`、`CONVERSATION`、`PARTNER_POST`、`USER`、`ADMIN_DECISION` |
| `targetId` | string | 是 | 目标对象 ID |
| `reasonCode` | string | 是 | 预设原因 |
| `description` | string | 是 | 补充说明 |
| `evidenceIds` | string[] | 否 | 创建时附带的证据附件 ID，最多 10 个 |

`AddCaseEvidenceRequest` 至少包含：

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `attachmentId` | string | 是 | 已上传证据附件 ID |
| `description` | string | 否 | 证据说明 |

`CreateCaseResponseRequest` 至少包含：

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `responseText` | string | 是 | 回应说明 |
| `evidenceIds` | string[] | 否 | 回应附带证据，计入案件 10 个附件上限 |

`RequestCaseUpdateRequest` 至少包含：

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `waitingForRole` | string | 是 | `COMPLAINANT`、`RESPONDENT`、`BOTH` |
| `updateReason` | string | 是 | `MORE_EVIDENCE_REQUIRED`、`RESPONSE_REQUIRED`、`CLARIFICATION_REQUIRED` |
| `updateNote` | string | 是 | 管理员说明 |

`DecideCaseRequest` 至少包含：

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `decisionResult` | string | 是 | 案件裁定结果 |
| `decisionNote` | string | 是 | 裁定说明 |
| `escalateGovernance` | boolean | 否 | 是否转入账号治理 |

两组接口统一使用项目错误响应结构 `code`、`message`、`details`、`traceId`。P1-2 至少需要约定以下错误码：

| 错误码 | 适用场景 |
|---|---|
| `UNAUTHORIZED` | 未登录访问需要登录的接口 |
| `FORBIDDEN` | 当前用户无权创建、查看、补充或处理该案件 |
| `CASE_TARGET_NOT_FOUND` | 目标对象不存在或当前用户不可感知 |
| `CASE_TARGET_NOT_REPORTABLE` | 目标对象状态不允许创建案件 |
| `CASE_NOT_FOUND` | 案件不存在或当前用户无权感知 |
| `CASE_STATUS_CONFLICT` | 当前案件状态不允许该操作，例如关闭后继续补充 |
| `CASE_EVIDENCE_LIMIT_EXCEEDED` | 案件证据附件超过 10 个 |
| `VALIDATION_FAILED` | 请求字段、原因码、裁定结果或附件格式校验失败 |

Qt 客户端 API Client 层应分别封装 `CaseApiService` 和 `AdminCaseApiService`。学生端案件页不得调用管理侧接口；管理端案件队列不应复用用户侧补证据或回应接口执行裁定。

### 4.10 P1-4 通知与留痕

#### 4.10.1 模块定位

通知与留痕模块承接平台内关键状态变化后的用户可见提醒和系统事件记录。V1 只做“站内通知 + 系统事件留痕”，不做实时推送、短信、邮件、移动端 Push 或 WebSocket 通知。该模块的目标是让用户在打开 Qt 客户端后能看到与自己相关的审核结果、评价提醒、案件进度、治理结果和信用变化，同时让系统能够用结构化事件记录串联后续排查、测试留档和通知触发。

本模块不替代管理端审计日志。`AdminAuditLog` 面向管理员关键操作追责；`SystemEventLog` 面向系统业务事件留痕；`Notification` 面向用户可见提醒。三者可以关联同一业务来源，但职责不能混用。

V1 发送方式采用站内通知，由 Qt 客户端通过 REST 轮询或进入通知中心时拉取。通知模块不引入离线触达能力；用户未打开客户端时，不保证实时看到提醒。

第一轮核心对象包括：

- `Notification`
- `SystemEventLog`

#### 4.10.2 边界与非目标

V1 通知与留痕模块明确不做以下能力：

- 不做短信通知。
- 不做邮件通知。
- 不做移动端 Push。
- 不做 WebSocket 实时推送。
- 不做复杂消息队列投递保证。
- 不做营销、运营或批量群发通知。

该边界与当前 Win11 Qt 桌面端交付形态一致。系统可以在后续扩展中基于 `SystemEventLog` 或通知触发点接入 WebSocket、邮件、短信或消息队列，但 V1 不把这些基础设施作为详细设计和实现前置条件。

#### 4.10.3 `Notification` 与 `SystemEventLog` 职责划分

`Notification` 是面向用户的可见提醒。它回答“谁需要看到什么提醒、是否已读、点击后去哪里”。通知内容应可直接用于 Qt 通知中心、首页提醒角标或相关页面入口。

`SystemEventLog` 是面向系统的业务事件留痕。它回答“系统中发生了什么业务事件、由什么对象触发、影响了哪些对象、是否已派生通知”。它不等同于管理员审计日志，也不直接面向普通用户展示。

两者关系如下：

| 对象 | 面向对象 | 典型用途 | 是否用户可见 |
|---|---|---|---|
| `Notification` | 普通用户或管理员 | 审核结果、案件进度、评价提醒、治理结果、信用变化提醒 | 是 |
| `SystemEventLog` | 系统、测试、后续排查 | 记录业务事件、触发通知、串联状态变化 | 否 |
| `AdminAuditLog` | 管理端追责和复核 | 记录管理员关键操作、权限和原因 | 普通用户不可见 |

`Notification` 可以由 `SystemEventLog` 派生，也可以由业务服务在状态变化时直接创建；但同一类事件需要保持一致的触发策略，避免同一个状态变化有时写事件、有时直接写通知而无法追踪。

#### 4.10.4 通知触发点范围

V1 通知触发点先限定为 8 类，均围绕用户需要知道、可能需要处理或会影响信用/治理状态的主链路事件。不把点赞、浏览、推荐曝光、营销公告、批量广播等运营增强类消息纳入 P1-4。

通知触发点如下：

| 触发点 | 典型接收方 | 触发时机 | 说明 |
|---|---|---|---|
| 认证审核结果 | 认证申请人 | 认证资料通过或驳回 | 告知认证状态变化和驳回原因摘要 |
| 需求审核结果 | 需求发布者 | `PartnerPost` 审核通过或驳回 | 告知需求是否进入广场或需要修改 |
| 收到新会话/邀约 | 需求发布者或被邀约用户 | 新 `Conversation` 创建或新邀约上下文追加 | 告知有人发起低压力联系或基于新需求再次邀约 |
| 评价提醒 | 有效对话参与者 | 会话达到可评价条件或联系方式解锁后 | 提醒用户可以主动评价，不强制评价 |
| 评价争议处理结果 | 评价争议相关方 | 评价申诉案件裁定后 | 告知评价维持、标签隐藏、评价影响撤销等结果 |
| 案件状态变化 | 案件发起人、回应方或相关方 | 案件进入 `WAITING_FOR_UPDATE`、收到回应、结案等 | 告知补证据、回应、裁定结果和期限 |
| 治理动作生效/撤销 | 被治理用户 | `GovernanceAction` 创建或撤销 | 告知警告、禁言、封禁、限制举报、恢复账号等结果 |
| 信用摘要人工调整 | 被调整用户 | `ADJUST_CREDIT` 生效或撤销 | 告知信用影响摘要，例如评价影响撤销、标签隐藏、人工小幅调整 |

同一业务事件可以同时写入 `SystemEventLog` 并派生一条或多条 `Notification`。例如案件进入 `WAITING_FOR_UPDATE` 时，应写入系统事件，并分别通知需要补充材料的一方；治理动作生效时，应写入系统事件，通知被治理用户，若该动作来自案件也保留来源案件引用。

通知触发不得绕过业务权限。接收方必须是该事件的合法感知者，例如案件通知只能发给案件相关方，需求审核通知只能发给发布者，治理动作通知只能发给被治理用户或必要管理端接收方。

V1 不做以下通知：

- 浏览量、点赞、收藏、推荐曝光等轻互动提醒。
- 营销、活动、公告或批量广播。
- 站外触达提醒，例如短信、邮件、移动端 Push。
- 高频聊天消息逐条通知；站内消息未读数仍由低压力联系模块的会话未读机制承接。

#### 4.10.5 `Notification` 字段与状态规则

V1 的 `Notification` 只保留 `UNREAD` 与 `READ` 两种状态，不提供删除、归档、撤回和过期能力。用户只能将服务端通知标记为已读；历史通知仍保留，用于回看关键状态变化。

`Notification` 至少包含：

| 字段 | 含义 | 说明 |
|---|---|---|
| `id` | 通知 ID | 系统生成 |
| `recipientUserId` | 接收用户 ID | 通知所属用户 |
| `type` | 通知类型 | 对应 V1 8 类通知触发点 |
| `title` | 标题 | 通知列表主文案，短文本 |
| `content` | 内容 | 通知详情或列表摘要，可包含必要原因摘要 |
| `sourceType` | 来源对象类型 | 例如 `IDENTITY_VERIFICATION`、`PARTNER_POST`、`CONVERSATION`、`REVIEW`、`CASE_TICKET`、`GOVERNANCE_ACTION`、`CREDIT_ADJUSTMENT` |
| `sourceId` | 来源对象 ID | 用于跳转和追踪 |
| `actionUrl` | 客户端跳转目标 | 可为空；Qt 客户端内部路由或页面标识 |
| `status` | 通知状态 | `UNREAD`、`READ` |
| `createdAt` | 创建时间 | 用于排序 |
| `readAt` | 已读时间 | 未读时为空 |

`Notification.type` 建议值如下：

| 类型 | 对应触发点 |
|---|---|
| `IDENTITY_REVIEW_RESULT` | 认证审核结果 |
| `POST_REVIEW_RESULT` | 需求审核结果 |
| `NEW_CONVERSATION_INVITATION` | 收到新会话/邀约 |
| `REVIEW_REMINDER` | 评价提醒 |
| `REVIEW_DISPUTE_RESULT` | 评价争议处理结果 |
| `CASE_STATUS_CHANGED` | 案件状态变化 |
| `GOVERNANCE_ACTION_CHANGED` | 治理动作生效/撤销 |
| `CREDIT_ADJUSTMENT_CHANGED` | 信用摘要人工调整 |

状态流转规则如下：

| 起始状态 | 触发动作 | 目标状态 | 说明 |
|---|---|---|---|
| 无记录 | 系统创建通知 | `UNREAD` | 默认未读 |
| `UNREAD` | 用户标记已读或打开通知详情 | `READ` | 写入 `readAt` |
| `READ` | 用户重复标记已读 | `READ` | 幂等，不更新时间或按后端实现保持稳定 |

V1 不允许用户删除服务端通知，也不允许管理员撤回已发给用户的通知。若业务处理结果后续被撤销或修正，应创建新的通知说明变化，而不是修改或删除旧通知。该规则保证关键状态变化的用户可见历史完整。

通知列表需要支持分页和按状态筛选。未读数只统计当前用户 `status = UNREAD` 的通知，不统计会话消息未读数；会话未读数仍由低压力联系模块维护，通知中心可以在 UI 上并列展示，但后端统计口径分开。

#### 4.10.6 `SystemEventLog` 字段与事件类型

`SystemEventLog` 只做业务事件留痕和通知派生追踪，不作为证据仓库、聊天归档、认证材料快照或联系方式历史表。它记录结构化引用、摘要和追踪信息，帮助系统串联“发生了什么、影响了谁、是否派生了通知”。

`SystemEventLog` 至少包含：

| 字段 | 含义 | 说明 |
|---|---|---|
| `id` | 系统事件 ID | 系统生成 |
| `eventType` | 事件类型 | 对应业务状态变化或通知触发点 |
| `sourceType` | 来源对象类型 | 例如 `IDENTITY_VERIFICATION`、`PARTNER_POST`、`CONVERSATION`、`REVIEW`、`CASE_TICKET`、`GOVERNANCE_ACTION`、`CREDIT_ADJUSTMENT` |
| `sourceId` | 来源对象 ID | 指向发生变化的业务对象 |
| `actorUserId` | 触发用户 ID | 可为空；系统自动事件或定时任务可为空 |
| `affectedUserIds` | 受影响用户集合 | 用于派生通知或排查影响范围 |
| `notificationCreated` | 是否已派生通知 | 表示该事件是否产生用户可见通知 |
| `notificationIds` | 派生通知 ID 集合 | 可为空；一个事件可派生多条通知 |
| `summary` | 事件摘要 | 简短说明，不记录敏感正文 |
| `createdAt` | 事件时间 | 事件创建时间 |
| `traceId` | 请求追踪 ID | 用于串联接口请求、错误响应和后端日志 |

`SystemEventLog.eventType` 建议值如下：

| 事件类型 | 对应业务事件 |
|---|---|
| `IDENTITY_REVIEW_RESULT` | 认证审核通过或驳回 |
| `POST_REVIEW_RESULT` | 需求审核通过或驳回 |
| `CONVERSATION_INVITATION_CREATED` | 新会话创建或新邀约上下文追加 |
| `REVIEW_AVAILABLE` | 会话达到可评价条件或联系方式解锁后可评价 |
| `REVIEW_DISPUTE_DECIDED` | 评价争议处理完成 |
| `CASE_STATUS_CHANGED` | 案件进入待回应、收到回应、结案等状态变化 |
| `GOVERNANCE_ACTION_CHANGED` | 治理动作创建、撤销或恢复 |
| `CREDIT_ADJUSTMENT_CHANGED` | 信用摘要人工调整生效或撤销 |

`SystemEventLog` 不保存以下内容：

- 认证材料原文、证件或图片内容。
- 聊天消息原文。
- 投诉证据原文或对象存储裸地址。
- 联系方式值，例如微信号、手机号、QQ 号。
- 管理员内部审计原文或敏感复核材料。

若排查或业务展示需要查看详情，应通过 `sourceType/sourceId` 回到对应业务对象，并由对应业务接口执行权限校验。日志表只提供索引和摘要，不绕过原对象权限。

系统事件与通知派生应尽量在同一业务处理流程中完成。若事件已创建但通知创建失败，应根据业务重要性选择整体回滚或进入可重试的通知派生流程；V1 可优先采用同事务创建，保证关键状态变化和通知记录一致。

#### 4.10.7 接口契约草案

通知与留痕接口按“用户侧通知接口”和“管理侧系统事件查询接口”拆分。用户侧接口只允许当前用户查看自己的通知、读取未读数和标记已读；管理侧接口只允许查询系统事件用于排查和管理辅助，不提供人工创建通知接口。

用户侧通知接口草案如下：

| 接口 | 用途 | 认证要求 | 关键规则 |
|---|---|---|---|
| `GET /api/me/notifications` | 查询我的通知列表 | 登录用户 | 支持分页、按 `status` 和 `type` 筛选，只返回当前用户通知 |
| `GET /api/me/notifications/unread-count` | 查询我的未读通知数 | 登录用户 | 只统计 `Notification.status = UNREAD`，不包含会话消息未读 |
| `POST /api/me/notifications/{notificationId}/read` | 标记单条通知已读 | 登录用户且为接收方 | `UNREAD -> READ`；重复标记已读保持幂等 |
| `POST /api/me/notifications/read-all` | 全部标记已读 | 登录用户 | 将当前用户所有未读通知置为 `READ` |

管理侧系统事件查询接口草案如下：

| 接口 | 用途 | 认证要求 | 关键规则 |
|---|---|---|---|
| `GET /api/admin/system-events` | 查询系统事件列表 | `ROLE_ADMIN` | 支持按事件类型、来源对象、受影响用户、时间范围、`traceId` 筛选 |
| `GET /api/admin/system-events/{eventId}` | 查看系统事件详情 | `ROLE_ADMIN` | 返回事件摘要、来源引用、派生通知引用，不返回敏感正文 |

V1 不提供以下接口：

- `POST /api/admin/notifications`
- `POST /api/admin/notifications/broadcast`
- `DELETE /api/me/notifications/{notificationId}`
- `POST /api/admin/notifications/{notificationId}/withdraw`

也就是说，管理员不能手工发通知、群发通知、撤回通知或替用户删除通知。通知必须由业务事件触发；若出现漏发，应修复触发逻辑或在后续补偿任务中处理，而不是通过人工通知接口绕过事件链路。

`NotificationListItemResponse` 至少包含：

| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | string | 通知 ID |
| `type` | string | 通知类型 |
| `title` | string | 标题 |
| `content` | string | 内容摘要 |
| `sourceType` | string | 来源类型 |
| `sourceId` | string | 来源 ID |
| `actionUrl` | string | Qt 客户端跳转目标，可为空 |
| `status` | string | `UNREAD` 或 `READ` |
| `createdAt` | datetime | 创建时间 |
| `readAt` | datetime | 已读时间，可为空 |

`SystemEventLogResponse` 至少包含事件字段、来源对象引用、受影响用户 ID 摘要、是否已派生通知、通知 ID 列表、事件摘要、时间和 `traceId`。该响应不得返回认证材料、证据、聊天原文或联系方式值。

接口统一使用项目错误响应结构 `code`、`message`、`details`、`traceId`。P1-4 至少需要约定以下错误码：

| 错误码 | 适用场景 |
|---|---|
| `UNAUTHORIZED` | 未登录访问用户侧或管理侧接口 |
| `FORBIDDEN` | 非管理员访问管理侧系统事件，或用户访问不属于自己的通知 |
| `NOTIFICATION_NOT_FOUND` | 通知不存在或当前用户不可见 |
| `SYSTEM_EVENT_NOT_FOUND` | 系统事件不存在或当前管理员不可见 |
| `VALIDATION_FAILED` | 分页、筛选条件或状态值非法 |

Qt 客户端应封装 `NotificationApiService`；管理端可封装 `AdminSystemEventApiService`。通知中心不得调用管理侧系统事件接口；系统事件页面不得修改通知状态。

#### 4.10.8 测试入口与后续实现批次

通知与留痕模块后续实现按 3 个批次推进，每批必须遵守测试先行、红灯确认、最小实现、绿灯确认、回归测试和测试留档规则。V1 不在本模块引入 WebSocket、短信、邮件、移动端 Push 或人工群发后台。

| 批次 | 范围 | 主要测试入口 | 阶段闭环 |
|---|---|---|---|
| Batch 1 | 后端 `Notification` 持久化、列表、未读数、标记已读 | JUnit 5 + Spring Boot 接口/安全测试 | 用户可查看自己的通知、查询未读数、单条或全部标记已读 |
| Batch 2 | 后端 `SystemEventLog` 持久化、8 类事件触发、通知派生、管理侧事件查询 | JUnit 5 + Spring Boot 接口/事务测试 | 关键业务状态变化可写入事件并派生通知，管理员可查询事件 |
| Batch 3 | Qt 通知中心、未读角标、管理端系统事件查询页面 | Qt Test + 必要手工验证清单 | 用户端可查看通知和未读数，管理端可查看系统事件 |

Batch 1 的后端测试至少覆盖：

- 登录用户可分页查询自己的通知。
- 用户不能查询或标记他人的通知。
- 未读数只统计当前用户 `UNREAD` 通知，不包含会话消息未读。
- 单条通知可从 `UNREAD` 标记为 `READ`，并写入 `readAt`。
- 重复标记已读保持幂等。
- 全部标记已读只影响当前用户通知。

Batch 2 的后端测试至少覆盖：

- 8 类业务事件可写入 `SystemEventLog`。
- 系统事件不保存认证材料、聊天原文、证据原文和联系方式值。
- 业务事件可派生一条或多条 `Notification`。
- 事件与通知派生失败时按既定事务策略处理，避免关键状态变化和通知记录不一致。
- 管理员可按事件类型、来源对象、受影响用户、时间范围和 `traceId` 查询系统事件。
- 非管理员访问管理侧系统事件接口返回 `FORBIDDEN`。
- 不存在人工创建通知或群发通知接口。

Batch 3 的 Qt 客户端测试与验证至少覆盖：

- 用户端通知中心展示通知列表、类型、标题、内容摘要、时间和已读状态。
- 用户端展示通知未读角标或未读数。
- 用户打开通知详情或点击标记已读后状态更新。
- 用户可一键全部标记已读。
- 管理端系统事件页面支持筛选、分页和查看详情。
- 管理端系统事件详情只展示引用、摘要和追踪信息，不展示敏感正文。

本模块后续测试留档应写入 `docs/validation/`，每批至少记录对应需求编号、设计入口、测试文件、红灯结果、实现摘要、绿灯结果、回归范围和未覆盖风险。

#### 4.10.9 第一轮收束说明

通知与留痕模块第一轮正文补全已收束。当前已确认 V1 只做站内通知和系统事件留痕，不做短信、邮件、移动端 Push、WebSocket、人工群发或运营通知；已确认 8 类通知触发点、`Notification` 两态规则、`SystemEventLog` 字段与敏感边界、用户侧通知接口、管理侧系统事件查询接口，以及后续 3 个实现批次。后续若其他模块新增需要用户可见提醒的状态变化，应先判断是否属于 P1-4 的 8 类触发点，避免通知中心膨胀成杂项消息流。

| 评价与信用摘要 | `IR12`、`IR13`、`US6`、`US9`、`US12` |
| 管理端治理 | `IR16`、`US10`、`US11` |
| 通知与留痕 | 由 `IR2`、`IR6`、`IR9`、`IR12`、`IR14`、`IR15`、`IR16` 等状态变化派生，当前以 P1-4 追踪矩阵为准 |

### 5.6 P0-2 需求发布与审核用例图前说明

P0-2 需求发布与审核模块用例图聚焦“发布者如何形成可审核发布物”以及“管理员如何完成先审后发裁定”。本图不覆盖广场浏览、发起联系、评价和投诉；这些能力分别进入 P0-3、P0-4 和 P1 模块图。

参与者包括：

- 已认证学生 / 需求发布者。
- 管理员。
- 未认证但已登录用户。

核心用例包括：

| 用例 | 主参与者 | 前置条件 | 结果 |
|---|---|---|---|
| 创建草稿 | 已认证学生 | 用户登录且认证状态为 `VERIFIED` | 创建 `DRAFT` 状态 `PartnerPost` |
| 编辑草稿 | 需求发布者 | 发布物属于当前用户且状态为 `DRAFT` | 更新公共字段或 `scenePayload` |
| 提交审核 | 需求发布者 | 发布物属于当前用户、状态为 `DRAFT`、字段强校验通过 | 状态进入 `PENDING_REVIEW` |
| 撤回审核 | 需求发布者 | 发布物属于当前用户且仍为 `PENDING_REVIEW` | 状态回到 `DRAFT` |
| 主动下架 | 需求发布者 | 发布物属于当前用户且状态为 `PUBLISHED` | 状态回到 `DRAFT`，再次发布需重新审核 |
| 查看我的发布 | 需求发布者 | 用户登录 | 查看自己的发布列表、详情、状态和可执行动作 |
| 查看审核队列 | 管理员 | 用户具备 `ROLE_ADMIN` | 查看 `PENDING_REVIEW` 需求列表 |
| 审核需求 | 管理员 | 目标需求仍为 `PENDING_REVIEW` | 状态进入 `PUBLISHED` 或 `REJECTED` |

未认证但已登录用户不是发布链路的有效操作者。用例图中可将其连接到“尝试创建草稿被拒绝”或“引导完成认证”的扩展说明，但不得把“保存服务端草稿”画成其合法用例。

用例关系建议：

- “提交审核”包含“公共字段校验”和“场景字段校验”。
- “审核需求”扩展为“审核通过”和“审核驳回”两个结果分支。
- “撤回审核”和“审核需求”存在状态冲突约束：两者都只能作用于仍处于 `PENDING_REVIEW` 的发布物。
- “主动下架”不等同于“撤回审核”，前者作用于 `PUBLISHED`，后者作用于 `PENDING_REVIEW`。

## 6. 核心领域对象与类图前说明

### 6.1 类图建模原则

类图采用领域对象优先，不在详细设计阶段过早绑定数据库物理表结构。

类图不要求展示所有 DTO、Controller、Repository 和数据库字段。正式类图应优先表达领域对象、对象间关联、聚合边界和关键状态字段。接口 DTO 可以在接口契约或实现批次中补充，不应把类图画成后端代码包结构图。

### 6.2 核心领域对象簇

核心领域对象按 8 个对象簇组织，避免把全部对象平铺成一张难以阅读的大图。

| 对象簇 | 主要对象 | 职责 |
|---|---|---|
| 账号与认证 | `User`、`UserProfile`、`AuthenticationRecord` | 账号身份、展示资料、校内认证状态和认证资料审核结果 |
| 需求发布 | `PartnerPost`、`PostSearchCriteria`、`PostListItemView`、`PostDetailView` | 搭子需求发布物、场景字段、审核状态、广场输入和展示视图 |
| 低压力联系 | `Conversation`、`Message`、`ContactCard`、`ContactUnlockRecord`、`BlockRelation` | 会话、站内消息、联系方式卡片、双方解锁、拉黑关系 |
| 评价与信用 | `Review`、`CreditSummary`、`CreditChangeLog` | 用户主动评价、星级和标签、信用摘要计算和变化记录 |
| 投诉申诉与案件 | `CaseTicket`、`CaseEvidence`、`CaseResponse` | 统一案件、证据、回应、状态流转和裁定结果 |
| 管理端治理 | `GovernanceAction`、`AdminAuditLog`、`RestrictionPolicy`、`CreditAdjustmentRecord` | 账号处置、管理员审计、功能限制策略、结构化信用干预 |
| 通知与留痕 | `Notification`、`SystemEventLog` | 用户可见站内通知、系统业务事件留痕和通知派生 |
| 附件与对象存储 | `Attachment` | 认证材料、发布图片、案件证据等文件元数据和受控访问引用 |

### 6.3 对象关系与聚合边界

类图建议按对象簇拆分绘制，再用跨模块关系说明连接。每张类图都应避免变成“全系统对象网”，优先表达当前模块内强关联对象。

账号与认证对象关系：

- `User` 是账号主对象，承载登录身份、角色、认证状态和基础账号状态。
- `UserProfile` 依附于 `User`，承载昵称、头像、简介等展示资料。
- `AuthenticationRecord` 或认证提交对象依附于 `User`，记录校内认证资料提交、审核状态、驳回原因和材料附件引用。
- 管理员角色本质上仍是 `User` 的角色或权限属性，不单独拆为管理员账号对象。

需求发布对象关系：

- `PartnerPost` 由 `User` 发布，`publisherId` 指向发布者。
- `PartnerPost` 统一承载草稿、待审核、已发布、已驳回状态，不拆分 `PostDraft`。
- `PostSearchCriteria`、`PostListItemView`、`PostDetailView` 是查询与展示视图对象，不是核心持久化聚合根。
- `PartnerPost.attachmentIds` 指向 `Attachment`，不得保存对象存储裸地址。

低压力联系对象关系：

- `Conversation` 连接两个 `User`，并可记录关联 `PartnerPost` 或最近邀约上下文。
- `Message` 归属于 `Conversation`，发送者为其中一个参与用户。
- `ContactCard` 归属于 `User`，保存用户当前可交换联系方式；联系方式值不写入 `PartnerPost`。
- `ContactUnlockRecord` 归属于 `Conversation`，记录双方是否确认交换联系方式和最终解锁时间。
- `BlockRelation` 表达用户间拉黑关系，是消息、联系、联系方式查看和定向推送的统一前置约束。

评价与信用对象关系：

- `Review` 绑定 `Conversation`，并区分 `reviewerId` 与 `revieweeId`。
- `CreditSummary` 是按用户聚合的信用摘要，可由评价、默认有效对话、初始虚拟评分基线和信用干预计算得到。
- `CreditChangeLog` 记录信用摘要变化来源，不应替代 `Review` 或 `CreditAdjustmentRecord`。

投诉申诉与案件对象关系：

- `CaseTicket` 是统一案件对象，通过 `targetType + targetId` 指向 `Review`、`Conversation`、`PartnerPost`、`User` 或 `ADMIN_DECISION`。
- `CaseEvidence` 归属于 `CaseTicket`，通过 `Attachment` 引用证据文件。
- `CaseResponse` 归属于 `CaseTicket`，记录发起人、被投诉方或相关方的回应说明和证据引用。

管理端治理对象关系：

- `GovernanceAction` 指向被处置 `User`，并通过 `sourceType/sourceId` 关联案件、人工复核或系统检测来源。
- `AdminAuditLog` 记录管理员关键操作，可关联治理动作、案件裁定、审核动作或信用干预。
- `RestrictionPolicy` 可作为从生效 `GovernanceAction` 计算得到的策略视图，不强制独立建表。
- `CreditAdjustmentRecord` 依附于 `GovernanceAction`，只在 `ADJUST_CREDIT` 下承接结构化信用干预明细。

通知与留痕对象关系：

- `SystemEventLog` 记录业务事件，包含 `sourceType/sourceId`、受影响用户和派生通知引用。
- `Notification` 面向接收用户，通过 `recipientUserId` 归属于 `User`，通过 `sourceType/sourceId` 指向业务来源。
- `Notification` 不替代 `SystemEventLog`，`SystemEventLog` 也不替代 `AdminAuditLog`。

附件与对象存储对象关系：

- `Attachment` 只保存文件元数据、归属关系、用途、存储键和受控访问所需信息。
- 认证材料、发布图片、案件证据均通过 `Attachment` 引用对象存储文件。
- 普通业务对象不得保存对象存储裸 URL，也不得复制附件原文内容。

### 6.4 类图拆分建议

正式类图建议拆为 4 组，而不是一张巨型类图：

1. 学生主链路类图：`User`、`UserProfile`、`PartnerPost`、`Conversation`、`Message`、`ContactCard`、`ContactUnlockRecord`、`BlockRelation`、`Attachment`。
2. 评价与信用类图：`Conversation`、`Review`、`CreditSummary`、`CreditChangeLog`、`CreditAdjustmentRecord`。
3. 案件与治理类图：`CaseTicket`、`CaseEvidence`、`CaseResponse`、`GovernanceAction`、`AdminAuditLog`、`RestrictionPolicy`。
4. 通知与留痕类图：`SystemEventLog`、`Notification`，并通过 `sourceType/sourceId` 指向其他模块对象。

### 6.5 类图扩展点

- 多学校扩展时，`User`、`PartnerPost`、`CaseTicket`、`SystemEventLog` 等对象可通过 `schoolId` 或学校上下文扩展。
- 更多搭子场景扩展时，优先扩展 `PartnerPost.sceneType` 和 `scenePayload` 校验结构，不新增平行发布对象。
- 消息能力扩展时，可在 `Message` 增加更多消息类型，但 V1 纯文本消息边界不变。
- 通知渠道扩展时，可在 `Notification` 或后续投递对象上增加渠道，不改变 `SystemEventLog` 作为事件源的定位。
- 对象存储扩展时，业务对象继续引用 `Attachment`，不直接依赖具体云厂商 SDK。

## 7. 关键状态设计与状态图前说明

### 7.1 状态图覆盖原则

状态图只覆盖状态复杂、容易出错、影响流程判断的对象。

状态图不覆盖所有领域对象。`SystemEventLog`、`Attachment`、`CreditSummary` 等对象更多是事件记录、文件元数据或计算摘要，不作为 V1 主状态图对象。若某对象只有“创建后保留”这一类简单生命周期，不单独绘制状态图。

### 7.2 第一版状态图对象

第一版状态图只覆盖 8 个高风险状态对象：

| 状态对象 | 状态字段或状态视图 | 覆盖理由 |
|---|---|---|
| 用户认证状态 | `User.authenticationStatus` | 决定是否可发布需求、发起联系、保存服务端草稿等关键权限 |
| 搭子需求状态 | `PartnerPost.status` | 决定发布、审核、广场展示、撤回和下架链路 |
| 会话状态 | `Conversation.status` | 决定能否发送消息、解锁联系方式、重新激活会话 |
| 联系方式解锁状态 | `ContactUnlockRecord.status` 或双方确认视图 | 决定是否可查看联系方式 |
| 评价状态 | `Review.status` | 决定评价是否计入信用摘要、是否处于争议或被撤销影响 |
| 案件状态 | `CaseTicket.status` | 决定能否补证据、回应、进入管理端队列或关闭 |
| 治理动作生效/撤销状态 | `GovernanceAction` 生效视图 | 决定账号限制是否当前生效、是否可撤销 |
| 通知状态 | `Notification.status` | 决定未读数和通知中心展示 |

### 7.3 状态集合与合法流转摘要

#### 7.3.1 用户认证状态

`User.authenticationStatus` 状态集合：

- `UNVERIFIED`
- `PENDING_REVIEW`
- `VERIFIED`
- `REJECTED`
- `SUSPENDED`

核心流转：

| 起始状态 | 触发动作 | 目标状态 |
|---|---|---|
| `UNVERIFIED` | 用户提交认证资料 | `PENDING_REVIEW` |
| `PENDING_REVIEW` | 管理员审核通过 | `VERIFIED` |
| `PENDING_REVIEW` | 管理员审核驳回 | `REJECTED` |
| `REJECTED` | 用户重新提交认证资料 | `PENDING_REVIEW` |
| `VERIFIED` | 高风险治理动作限制账号 | `SUSPENDED` |
| `SUSPENDED` | 超级管理员恢复账号 | `VERIFIED` 或恢复前状态 |

认证状态图应体现：`SUSPENDED` 是治理动作触发的受限状态，不是普通认证审核结果；未认证或待审核用户不得进入发布和联系等已认证能力链路。

#### 7.3.2 搭子需求状态

`PartnerPost.status` 状态集合：

- `DRAFT`
- `PENDING_REVIEW`
- `PUBLISHED`
- `REJECTED`

核心流转：

| 起始状态 | 触发动作 | 目标状态 |
|---|---|---|
| 无记录 | 发布者创建草稿 | `DRAFT` |
| `DRAFT` | 发布者提交审核且校验通过 | `PENDING_REVIEW` |
| `PENDING_REVIEW` | 发布者撤回审核 | `DRAFT` |
| `PENDING_REVIEW` | 管理员审核通过 | `PUBLISHED` |
| `PENDING_REVIEW` | 管理员审核驳回 | `REJECTED` |
| `REJECTED` | 发布者编辑被驳回需求 | `DRAFT` |
| `PUBLISHED` | 发布者主动下架 | `DRAFT` |

非法流转包括：未认证用户创建服务端草稿、`PENDING_REVIEW` 直接编辑内容、`PUBLISHED` 直接编辑并保持发布、`REJECTED` 直接提交审核而未进入编辑草稿链路。

#### 7.3.3 会话状态

`Conversation.status` 状态集合：

- `ACTIVE`
- `CLOSED`

核心流转：

| 起始状态 | 触发动作 | 目标状态 |
|---|---|---|
| 无记录 | 发起联系并创建会话 | `ACTIVE` |
| `ACTIVE` | 任一参与方主动关闭会话 | `CLOSED` |
| `ACTIVE` | 任一参与方拉黑对方 | `CLOSED` |
| `CLOSED` | 同一双方基于新的 `PartnerPost` 发起新邀约 | `ACTIVE` |

状态图应体现：关闭后的会话不能手动直接重新打开；重新激活必须由新的需求邀约触发，并重新满足发起联系、拉黑关系和联系方式卡片等前置条件。

#### 7.3.4 联系方式解锁状态

联系方式解锁可通过 `ContactUnlockRecord` 的状态字段或双方确认视图表达。建议状态集合：

- `LOCKED`
- `WAITING_FOR_PEER`
- `UNLOCKED`

核心流转：

| 起始状态 | 触发动作 | 目标状态 |
|---|---|---|
| `LOCKED` | 一方确认愿意交换联系方式 | `WAITING_FOR_PEER` |
| `WAITING_FOR_PEER` | 另一方也确认交换 | `UNLOCKED` |
| `WAITING_FOR_PEER` | 会话关闭或拉黑 | `LOCKED` 或失效视图 |
| `UNLOCKED` | 会话关闭或拉黑 | 失效视图，普通客户端不再展示联系方式 |

状态图应体现：`UNLOCKED` 不等于永久可见。会话关闭或拉黑后，即使存在历史解锁记录，普通客户端也不继续展示联系方式。

#### 7.3.5 评价状态

`Review.status` 状态集合：

- `ACTIVE`
- `DISPUTED`
- `INVALIDATED`

核心流转：

| 起始状态 | 触发动作 | 目标状态 |
|---|---|---|
| 无记录 | 用户提交评价 | `ACTIVE` |
| `ACTIVE` | 被评价方发起评价争议案件 | `DISPUTED` |
| `DISPUTED` | 管理员维持评价 | `ACTIVE` |
| `DISPUTED` | 管理员撤销评价影响 | `INVALIDATED` |

评价状态图应体现：标签隐藏可以作为信用干预或评价处理结果存在，不一定改变 `Review.status`；`INVALIDATED` 后评价不再计入平均星数和标签统计，但原评价记录不物理删除。

#### 7.3.6 案件状态

`CaseTicket.status` 状态集合：

- `SUBMITTED`
- `WAITING_FOR_UPDATE`
- `CLOSED`

核心流转：

| 起始状态 | 触发动作 | 目标状态 |
|---|---|---|
| 无记录 | 用户创建案件 | `SUBMITTED` |
| `SUBMITTED` | 管理员要求补充或回应 | `WAITING_FOR_UPDATE` |
| `SUBMITTED` | 管理员直接裁定 | `CLOSED` |
| `WAITING_FOR_UPDATE` | 相关方补充证据或回应 | `SUBMITTED` |
| `WAITING_FOR_UPDATE` | 管理员基于现有材料裁定 | `CLOSED` |

案件状态图应体现：`CLOSED` 后不得继续补证据或回应；用户不认可处理结果时，应基于 `ADMIN_DECISION` 创建新案件，而不是重开旧案件。

#### 7.3.7 治理动作生效与撤销状态

`GovernanceAction` 不一定需要单一 `status` 字段，可由 `effectiveFrom`、`effectiveUntil`、`reversedAt` 计算生效视图。建议状态视图：

- `SCHEDULED`
- `ACTIVE`
- `EXPIRED`
- `REVERSED`

核心流转：

| 起始视图 | 触发条件或动作 | 目标视图 |
|---|---|---|
| 无记录 | 管理员创建未来生效治理动作 | `SCHEDULED` |
| 无记录 | 管理员创建立即生效治理动作 | `ACTIVE` |
| `SCHEDULED` | 到达 `effectiveFrom` | `ACTIVE` |
| `ACTIVE` | 到达 `effectiveUntil` | `EXPIRED` |
| `SCHEDULED` / `ACTIVE` | 超级管理员撤销 | `REVERSED` |

状态图应体现：撤销不是删除；`REVERSED` 保留原治理动作和撤销审计记录。账号功能限制由当前仍生效的治理动作计算，而不是只看用户表单一字段。

#### 7.3.8 通知状态

`Notification.status` 状态集合：

- `UNREAD`
- `READ`

核心流转：

| 起始状态 | 触发动作 | 目标状态 |
|---|---|---|
| 无记录 | 系统创建通知 | `UNREAD` |
| `UNREAD` | 用户标记已读或打开通知详情 | `READ` |
| `READ` | 用户重复标记已读 | `READ` |

通知状态图应体现：V1 不提供删除、归档、撤回和过期；业务结果修正时创建新通知说明变化，不修改或删除旧通知。

### 7.4 状态图一致性约束

- 状态图中的状态名称必须与正文、接口 DTO、错误码和测试用例保持一致。
- 状态变更必须由明确业务动作触发，不能由客户端本地状态直接决定。
- 客户端可根据状态隐藏或禁用按钮，但后端必须重新校验状态和权限。
- 并发状态冲突统一返回对应状态冲突错误，例如 `POST_STATUS_CONFLICT`、`CASE_STATUS_CONFLICT`、`GOVERNANCE_ACTION_CONFLICT`。
- 终止或完成类状态不等于物理删除；历史记录应保留，用于评价、案件、治理、通知和审计。

## 8. 关键业务流程与活动图前说明

### 8.1 活动图覆盖原则

活动图优先覆盖 V1 主链路，不优先绘制零散页面操作。

活动图用于说明跨角色、跨模块的业务推进顺序。列表查询、筛选排序、普通资料编辑、单页表单交互等局部操作不单独绘制活动图，可由用例说明、接口契约和测试用例覆盖。

### 8.2 第一版活动图范围

第一版活动图只覆盖 6 条主流程：

1. 认证闭环。
2. 需求发布审核。
3. 广场联系解锁。
4. 评价信用。
5. 投诉申诉案件。
6. 治理通知闭环。

### 8.3 认证闭环活动图说明

参与角色：未认证学生、已认证学生、管理员。

起点：用户使用校园邮箱注册或登录后，尚未完成校内身份认证。

终点：用户认证状态进入 `VERIFIED` 或 `REJECTED`，客户端展示下一步可执行动作。

主路径：

1. 用户通过校园邮箱验证码完成注册或登录。
2. 系统创建或识别 `User`，认证状态为 `UNVERIFIED`。
3. 用户提交认证资料和材料附件。
4. 系统校验字段、附件归属和认证状态，将认证提交置为待审核。
5. 管理员查看认证审核队列和材料摘要。
6. 管理员审核通过，用户认证状态进入 `VERIFIED`。
7. 系统生成认证审核结果通知和系统事件。

异常分支：

- 验证码无效或过期：登录/注册失败。
- 认证资料字段缺失：返回字段校验错误。
- 附件不属于当前用户或格式不合法：提交失败。
- 认证审核驳回：状态进入 `REJECTED`，用户可修改后重提。
- 用户已处于 `PENDING_REVIEW` 或 `VERIFIED`：重复提交按既有认证规则拒绝。

状态变化：`User.authenticationStatus` 从 `UNVERIFIED` 到 `PENDING_REVIEW`，再到 `VERIFIED` 或 `REJECTED`。

需求来源：`IR1`、`IR2`。

### 8.4 需求发布审核活动图说明

参与角色：已认证学生、需求发布者、管理员、系统事件触发器。

起点：已认证学生准备发布搭子需求。

终点：需求进入 `PUBLISHED` 并成为广场输入，或进入 `REJECTED` 等待修改。

主路径：

1. 已认证学生创建 `PartnerPost` 草稿。
2. 发布者填写公共字段和场景字段。
3. 发布者提交审核。
4. 后端执行发布资格、归属、字段、附件和状态校验。
5. 需求状态进入 `PENDING_REVIEW`。
6. 管理员查看审核队列和需求详情。
7. 管理员审核通过，需求状态进入 `PUBLISHED`。
8. 系统生成需求审核结果通知和系统事件。
9. 广场发现模块可读取该需求作为展示输入。

异常分支：

- 用户未认证：不得创建服务端草稿或提交审核。
- 提交字段不完整：返回 `VALIDATION_FAILED`。
- 发布者撤回审核：`PENDING_REVIEW -> DRAFT`。
- 管理员驳回：`PENDING_REVIEW -> REJECTED`，记录驳回原因。
- 审核与撤回并发：返回 `POST_STATUS_CONFLICT`。

状态变化：`PartnerPost.status` 在 `DRAFT`、`PENDING_REVIEW`、`PUBLISHED`、`REJECTED` 间流转。

需求来源：`IR3`、`IR4`、`IR5`、`IR6`、`IR7`、`US1`、`US2`、`US3`、`US4`、`US5`、`US8`。

### 8.5 广场联系解锁活动图说明

参与角色：已认证学生、需求发布者、被联系对象、系统事件触发器。

起点：用户在广场或需求详情页看到已发布需求。

终点：双方建立站内会话并在双方确认后解锁联系方式；或会话关闭/拉黑后终止互动。

主路径：

1. 用户浏览广场、筛选需求并打开需求详情。
2. 用户发起低压力联系并输入初始邀约消息。
3. 后端校验双方认证状态、需求状态、拉黑关系和联系方式卡片要求。
4. 系统创建或复用双方有效 `Conversation`，发送初始邀约消息。
5. 系统生成新会话/邀约通知和系统事件。
6. 双方通过站内消息沟通。
7. 一方确认愿意交换联系方式，解锁状态进入等待对方确认。
8. 另一方确认后，联系方式解锁。
9. 双方可查看对方当前有效 `ContactCard` 内容。

异常分支：

- 目标需求非 `PUBLISHED`：不得发起联系。
- 发起者等于发布者：拒绝发起。
- 任一方被治理限制或拉黑：拒绝发起或发送消息。
- 任一方缺少可交换联系方式：在解锁前要求补全。
- 会话关闭或拉黑：不得继续发送消息或查看联系方式。

状态变化：`Conversation.status` 在 `ACTIVE` / `CLOSED` 间流转；`ContactUnlockRecord` 在 `LOCKED`、`WAITING_FOR_PEER`、`UNLOCKED` 或失效视图间变化。

需求来源：`IR7`、`IR8`、`IR9`、`IR10`、`US7`、`US8`。

### 8.6 评价信用活动图说明

参与角色：已认证学生、评价者、被评价者、系统事件触发器。

起点：双方存在有效对话，即同一 `Conversation` 中双方各自至少发送过 2 条用户消息。

终点：评价记录保存，信用摘要重算；或无显式评价时按默认规则进入信用摘要。

主路径：

1. 系统识别会话达到有效对话门槛。
2. 系统生成评价提醒通知和系统事件。
3. 用户主动进入评价入口。
4. 用户提交星级和可选预设标签。
5. 后端校验会话可评价、评价上限和重复评价规则。
6. 系统保存 `Review`。
7. 系统重算被评价方 `CreditSummary`。
8. 用户信用卡片展示平均星数、真实有效总对话次数和高频标签。

异常分支：

- 会话未达到有效对话门槛：不得评价。
- 已存在同一会话同一评价者对同一被评价者的有效评价：按修改规则处理或返回重复错误。
- 超过 24 小时修改窗口或已修改一次：拒绝修改。
- 评价被申诉：进入评价争议案件，`Review.status -> DISPUTED`。
- 管理员撤销评价影响：`Review.status -> INVALIDATED` 并重算信用摘要。

状态变化：`Review.status` 在 `ACTIVE`、`DISPUTED`、`INVALIDATED` 间流转；`CreditSummary` 按评价、默认有效对话、初始基线和信用干预重算。

需求来源：`IR12`、`IR13`、`US6`、`US9`、`US12`。

### 8.7 投诉申诉案件活动图说明

参与角色：案件发起人、回应方、管理员、系统事件触发器。

起点：用户对评价、会话、需求、用户或管理处理结果产生争议。

终点：案件进入 `CLOSED`，相关业务对象按裁定结果回写，必要时形成治理线索。

主路径：

1. 用户选择目标对象和原因，创建 `CaseTicket`。
2. 用户填写说明并上传 0 到 10 个证据附件。
3. 案件状态进入 `SUBMITTED`。
4. 管理员查看案件队列和详情。
5. 管理员认为材料不足或需要对方回应，案件进入 `WAITING_FOR_UPDATE`。
6. 系统通知需要补充或回应的一方。
7. 相关方补充证据或提交回应，案件回到 `SUBMITTED`。
8. 管理员裁定案件，案件进入 `CLOSED`。
9. 系统根据裁定结果回写评价、会话、需求或治理线索。
10. 系统生成案件状态变化通知和系统事件。

异常分支：

- 用户无权针对目标对象创建案件：返回不可感知或无权限错误。
- 附件数量超过 10 个或格式不合法：拒绝提交。
- 72 小时未回应：管理员可基于现有材料裁定，但未回应本身不自动等于违规。
- 案件关闭后继续补证据或回应：返回 `CASE_STATUS_CONFLICT`。
- 用户不认可处理结果：基于 `ADMIN_DECISION` 创建新案件，不重开旧案。

状态变化：`CaseTicket.status` 在 `SUBMITTED`、`WAITING_FOR_UPDATE`、`CLOSED` 间流转；相关 `Review`、`Conversation`、`PartnerPost` 或治理线索按裁定结果变化。

需求来源：`IR11`、`IR14`、`IR15`、`IR16`、`US10`、`US11`。

### 8.8 治理通知闭环活动图说明

参与角色：管理员、超级管理员、被治理用户、系统事件触发器。

起点：案件裁定、恶意举报认定、人工复核或系统检测形成治理需求。

终点：治理动作生效或撤销，限制策略或信用摘要变化完成，用户收到通知，系统事件和审计日志留痕。

主路径：

1. 管理员从案件或用户治理档案发起治理动作。
2. 后端校验管理员角色、目标用户、来源对象和治理动作权限。
3. 系统创建 `GovernanceAction`。
4. 若动作限制功能，`RestrictionPolicy` 计算当前能力限制。
5. 若动作是 `ADJUST_CREDIT`，系统创建 `CreditAdjustmentRecord` 并重算信用摘要。
6. 后端写入 `AdminAuditLog`。
7. 系统写入 `SystemEventLog`。
8. 系统向被治理用户创建 `Notification`。
9. 被治理用户查看处理结果和必要申诉入口。
10. 若超级管理员撤销治理动作，系统追加撤销信息、审计日志、系统事件和通知。

异常分支：

- 普通管理员执行高风险动作：返回 `FORBIDDEN`。
- 治理动作与现有生效动作冲突：返回 `GOVERNANCE_ACTION_CONFLICT`。
- 审计日志写入失败：治理动作回滚。
- 通知或事件派生失败：按本模块事务策略回滚或进入可重试流程。
- 被限制用户申诉：进入 `ADMIN_DECISION` 类型案件。

状态变化：`GovernanceAction` 生效视图在 `SCHEDULED`、`ACTIVE`、`EXPIRED`、`REVERSED` 间变化；`Notification.status` 初始为 `UNREAD`；信用摘要只由 `ADJUST_CREDIT` 下的结构化记录影响。

需求来源：`IR16`、`US10`、`US11`，并承接 P1-4 通知与留痕触发点。

## 9. 关键交互协作与顺序图前说明

### 9.1 顺序图覆盖原则

顺序图只画跨模块协作最强的关键用例，不覆盖每个页面操作。

顺序图用于服务后续实现和联调，重点表达 Qt API Client、后端 Controller、Service、Repository、对象存储、通知事件、审计治理等参与对象之间的调用顺序。纯查询类页面、筛选排序、表单本地校验等场景不单独绘制顺序图。

### 9.2 第一版顺序图范围

1. 提交认证资料。
2. 审核认证。
3. 发布并提交审核。
4. 审核需求。
5. 发起联系并解锁联系方式。
6. 提交评价并重算信用。
7. 创建案件并裁定。
8. 治理动作并通知。

### 9.3 顺序图共用约束

- 顺序图中的接口路径必须与正文接口契约一致。
- Qt UI Widget 不直接调用网络层，应通过 ViewModel/Presenter 和 API Client 发起请求。
- 后端 Controller 负责认证上下文、请求 DTO 和响应结构，核心业务校验放在 Service。
- Repository 只表达持久化边界，不在顺序图中展开数据库表结构。
- 涉及附件读取或上传时，通过 `ObjectStorageService` 或附件服务表达，不暴露对象存储裸 URL。
- 涉及管理操作时，必须体现角色校验和审计/事件留痕。
- 涉及状态变化时，必须体现状态冲突失败分支。
- 涉及通知时，必须体现 `SystemEventLog` 与 `Notification` 的派生位置。

### 9.4 提交认证资料顺序图前说明

参与对象：

- `Student`
- `Qt IdentityVerificationWidget / ViewModel`
- `AuthApiService`
- `IdentityVerificationController`
- `IdentityVerificationService`
- `IdentityVerificationSubmissionRepository`
- `IdentityMaterialAttachmentService`
- `ObjectStorageService`

主路径：

1. 学生在 Qt 认证资料页面填写真实姓名、学号、学院、专业、年级并选择材料文件。
2. Qt 客户端先调用认证材料上传接口。
3. 后端校验登录态、文件类型、大小和用途。
4. `IdentityMaterialAttachmentService` 通过 `ObjectStorageService` 保存材料文件，返回 `materialAttachmentId`。
5. Qt 客户端提交认证资料表单和 `materialAttachmentId`。
6. `IdentityVerificationController` 接收请求并传入当前用户上下文。
7. `IdentityVerificationService` 校验当前认证状态和材料归属。
8. 后端创建或更新认证提交记录，用户认证状态进入 `PENDING_REVIEW`。
9. Qt 客户端刷新认证状态。

失败分支：

- 未登录：返回 `UNAUTHORIZED`。
- 文件类型、大小或用途不合法：返回 `VALIDATION_FAILED`。
- 附件不属于当前用户：返回 `FORBIDDEN`。
- 用户已认证或已有待审核提交：返回对应认证状态冲突错误。

### 9.5 审核认证顺序图前说明

参与对象：

- `Admin`
- `Qt AdminIdentityReviewView / ViewModel`
- `AdminIdentityApiService`
- `AdminIdentityVerificationController`
- `IdentityVerificationAdminService`
- `IdentityVerificationSubmissionRepository`
- `UserRepository`
- `AdminAuditLogService`
- `SystemEventLogService`
- `NotificationService`

主路径：

1. 管理员打开认证审核队列。
2. Qt 管理端调用认证审核队列接口。
3. 后端校验 `ROLE_ADMIN`。
4. 管理员打开认证详情并查看材料摘要或受控材料。
5. 管理员提交审核通过或驳回。
6. 后端确认认证提交仍处于待审核状态。
7. 审核通过时，用户认证状态进入 `VERIFIED`；驳回时进入 `REJECTED` 并记录原因。
8. 后端写入必要审计或审核留痕。
9. 后端写入 `SystemEventLog` 并创建认证审核结果 `Notification`。
10. Qt 管理端刷新审核队列。

失败分支：

- 非管理员访问：返回 `FORBIDDEN`。
- 审核目标不存在或已被处理：返回状态冲突。
- 驳回未填写原因：返回 `VALIDATION_FAILED`。
- 事件或通知派生失败：按通知模块事务策略处理。

### 9.6 发布并提交审核顺序图前说明

P0-2 至少需要两张顺序图前说明：一张覆盖发布者保存草稿并提交审核，一张覆盖管理员审核通过或驳回。顺序图应体现 Qt 客户端、后端 Controller、Service、Repository、附件访问服务和权限校验之间的调用关系，但不需要展开数据库物理表细节。

#### 9.6.1 发布需求并提交审核

参与对象：

- `Publisher`
- `Qt PartnerPostView / ViewModel`
- `PartnerPostApiClient`
- `PartnerPostController`
- `PartnerPostService`
- `PartnerPostRepository`
- `ObjectStorageService` 或附件服务

主路径：

1. 发布者在 Qt 发布表单选择 `sceneType`，填写公共字段和场景字段。
2. Qt 客户端调用学生侧接口创建或更新 `DRAFT`。
3. 后端校验登录态和用户认证状态，非 `VERIFIED` 用户直接拒绝。
4. 后端执行草稿弱校验，保存 `PartnerPost`。
5. 发布者点击提交审核。
6. Qt 客户端调用 `POST /api/me/partner-posts/{postId}/submit-review`。
7. 后端确认发布物归属当前用户且状态为 `DRAFT`。
8. 后端执行公共字段、场景字段、附件引用和数量上限的强校验。
9. 校验通过后，后端将状态更新为 `PENDING_REVIEW`，返回详情或状态摘要。
10. Qt 客户端刷新我的发布详情，展示“待审核”和可撤回动作。

失败分支：

- 未登录：返回 `UNAUTHORIZED`。
- 用户非 `VERIFIED`：返回 `AUTHENTICATION_STATUS_REQUIRED`。
- 非发布者访问：返回 `FORBIDDEN` 或不可感知的 `POST_NOT_FOUND`。
- 字段校验失败：返回 `VALIDATION_FAILED`，并在 `details` 中返回字段级错误。
- 当前状态不是 `DRAFT`：返回 `POST_STATUS_CONFLICT`。

### 9.7 审核需求顺序图前说明

参与对象：

- `Admin`
- `Qt AdminReviewView / ViewModel`
- `AdminPartnerPostApiClient`
- `AdminPartnerPostController`
- `PartnerPostReviewService`
- `PartnerPostRepository`
- `ObjectStorageService` 或附件服务

主路径：

1. 管理员打开审核队列。
2. Qt 管理端调用 `GET /api/admin/partner-posts/review-queue`。
3. 后端校验管理员角色，只返回 `PENDING_REVIEW` 发布物。
4. 管理员打开某条审核详情。
5. 后端返回公共字段、`scenePayload`、附件受控预览入口和发布者必要摘要。
6. 管理员提交审核裁定。
7. Qt 管理端调用 `POST /api/admin/partner-posts/{postId}/review`。
8. 后端确认目标仍为 `PENDING_REVIEW`。
9. 若 `decision = APPROVE`，状态进入 `PUBLISHED`，记录 `publishedAt`、`reviewedBy`、`reviewedAt`。
10. 若 `decision = REJECT`，状态进入 `REJECTED`，记录 `rejectReason`、`reviewedBy`、`reviewedAt`。
11. Qt 管理端刷新审核队列或详情。

失败分支：

- 非管理员访问：返回 `FORBIDDEN`。
- 驳回未填写原因：返回 `VALIDATION_FAILED`。
- 发布者已撤回或需求已被其他管理员裁定：返回 `POST_STATUS_CONFLICT`。
- 审核通过时发布者已达到 10 条 `PUBLISHED` 上限：返回 `PUBLISHED_POST_LIMIT_EXCEEDED`。

通过或驳回后，后端还应写入 `SystemEventLog` 并创建需求审核结果 `Notification`，通知发布者审核结果。若后续接入 `AdminAuditLog`，管理端审核动作也应进入管理操作留痕。

### 9.8 发起联系并解锁联系方式顺序图前说明

参与对象：

- `Requester`
- `Publisher`
- `Qt PostDetail / ConversationViewModel`
- `ConversationApiService`
- `ConversationController`
- `ConversationService`
- `PartnerPostRepository`
- `ConversationRepository`
- `ContactCardService`
- `ContactUnlockService`
- `RestrictionPolicyService`
- `SystemEventLogService`
- `NotificationService`

主路径：

1. 用户在需求详情页点击发起联系并输入初始邀约消息。
2. Qt 客户端调用发起联系接口。
3. 后端校验登录态、认证状态、目标需求为 `PUBLISHED`、双方不是同一人。
4. 后端校验拉黑关系和治理限制。
5. 后端查找双方是否已有有效会话；若有则复用，若无则创建 `Conversation`。
6. 后端写入初始邀约 `Message`。
7. 后端写入 `SystemEventLog` 并创建新会话/邀约 `Notification`。
8. 双方在会话中发送站内消息。
9. 一方确认交换联系方式，`ContactUnlockService` 记录单方确认。
10. 另一方确认后，联系方式解锁。
11. 后端返回对方当前有效 `ContactCard` 内容。

失败分支：

- 用户未认证或被治理限制：拒绝发起联系。
- 目标需求不可见或非 `PUBLISHED`：返回不可感知或状态错误。
- 存在拉黑关系：拒绝互动。
- 初始邀约消息为空或超长：返回 `VALIDATION_FAILED`。
- 解锁时任一方未维护联系方式卡片：要求补全。
- 会话已关闭且无新需求邀约：不得静默恢复。

### 9.9 提交评价并重算信用顺序图前说明

参与对象：

- `Reviewer`
- `Qt ReviewView / CreditCardViewModel`
- `ReviewApiService`
- `ReviewController`
- `ReviewService`
- `ConversationRepository`
- `ContactUnlockRecordRepository`
- `ReviewRepository`
- `CreditSummaryService`
- `CreditChangeLogRepository`
- `SystemEventLogService`
- `NotificationService`

主路径：

1. 系统或用户进入可评价入口。
2. Qt 客户端提交星级和可选标签。
3. `ReviewController` 接收当前用户上下文和评价请求。
4. `ReviewService` 校验会话存在、双方参与关系和有效对话门槛。
5. 后端根据是否完成联系方式解锁校验最高星级上限。
6. 后端校验同一评价者对同一被评价者在同一会话中是否已有评价。
7. 后端保存或按修改规则更新 `Review`。
8. `CreditSummaryService` 重算被评价方信用摘要。
9. 后端写入必要信用变化记录。
10. Qt 客户端刷新评价结果和信用卡片。

失败分支：

- 会话不可评价：返回 `CONVERSATION_NOT_REVIEWABLE`。
- 重复创建评价：返回 `REVIEW_ALREADY_EXISTS`。
- 修改超过 24 小时或已修改一次：返回对应修改限制错误。
- 星级超过当前上限：返回 `VALIDATION_FAILED`。

### 9.10 创建案件并裁定顺序图前说明

参与对象：

- `Complainant`
- `Respondent`
- `Admin`
- `Qt CaseView / AdminCaseViewModel`
- `CaseApiService`
- `AdminCaseApiService`
- `CaseController`
- `AdminCaseController`
- `CaseService`
- `CaseTicketRepository`
- `CaseEvidenceService`
- `ObjectStorageService`
- `SystemEventLogService`
- `NotificationService`

主路径：

1. 用户针对评价、会话、需求、用户或管理处理结果创建案件。
2. Qt 客户端提交 `CreateCaseRequest` 和可选证据附件引用。
3. 后端校验目标对象存在、可感知性和创建权限。
4. 后端保存 `CaseTicket`，状态为 `SUBMITTED`。
5. 管理员查看案件队列和案件详情。
6. 管理员要求补充材料或回应，案件进入 `WAITING_FOR_UPDATE`，系统通知相关方。
7. 相关方补充证据或回应，案件回到 `SUBMITTED`。
8. 管理员裁定案件，设置 `decisionResult` 和 `decisionNote`。
9. 后端执行必要业务回写，例如撤销评价影响、隐藏标签、关闭会话、下架需求或标记恶意举报。
10. 案件进入 `CLOSED`。
11. 后端写入 `SystemEventLog` 并创建案件状态变化通知。

失败分支：

- 用户无权针对目标创建案件：返回 `FORBIDDEN` 或目标不可感知。
- 证据数量超过上限或格式不合法：返回 `CASE_EVIDENCE_LIMIT_EXCEEDED` 或 `VALIDATION_FAILED`。
- 案件已关闭后继续补证据或回应：返回 `CASE_STATUS_CONFLICT`。
- 非管理员裁定案件：返回 `FORBIDDEN`。

### 9.11 治理动作并通知顺序图前说明

参与对象：

- `Admin` / `SuperAdmin`
- `Qt AdminGovernanceViewModel`
- `AdminGovernanceApiService`
- `AdminGovernanceController`
- `GovernanceActionService`
- `GovernanceActionRepository`
- `RestrictionPolicyService`
- `CreditSummaryService`
- `AdminAuditLogService`
- `SystemEventLogService`
- `NotificationService`

主路径：

1. 管理员从案件详情或用户治理档案创建治理动作。
2. Qt 管理端调用 `POST /api/admin/governance-actions`。
3. 后端校验管理员角色、动作类型权限、目标用户和来源对象。
4. 后端检查现有生效治理动作是否冲突。
5. 后端保存 `GovernanceAction`。
6. 若动作限制功能，`RestrictionPolicyService` 更新或计算限制视图。
7. 若动作是 `ADJUST_CREDIT`，`CreditSummaryService` 应用结构化信用干预并重算信用摘要。
8. `AdminAuditLogService` 写入管理员审计日志。
9. `SystemEventLogService` 写入系统事件。
10. `NotificationService` 创建治理动作或信用调整通知。
11. Qt 管理端刷新治理档案。

1. 超级管理员提交撤销请求和 `reverseReason`。
2. 后端校验 `SUPER_ADMIN` 角色和治理动作可撤销性。
3. 后端写入撤销字段。
4. 后端解除限制或回退信用干预。
5. 后端追加审计日志、系统事件和通知。

- 普通管理员执行高风险动作：返回 `FORBIDDEN`。
- 治理动作不可撤销或已撤销：返回 `GOVERNANCE_ACTION_NOT_REVERSIBLE`。
- 治理动作冲突：返回 `GOVERNANCE_ACTION_CONFLICT`。
- 审计日志写入失败：治理动作创建或撤销回滚。
- 通知或事件派生失败：按通知模块事务策略处理。

## 10. 管理端治理详细设计

### 10.1 设计定位

管理端治理承接 `SF6 平台治理系统`，在详细设计中独立成重点，不作为学生端流程附属分支。

管理端治理模块负责将案件裁定、恶意举报认定、账号违规、信用纠偏等结果落为可审计、可追踪、可回滚或可解释的治理动作。案件模块中的 `decisionResult` 表达“案件结论”，而管理端治理模块中的 `GovernanceAction` 表达“平台实际采取的处置动作”。两者不能混为一个对象。

### 10.2 覆盖范围

- 内容审核
- 投诉/申诉案件处理
- 被投诉方回应查看
- 账号处置
- 信用摘要/靠谱分干预
- 操作留痕与通知触发

### 10.3 统一治理动作对象

账号处置、信用干预、恶意举报处罚等治理动作统一由 `GovernanceAction` 承载。V1 不为警告、禁言、封禁、信用调整和举报限制分别创建独立治理对象，避免管理端操作和审计记录分散。

`GovernanceAction` 至少包含：

| 字段 | 含义 | 说明 |
|---|---|---|
| `id` | 治理动作 ID | 系统生成 |
| `actionType` | 治理动作类型 | 见下方枚举 |
| `targetUserId` | 被处置用户 ID | 被警告、禁言、封禁、信用调整或限制举报的用户 |
| `sourceType` | 来源类型 | `CASE_TICKET`、`MANUAL_REVIEW`、`SYSTEM_DETECTION` 等 |
| `sourceId` | 来源 ID | 例如关联 `CaseTicket.id` |
| `reasonCode` | 原因类型 | 预设原因码 |
| `reasonNote` | 处置说明 | 管理员填写，进入审计留痕 |
| `effectiveFrom` | 生效时间 | 立即生效或指定时间 |
| `effectiveUntil` | 结束时间 | 临时处置使用；永久处置可为空或用专门标记 |
| `createdBy` | 操作管理员 ID | 执行动作的管理员 |
| `createdAt` | 创建时间 | 治理动作记录时间 |
| `reversedBy` | 撤销管理员 ID | 可为空 |
| `reversedAt` | 撤销时间 | 可为空 |
| `reverseReason` | 撤销原因 | 可为空 |

V1 治理动作类型包括：

| `actionType` | 含义 | 说明 |
|---|---|---|
| `WARNING` | 警告 | 记录违规提醒，不直接限制主要功能 |
| `MUTE` | 禁言 | 限制站内消息、发布或评价等表达类能力，具体范围后续确认 |
| `SUSPEND_ACCOUNT` | 限制账号 | 使账号进入受限状态，阻断关键互动 |
| `RESTORE_ACCOUNT` | 恢复账号 | 解除此前限制 |
| `ADJUST_CREDIT` | 信用干预 | 人工调整信用摘要或标记信用纠偏 |
| `LIMIT_REPORTING` | 限制举报 | 针对恶意举报、批量举报或滥用申诉入口的账号限制案件创建频率或能力 |

`GovernanceAction` 必须写入独立审计日志或与审计日志一一关联。审计记录不得只依赖普通运行日志；也不得记录不必要的敏感原文。涉及证据、联系方式、认证材料等敏感内容时，应记录引用关系、摘要和操作结论，而不是复制敏感原文。

### 10.4 治理动作与功能限制边界

治理动作应按最小必要原则限制用户能力，避免一次处罚直接阻断所有平台功能。申诉救济入口应在限制状态下保留，确保用户可以对管理处理结果提出申诉。

| `actionType` | 功能限制边界 |
|---|---|
| `WARNING` | 只记录警告和通知用户，不限制主要功能 |
| `MUTE` | 限制发布需求、发送站内消息、提交评价、创建案件；允许登录、浏览、查看历史记录和查看处理结果 |
| `SUSPEND_ACCOUNT` | 限制发布、联系、发送消息、评价、创建案件、联系方式查看；允许登录查看基本状态、处理结果和申诉入口 |
| `RESTORE_ACCOUNT` | 解除此前 `MUTE` 或 `SUSPEND_ACCOUNT` 的限制，使账号恢复对应能力 |
| `LIMIT_REPORTING` | 只限制创建案件、举报和申诉能力，不影响发布、消息和评价 |
| `ADJUST_CREDIT` | 只影响信用摘要或信用纠偏记录，不直接限制账号功能 |

`MUTE` 与 `SUSPEND_ACCOUNT` 都可以是临时处置或长期处置。临时处置需要设置 `effectiveUntil`；长期处置需要明确原因和恢复路径。后端权限校验不得只依赖客户端隐藏入口，必须在发布、联系、消息、评价、案件创建、联系方式查看等接口中检查当前用户是否存在仍在生效期内的限制动作。

如果同一用户同时存在多个生效中的治理动作，按更严格的限制执行。例如用户同时存在 `MUTE` 和 `LIMIT_REPORTING`，则既不能发送消息、发布、评价，也不能创建案件；但仍可登录查看状态和必要的处理结果。

受限用户对管理处理结果的申诉入口需要保留。即使账号处于 `SUSPEND_ACCOUNT`，也应允许其针对相关 `ADMIN_DECISION` 创建申诉案件；但若用户被单独执行 `LIMIT_REPORTING`，则是否允许继续申诉需要由管理端治理模块根据处置原因决定，并至少保留联系管理员或查看处理说明的入口。

### 10.5 管理员角色与权限边界

V1 管理端治理采用两级管理员角色：`ADMIN` 与 `SUPER_ADMIN`。不在 V1 引入更细的审核管理员、案件管理员、信用管理员等复杂角色体系。

`ADMIN` 负责日常审核、案件处理和轻度治理：

- 查看认证审核、需求审核和案件队列。
- 审核认证资料和需求发布。
- 查看案件详情、证据和回应。
- 要求案件相关方补充材料或回应。
- 裁定普通案件。
- 执行 `WARNING`、`MUTE`、`LIMIT_REPORTING` 等轻度治理动作。
- 执行 `HIDE_REVIEW_TAGS`、`INVALIDATE_REVIEW` 等由案件裁定触发的评价处理动作。

`SUPER_ADMIN` 拥有 `ADMIN` 全部权限，并额外负责高风险治理动作：

- 执行 `SUSPEND_ACCOUNT`。
- 执行 `RESTORE_ACCOUNT`。
- 执行 `ADJUST_CREDIT`。
- 查看和撤销治理动作。
- 管理管理员账号或分配管理员角色。

权限边界遵循高风险动作上收原则。封禁、恢复账号、人工信用干预、撤销治理动作和管理员账号管理均不得由普通 `ADMIN` 执行。后端接口必须进行角色校验；管理端 UI 隐藏按钮不能替代后端授权。

所有管理员关键操作均需写入审计日志，包括操作者、目标对象、动作类型、原因、时间、来源案件或来源审核记录。审计日志不得记录不必要的敏感原文。

### 10.6 信用干预边界

`ADJUST_CREDIT` 属于高风险治理动作，仅允许 `SUPER_ADMIN` 执行。V1 不允许管理员直接手填某个用户的最终平均星数，例如直接将用户信用分改为 5.8。信用干预必须通过结构化动作影响信用摘要计算，保证可解释、可审计、可回溯。

`ADJUST_CREDIT` 允许的结构化动作包括：

| 干预动作 | 说明 |
|---|---|
| 撤销某条评价影响 | 使指定 `Review` 不再计入平均星数和标签统计 |
| 恢复某条评价影响 | 将此前被撤销的 `Review` 重新纳入信用摘要 |
| 增加人工信用调整记录 | 以小幅正负调整影响信用摘要，例如 `+0.2` 或 `-0.2` |
| 隐藏或恢复某些标签展示 | 不改变星级，只影响信用卡片标签展示或计数 |

每次信用干预必须记录：

| 字段 | 含义 |
|---|---|
| `targetUserId` | 被干预用户 |
| `adjustmentType` | 干预动作类型 |
| `adjustmentValue` | 调整数值，只有人工分值调整时需要 |
| `sourceType` / `sourceId` | 来源案件、管理复核或人工说明 |
| `reasonCode` | 原因类型 |
| `reasonNote` | 管理员说明 |
| `createdBy` / `createdAt` | 操作者和时间 |

人工信用调整值应限制在小范围内。V1 建议单次 `adjustmentValue` 取值范围为 `-0.5` 到 `+0.5`，并进入 `CreditChangeLog` 或等价信用变化记录。若需要更大幅度纠偏，应通过撤销/恢复具体评价、隐藏恶意标签或多个有来源的治理动作表达，而不是一次性手填最终分。

信用摘要展示时可以保留“含人工调整”弱提示，但不需要在普通信用卡片中展开每条管理记录。用户查看自己的信用明细时，可以看到人工调整的时间、原因摘要和来源类型；敏感案件细节仍按权限控制。

### 10.7 恶意举报处罚梯度

恶意举报、恶意申诉和批量针对性举报的治理目标不是压制正常反馈，而是防止投诉申诉入口被用作攻击他人的工具。管理员只有在已有案件材料、证据矛盾、上下文记录、批量行为线索或复核结论足以支持时，才能将案件标记为恶意举报。普通证据不足、事实理解偏差或情绪化表达不应直接等同恶意举报。

V1 按四档处罚梯度处理恶意举报：

| 档位 | 典型情形 | 治理动作 | 执行权限 |
|---|---|---|---|
| 轻微恶意举报 | 第一次轻微恶意举报、夸大事实但危害较低 | `WARNING` | `ADMIN` |
| 重复或明显捏造 | 重复提交无事实依据案件，或已有材料显示存在明显捏造 | `LIMIT_REPORTING` 7 天 | `ADMIN` |
| 批量或伪造证据 | 组织多人针对同一用户批量举报，或提交伪造证据但未达到长期封禁程度 | `MUTE` 7 天 + `LIMIT_REPORTING` 30 天 | `ADMIN` |
| 严重恶意行为 | 严重伪造证据、持续攻击他人、规避限制、造成明显平台治理风险 | `SUSPEND_ACCOUNT` | `SUPER_ADMIN` |

恶意举报处罚应优先关联来源案件。若 `CaseTicket.decisionResult = MARK_MALICIOUS` 或 `ESCALATE_GOVERNANCE`，管理端可以基于该案件创建一个或多个 `GovernanceAction`，并在 `sourceType = CASE_TICKET`、`sourceId = caseId` 中保留来源。批量恶意举报中涉及多个账号时，应分别生成各自的治理动作，不用一个治理动作同时处罚多个用户。

处罚可以叠加，但必须可解释。例如同一用户因伪造证据受到 `MUTE` 7 天，同时因滥用举报入口受到 `LIMIT_REPORTING` 30 天；功能限制按更严格规则执行。处罚到期后，临时限制自动失效，但治理记录和审计日志保留。若后续复核发现误判，应由 `SUPER_ADMIN` 撤销或修正对应 `GovernanceAction`，不得物理删除原记录。

系统需要避免把“败诉”自动推导为恶意举报。案件被驳回、证据不足、对方回应更可信，只能说明案件未成立；只有存在捏造、伪造、批量攻击、反复滥用或明显主观恶意线索时，才进入恶意举报处罚梯度。

### 10.8 管理端审计日志

管理端审计日志统一由 `AdminAuditLog` 承载。所有管理员关键写操作都必须写入审计日志，不能只依赖普通运行日志、业务表更新时间或管理员备注字段。审计日志的目标是支持事后追责、处理复核、误判纠正和平台治理透明化。

`AdminAuditLog` 至少包含：

| 字段 | 含义 | 说明 |
|---|---|---|
| `id` | 审计日志 ID | 系统生成 |
| `actorAdminId` | 操作管理员 ID | 执行动作的管理员 |
| `actorRole` | 操作时角色 | `ADMIN` 或 `SUPER_ADMIN` |
| `actionType` | 操作类型 | 认证审核、需求审核、案件裁定、治理动作、撤销治理动作、信用干预等 |
| `targetType` | 目标对象类型 | `USER`、`PARTNER_POST`、`CASE_TICKET`、`REVIEW`、`GOVERNANCE_ACTION` 等 |
| `targetId` | 目标对象 ID | 被审核、被处理或被调整的对象 |
| `sourceType` | 来源类型 | 来源案件、人工复核、系统检测、审核队列等 |
| `sourceId` | 来源 ID | 可为空；例如 `CaseTicket.id` |
| `reasonCode` | 原因类型 | 预设原因码 |
| `reasonSummary` | 原因摘要 | 面向复核的简短说明，不记录敏感原文 |
| `beforeSnapshotRef` | 操作前快照引用 | 可为空；指向受控快照或摘要记录 |
| `afterSnapshotRef` | 操作后快照引用 | 可为空；指向受控快照或摘要记录 |
| `traceId` | 请求追踪 ID | 与接口错误响应或服务端链路追踪对齐 |
| `clientIp` | 操作来源 IP | 管理端安全审计使用 |
| `createdAt` | 操作时间 | 审计记录创建时间 |

审计日志记录引用、摘要和操作结论，不直接复制敏感原文。涉及证据附件、联系方式、认证材料、聊天内容、对象存储文件等敏感内容时，`AdminAuditLog` 只能记录对象引用、摘要、哈希或快照引用；实际内容仍由原业务对象和权限控制接口提供受控访问。

必须写入 `AdminAuditLog` 的操作至少包括：

| 操作范围 | 典型动作 |
|---|---|
| 认证审核 | 认证资料通过、驳回、查看受控材料 |
| 需求审核 | 需求发布通过、驳回 |
| 案件处理 | 要求补充、查看完整证据、裁定结案、标记恶意举报 |
| 评价处理 | 隐藏评价标签、撤销评价影响、恢复评价影响 |
| 账号治理 | 创建 `GovernanceAction`、执行警告、禁言、限制举报、封禁、恢复账号 |
| 信用干预 | 创建人工信用调整、撤销或恢复评价影响、隐藏或恢复标签展示 |
| 权限管理 | 分配或撤销管理员角色 |

审计日志应尽量与业务写操作处于同一事务边界。若业务操作成功而审计日志写入失败，应视为本次管理操作失败并回滚，避免出现“有处置、无审计”的不可追责状态。只读查看类操作中，查看高度敏感材料也需要审计；普通列表查询可不逐条记录，避免日志膨胀。

普通用户不能直接查看完整 `AdminAuditLog`。用户可在自己的处理结果、信用明细或案件进度中看到必要的操作摘要，例如处理时间、处理结论、原因摘要和来源类型；完整审计日志只对具备权限的管理员或后续系统维护角色开放。

### 10.9 管理端治理接口契约

管理端治理接口按“治理动作接口”和“审计查询接口”拆分。治理动作接口负责创建、查看和撤销平台实际处置；审计查询接口负责查看管理员关键操作留痕。V1 不设计一个可修改任意对象状态的万能管理接口，避免权限边界、审计责任和测试入口混乱。

治理动作接口草案如下：

| 接口 | 用途 | 认证要求 | 关键规则 |
|---|---|---|---|
| `POST /api/admin/governance-actions` | 创建治理动作 | `ROLE_ADMIN`；高风险动作要求 `SUPER_ADMIN` | 根据 `actionType` 校验操作者角色、目标用户、来源对象和原因 |
| `POST /api/admin/governance-actions/{actionId}/reverse` | 撤销治理动作 | `SUPER_ADMIN` | 仅允许撤销仍可逆或仍在生效期内的治理动作；必须填写撤销原因 |
| `GET /api/admin/governance-actions` | 查询治理动作列表 | `ROLE_ADMIN` | 支持按目标用户、动作类型、生效状态、来源类型和时间范围筛选 |
| `GET /api/admin/governance-actions/{actionId}` | 查看治理动作详情 | `ROLE_ADMIN` | 返回治理动作、来源摘要、撤销信息和关联审计日志摘要 |

审计查询接口草案如下：

| 接口 | 用途 | 认证要求 | 关键规则 |
|---|---|---|---|
| `GET /api/admin/audit-logs` | 查询管理端审计日志 | `ROLE_ADMIN` | 支持按操作者、动作类型、目标类型、目标 ID、来源类型、时间范围和 `traceId` 筛选 |
| `GET /api/admin/audit-logs/{auditLogId}` | 查看审计日志详情 | `ROLE_ADMIN` | 返回审计字段、摘要和引用，不返回敏感原文 |

`CreateGovernanceActionRequest` 至少包含：

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `actionType` | string | 是 | `WARNING`、`MUTE`、`SUSPEND_ACCOUNT`、`RESTORE_ACCOUNT`、`ADJUST_CREDIT`、`LIMIT_REPORTING` |
| `targetUserId` | string | 是 | 被处置用户 |
| `sourceType` | string | 否 | `CASE_TICKET`、`MANUAL_REVIEW`、`SYSTEM_DETECTION` 等 |
| `sourceId` | string | 否 | 来源对象 ID |
| `reasonCode` | string | 是 | 预设原因码 |
| `reasonNote` | string | 是 | 管理员处置说明 |
| `effectiveFrom` | datetime | 否 | 默认立即生效 |
| `effectiveUntil` | datetime | 否 | 临时处置结束时间 |
| `creditAdjustment` | object | 否 | 仅 `ADJUST_CREDIT` 使用，承载结构化信用干预字段 |

`ReverseGovernanceActionRequest` 至少包含：

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `reverseReason` | string | 是 | 撤销原因 |

`GovernanceActionResponse` 至少包含 `GovernanceAction` 核心字段、当前是否生效、是否可撤销、来源对象摘要、创建管理员摘要、撤销信息和关联审计日志 ID。列表项可以返回摘要字段，但详情接口必须返回完整治理动作元数据。

创建治理动作时必须同步写入 `AdminAuditLog`。撤销治理动作时也必须写入新的审计日志，而不是修改原审计日志。原审计日志记录“当时执行了什么”，撤销审计日志记录“后来为什么撤销”，两者共同形成完整时间线。

治理动作接口必须执行以下权限规则：

| 操作 | 最低权限 |
|---|---|
| 创建 `WARNING`、`MUTE`、`LIMIT_REPORTING` | `ADMIN` |
| 创建 `SUSPEND_ACCOUNT`、`RESTORE_ACCOUNT`、`ADJUST_CREDIT` | `SUPER_ADMIN` |
| 撤销治理动作 | `SUPER_ADMIN` |
| 查询治理动作列表和详情 | `ADMIN` |
| 查询审计日志 | `ADMIN` |

接口统一使用项目错误响应结构 `code`、`message`、`details`、`traceId`。P1-3 至少需要约定以下错误码：

| 错误码 | 适用场景 |
|---|---|
| `UNAUTHORIZED` | 未登录访问管理端接口 |
| `FORBIDDEN` | 当前管理员角色不足，例如 `ADMIN` 执行 `SUSPEND_ACCOUNT` |
| `GOVERNANCE_ACTION_NOT_FOUND` | 治理动作不存在或不可见 |
| `GOVERNANCE_ACTION_NOT_REVERSIBLE` | 治理动作不可撤销、已撤销或已不适合撤销 |
| `GOVERNANCE_TARGET_NOT_FOUND` | 被处置用户不存在 |
| `GOVERNANCE_SOURCE_NOT_FOUND` | 来源案件、复核或系统检测记录不存在 |
| `GOVERNANCE_ACTION_CONFLICT` | 与现有生效治理动作冲突或重复 |
| `AUDIT_LOG_NOT_FOUND` | 审计日志不存在或不可见 |
| `VALIDATION_FAILED` | 请求字段、时间范围、原因码或信用干预结构校验失败 |

Qt 管理端客户端应分别封装 `AdminGovernanceApiService` 和 `AdminAuditApiService`。案件处理页面可以跳转创建治理动作，但不应直接调用审计写入接口；审计日志由后端在管理写操作成功时自动生成。

### 10.10 测试入口与后续实现批次

管理端治理模块后续实现按 3 个批次推进，每批必须遵守测试先行、红灯确认、最小实现、绿灯确认、回归测试和测试留档规则。P1-3 不与案件模块、评价模块或通知模块混成一个大批次；案件裁定、评价状态和通知触发只作为必要挂接点参与验证。

| 批次 | 范围 | 主要测试入口 | 阶段闭环 |
|---|---|---|---|
| Batch 1 | 后端 `GovernanceAction` 创建、撤销、查询和权限校验 | JUnit 5 + Spring Boot 接口/安全测试 | 管理员可按权限执行治理动作，限制规则可被业务接口识别 |
| Batch 2 | 后端 `AdminAuditLog` 自动写入、审计查询、事务一致性 | JUnit 5 + Spring Boot 接口/事务测试 | 管理端关键写操作均有审计，审计失败时业务操作回滚 |
| Batch 3 | Qt 管理端治理页面与审计日志查看页面 | Qt Test + 必要手工验证清单 | 管理端可查看用户治理历史、创建/撤销治理动作、查询审计日志 |

Batch 1 的后端测试至少覆盖：

- 非管理员访问治理动作接口返回 `FORBIDDEN`。
- `ADMIN` 可创建 `WARNING`、`MUTE`、`LIMIT_REPORTING`。
- `ADMIN` 创建 `SUSPEND_ACCOUNT`、`RESTORE_ACCOUNT`、`ADJUST_CREDIT` 返回 `FORBIDDEN`。
- `SUPER_ADMIN` 可创建高风险治理动作。
- 创建临时 `MUTE` 后，发布、发送站内消息、提交评价和创建案件等接口能识别生效限制。
- 创建 `LIMIT_REPORTING` 后，案件创建接口被限制，但发布、消息和评价不受该动作影响。
- 创建 `SUSPEND_ACCOUNT` 后，关键互动能力被阻断，但登录、查看处理结果和申诉入口保留。
- 重复或冲突治理动作返回 `GOVERNANCE_ACTION_CONFLICT`。
- `SUPER_ADMIN` 可撤销可逆治理动作，撤销后限制失效。
- 普通 `ADMIN` 撤销治理动作返回 `FORBIDDEN`。

Batch 2 的后端测试至少覆盖：

- 创建治理动作成功后自动写入 `AdminAuditLog`。
- 撤销治理动作成功后写入新的 `AdminAuditLog`，不覆盖原审计记录。
- 认证审核、需求审核、案件裁定、评价处理和信用干预等关键管理写操作均写入审计日志。
- 管理员查看高度敏感材料时写入审计日志。
- 普通列表查询不逐条写入审计日志，避免日志膨胀。
- 审计查询接口支持按操作者、动作类型、目标对象、来源对象、时间范围和 `traceId` 筛选。
- 审计详情不返回证据原文、联系方式、认证材料原文或对象存储裸地址。
- 模拟审计写入失败时，业务管理操作回滚，避免有处置无审计。

Batch 3 的 Qt 客户端测试与验证至少覆盖：

- 管理端用户治理详情页展示当前生效限制、历史治理动作和来源摘要。
- 管理员可创建 `WARNING`、`MUTE`、`LIMIT_REPORTING`。
- 超级管理员可创建 `SUSPEND_ACCOUNT`、`RESTORE_ACCOUNT`、`ADJUST_CREDIT`。
- 普通管理员界面不展示或不可操作高风险动作，后端仍以权限校验为准。
- 超级管理员可撤销治理动作并填写撤销原因。
- 审计日志页面支持筛选、分页和查看详情。
- 审计日志详情仅展示引用、摘要和结论，不展示敏感原文。

本模块后续测试留档应写入 `docs/validation/`，每批至少记录对应需求编号、设计入口、测试文件、红灯结果、实现摘要、绿灯结果、回归范围和未覆盖风险。

### 10.11 管理端治理核心用例

P1-3 管理端治理 V1 只收敛 6 个核心用例，不扩展为完整后台运营系统。管理员账号管理、复杂风控看板、批量处置、统计分析和多级审批流暂不进入本轮主链路。

核心用例如下：

| 用例 | 主要参与者 | 触发入口 | 结果 |
|---|---|---|---|
| 查看用户治理档案 | `ADMIN`、`SUPER_ADMIN` | 管理端用户详情、案件详情跳转 | 查看用户当前生效限制、历史治理动作、来源案件摘要和信用干预摘要 |
| 创建治理动作 | `ADMIN`、`SUPER_ADMIN` | 用户治理档案、案件处理结果页 | 生成 `GovernanceAction`，同步写入 `AdminAuditLog`，限制规则开始生效 |
| 撤销治理动作 | `SUPER_ADMIN` | 治理动作详情 | 写入撤销信息和新的 `AdminAuditLog`，对应限制失效或信用干预回退 |
| 查看审计日志 | `ADMIN`、`SUPER_ADMIN` | 审计日志页面、治理动作详情跳转 | 查询管理员关键操作留痕，查看摘要、引用和追踪信息 |
| 从案件升级治理 | `ADMIN`、`SUPER_ADMIN` | 案件裁定页 | 基于 `CaseTicket` 创建治理动作，保留 `sourceType/sourceId` 关联 |
| 查看受限用户申诉入口 | 受限学生、`ADMIN`、`SUPER_ADMIN` | 用户端处理结果页、管理端治理档案 | 用户可看到必要申诉入口，管理员可看到该限制是否允许申诉 |

用例边界说明：

- 查看用户治理档案只展示治理相关摘要，不替代用户完整个人资料管理。
- 创建治理动作必须经过后端角色校验；前端隐藏按钮不能替代权限控制。
- 撤销治理动作不删除原记录，只追加撤销字段和撤销审计记录。
- 查看审计日志只展示引用、摘要和结论，不展示敏感原文。
- 从案件升级治理必须保留来源案件关联，不能生成无来源的高风险处罚。
- 查看受限用户申诉入口用于保证救济链路，不代表所有限制状态下都允许无限次申诉。

### 10.12 活动流程与顺序图说明

管理端治理 V1 的图类说明优先覆盖两条高风险主流程：从案件升级治理、撤销治理动作。查询治理档案、查询审计日志、查看受限用户申诉入口等流程以用例和接口契约说明为主，不在 V1 逐条展开活动图。

#### 10.12.1 从案件升级治理活动流程

从案件升级治理用于承接投诉申诉与案件模块中的管理裁定结果。当管理员在案件裁定中选择 `ESCALATE_GOVERNANCE`，或在案件关闭后发现需要进一步账号处置时，应通过治理动作接口创建 `GovernanceAction`。

活动流程如下：

1. 管理员打开案件详情，查看案件目标、双方材料、证据摘要和历史处理记录。
2. 管理员在案件裁定或案件处理结果页选择升级治理。
3. 前端带入 `sourceType = CASE_TICKET`、`sourceId = caseId`，进入创建治理动作表单。
4. 管理员选择 `actionType`、填写 `reasonCode`、`reasonNote`，必要时填写 `effectiveUntil` 或信用干预结构。
5. 后端校验操作者角色、案件可感知性、目标用户存在性、治理动作权限和动作冲突。
6. 后端创建 `GovernanceAction`，并使对应功能限制或信用干预生效。
7. 后端在同一事务中写入 `AdminAuditLog`。
8. 前端刷新用户治理档案、案件详情和必要的处理结果摘要。

失败分支至少包括：

- 操作者角色不足：返回 `FORBIDDEN`。
- 来源案件不存在或不可见：返回 `GOVERNANCE_SOURCE_NOT_FOUND`。
- 目标用户不存在：返回 `GOVERNANCE_TARGET_NOT_FOUND`。
- 治理动作与现有生效动作冲突：返回 `GOVERNANCE_ACTION_CONFLICT`。
- 请求字段、原因码、时间范围或信用干预结构不合法：返回 `VALIDATION_FAILED`。
- 审计日志写入失败：本次治理动作创建回滚，并返回服务端错误或统一失败响应。

#### 10.12.2 从案件升级治理顺序图说明

顺序图参与者建议包括：`AdminUser`、`QtAdminClient`、`AdminGovernanceController`、`GovernanceActionService`、`CaseTicketRepository`、`GovernanceActionRepository`、`AdminAuditLogService`、`RestrictionPolicyService`。

顺序主干：

1. `AdminUser` 在 `QtAdminClient` 中从案件详情点击升级治理。
2. `QtAdminClient` 调用 `POST /api/admin/governance-actions`。
3. `AdminGovernanceController` 校验认证身份并传入当前管理员上下文。
4. `GovernanceActionService` 校验 `sourceType/sourceId` 指向的案件存在且可用于治理升级。
5. `GovernanceActionService` 校验 `actionType` 与管理员角色匹配。
6. `GovernanceActionService` 检查目标用户和现有生效治理动作，避免冲突。
7. `GovernanceActionService` 保存 `GovernanceAction`。
8. `RestrictionPolicyService` 使限制策略或信用干预挂接生效。
9. `AdminAuditLogService` 写入 `AdminAuditLog`。
10. 后端返回 `GovernanceActionResponse`，前端刷新相关页面。

该顺序图必须体现：治理动作创建和审计日志写入处于同一事务边界；审计失败时，治理动作不得留下半成功记录。

#### 10.12.3 撤销治理动作活动流程

撤销治理动作用于纠正误判、解除不再适用的限制或回退特定信用干预。V1 中撤销治理动作只允许 `SUPER_ADMIN` 执行。

活动流程如下：

1. 超级管理员打开用户治理档案或治理动作详情。
2. 系统展示该治理动作是否仍可撤销，以及撤销后影响范围。
3. 超级管理员点击撤销，填写 `reverseReason`。
4. 后端校验操作者为 `SUPER_ADMIN`。
5. 后端校验治理动作存在、未撤销、仍可逆且撤销不会破坏业务一致性。
6. 后端写入 `reversedBy`、`reversedAt`、`reverseReason`。
7. 后端使对应限制失效，或对信用干预执行回退/重算。
8. 后端写入新的 `AdminAuditLog`，记录撤销操作。
9. 前端刷新用户治理档案、治理动作详情和必要的信用摘要。

失败分支至少包括：

- 非 `SUPER_ADMIN` 撤销：返回 `FORBIDDEN`。
- 治理动作不存在：返回 `GOVERNANCE_ACTION_NOT_FOUND`。
- 治理动作不可撤销、已撤销或不适合撤销：返回 `GOVERNANCE_ACTION_NOT_REVERSIBLE`。
- 撤销原因为空或过短：返回 `VALIDATION_FAILED`。
- 撤销审计日志写入失败：撤销操作回滚，原治理动作保持未撤销状态。

#### 10.12.4 撤销治理动作顺序图说明

顺序图参与者建议包括：`SuperAdminUser`、`QtAdminClient`、`AdminGovernanceController`、`GovernanceActionService`、`GovernanceActionRepository`、`RestrictionPolicyService`、`CreditSummaryService`、`AdminAuditLogService`。

顺序主干：

1. `SuperAdminUser` 在治理动作详情页提交撤销请求。
2. `QtAdminClient` 调用 `POST /api/admin/governance-actions/{actionId}/reverse`。
3. `AdminGovernanceController` 校验认证身份和 `SUPER_ADMIN` 角色。
4. `GovernanceActionService` 读取治理动作并校验可撤销性。
5. `GovernanceActionService` 写入撤销字段。
6. `RestrictionPolicyService` 解除账号限制类动作的生效影响。
7. 若撤销对象涉及 `ADJUST_CREDIT`，`CreditSummaryService` 触发信用摘要重算或回退。
8. `AdminAuditLogService` 写入撤销审计日志。
9. 后端返回更新后的 `GovernanceActionResponse`，前端刷新治理档案和详情。

该顺序图必须体现：撤销不是删除；原治理动作保留，撤销信息和新的审计记录共同构成复核链路。

### 10.13 管理端治理关键对象

管理端治理 V1 的关键对象收束为 4 个：`GovernanceAction`、`AdminAuditLog`、`RestrictionPolicy`、`CreditAdjustmentRecord`。这 4 个对象分别回答“平台做了什么处置”“谁做了什么操作”“用户当前受什么限制”“信用摘要为什么变化”。

| 对象 | 定位 | 持久化要求 | 说明 |
|---|---|---|---|
| `GovernanceAction` | 平台实际处置记录 | 必须持久化 | 记录警告、禁言、封禁、恢复、信用干预、限制举报等治理动作 |
| `AdminAuditLog` | 管理员操作留痕 | 必须持久化 | 记录管理员关键操作、目标、来源、原因摘要和追踪信息 |
| `RestrictionPolicy` | 当前功能限制策略 | 可作为计算视图 | 从仍在生效期内的 `GovernanceAction` 计算当前用户能否发布、联系、消息、评价、创建案件等 |
| `CreditAdjustmentRecord` | 信用干预明细 | 建议持久化 | 承接 `ADJUST_CREDIT` 的结构化调整，支撑信用摘要重算和用户信用明细展示 |

`GovernanceAction` 是治理模块的核心业务对象，表达平台已经或正在执行的处置动作。它不直接替代案件裁定、评价状态或用户认证状态；案件模块给出结论，治理模块记录处置。账号功能限制、信用摘要变化和用户可见处理结果都应能追溯到一个或多个 `GovernanceAction`。

`AdminAuditLog` 是追责对象，不是业务状态对象。它记录管理员何时、以什么角色、对什么目标、基于什么来源和原因执行了什么操作。撤销治理动作时，不修改原审计日志表达“它没有发生过”，而是新增一条撤销审计日志，让复核链路完整保留。

`RestrictionPolicy` 不要求 V1 建独立数据表。后端可以通过策略服务实时读取用户生效中的 `GovernanceAction`，计算出当前用户是否允许执行某类能力。它至少需要覆盖以下能力位：

| 能力位 | 说明 |
|---|---|
| `canPublishPost` | 是否可发布或提交需求 |
| `canStartConversation` | 是否可发起联系 |
| `canSendMessage` | 是否可发送站内消息 |
| `canSubmitReview` | 是否可提交或修改评价 |
| `canCreateCase` | 是否可创建投诉、举报或申诉案件 |
| `canViewContact` | 是否可查看已解锁联系方式 |
| `canAppealAdminDecision` | 是否可针对管理处理结果发起申诉 |

当多个治理动作同时生效时，`RestrictionPolicy` 按更严格限制计算。若 `MUTE` 限制消息和评价，`LIMIT_REPORTING` 限制案件创建，则最终策略同时限制这些能力；若 `SUSPEND_ACCOUNT` 生效，则关键互动能力均应被阻断，但仍保留登录、查看处理结果和必要申诉入口。

`CreditAdjustmentRecord` 用于承接 `ADJUST_CREDIT` 的结构化信用干预，不允许直接保存“最终信用分”。它至少包含：

| 字段 | 含义 |
|---|---|
| `id` | 信用干预记录 ID |
| `governanceActionId` | 关联治理动作 |
| `targetUserId` | 被干预用户 |
| `adjustmentType` | 干预类型，例如撤销评价影响、恢复评价影响、人工小幅调整、隐藏标签 |
| `adjustmentValue` | 人工小幅调整值，仅对应类型需要 |
| `reviewId` | 涉及具体评价时使用 |
| `tagName` | 涉及标签隐藏或恢复时使用 |
| `reasonCode` | 原因类型 |
| `reasonSummary` | 用户或管理员可见的摘要说明 |
| `createdBy` / `createdAt` | 操作者和时间 |
| `reversedBy` / `reversedAt` | 撤销者和撤销时间，可为空 |

信用摘要计算时，应把 `Review`、默认有效对话样本、初始虚拟评分基线和 `CreditAdjustmentRecord` 一并纳入计算口径。普通信用卡片只需要展示“含人工调整”弱提示；用户查看自己的信用明细时，可看到人工调整摘要，但不能越权看到敏感案件证据或管理员内部审计原文。

### 10.14 治理操作与信用摘要变化关系

信用摘要变化只允许由结构化信用干预触发，不由所有账号处罚自动触发。`WARNING`、`MUTE`、`SUSPEND_ACCOUNT`、`RESTORE_ACCOUNT`、`LIMIT_REPORTING` 本身不直接改变用户平均星数、标签计数或总对话次数；它们只影响用户当前可执行的功能能力。

只有 `actionType = ADJUST_CREDIT` 且生成 `CreditAdjustmentRecord` 时，才允许影响信用摘要。V1 支持 3 类信用变化：

| 信用变化类型 | `adjustmentType` 建议值 | 影响范围 |
|---|---|---|
| 撤销或恢复评价影响 | `INVALIDATE_REVIEW_EFFECT`、`RESTORE_REVIEW_EFFECT` | 指定 `Review` 是否计入平均星数和标签统计 |
| 隐藏或恢复评价标签 | `HIDE_REVIEW_TAGS`、`RESTORE_REVIEW_TAGS` | 指定标签是否进入信用卡片高频标签计数，不影响星级 |
| 人工小幅分值调整 | `MANUAL_SCORE_DELTA` | 在允许范围内对信用摘要做小幅正负调整 |

该规则意味着：管理员若要“处罚账号”，应创建 `MUTE`、`LIMIT_REPORTING` 或 `SUSPEND_ACCOUNT` 等治理动作；若要“纠正信用摘要”，应另行创建 `ADJUST_CREDIT`，并写入结构化 `CreditAdjustmentRecord`。二者可以来自同一个案件，但必须是两条可审计、可解释的动作记录。

信用摘要计算口径如下：

1. 读取被评价用户收到的有效 `Review`。
2. 排除已被撤销影响的评价。
3. 对无显式评价的真实有效对话按默认 4 星计入。
4. 加入初始虚拟评分基线：3 次 3 星样本和 3 次 4 星样本。
5. 根据 `CreditAdjustmentRecord` 应用人工小幅分值调整。
6. 统计未被隐藏的评价标签，生成 `标签名*次数`。
7. 输出平均星数、真实有效总对话次数、高频标签和是否含人工调整提示。

人工小幅分值调整不得改变真实有效总对话次数，也不得绕过评价和默认对话样本直接覆盖最终平均星数。若某条评价被撤销影响，该评价对应星级和标签均不参与信用摘要；若只是隐藏标签，则星级仍参与平均星数，标签不参与展示和计数。

撤销 `ADJUST_CREDIT` 时，应根据 `CreditAdjustmentRecord.adjustmentType` 回退对应影响：恢复被撤销的评价影响、取消人工分值调整、恢复被隐藏的标签，或重新执行信用摘要重算。撤销信用干预也必须写入新的 `AdminAuditLog`，并在用户信用明细中展示可理解的撤销摘要。

用户信用明细可展示信用变化摘要，但不展示管理员内部审计原文或敏感案件证据。示例展示口径包括：评价影响已撤销、评价标签已隐藏、平台人工调整 `+0.2`、平台人工调整已撤销等。
