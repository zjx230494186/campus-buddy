### 4.4 P0-1 账号准入与身份认证

#### 4.4.1 模块定位

账号准入与身份认证模块负责用户进入平台前后的基础身份闭环，覆盖注册/登录、JWT 签发与刷新、校内身份认证资料提交、认证状态回显、退出登录与令牌撤销。

本模块处理学生侧的身份准入、登录态维护和认证状态感知；管理员对认证资料的审核动作归入管理端治理模块，但其审核结果会回写并影响本模块中的用户认证状态。

#### 4.4.2 参与角色

- 未认证用户：可注册、登录、提交校内身份认证资料、查看认证状态。
- 已认证学生：可保持登录态、刷新令牌、退出登录，并获得后续 P0 主链路能力入口。
- 管理员：不直接属于本模块主操作角色，但其审核结果影响用户认证状态。

未认证用户允许浏览广场和需求详情，但不得发布需求、发起联系、解锁联系方式、评价或投诉。发布、联系、解锁联系方式、评价和投诉均要求用户处于 `VERIFIED` 状态；`SUSPENDED` 用户按治理限制处理。

#### 4.4.3 核心对象

- `User`
- `UserProfile`
- `AuthenticationRecord`
- `CampusEmailVerification`
- `Attachment`
- `RefreshTokenRecord`

#### 4.4.4 状态与约束

本模块至少涉及用户认证状态和令牌状态。

用户认证状态采用以下状态集合：

| 状态 | 含义 | 说明 |
|---|---|---|
| `UNVERIFIED` | 未认证 | 用户已注册或可登录，但尚未提交校内身份认证资料 |
| `PENDING_REVIEW` | 待审核 | 用户已提交认证资料，等待管理员审核 |
| `VERIFIED` | 已认证 | 管理员审核通过，用户获得已认证学生能力 |
| `REJECTED` | 已驳回 | 管理员驳回认证申请，用户可根据规则重新提交 |
| `SUSPENDED` | 已限制 | 治理处置导致账号能力受限，不等同于认证失败 |

认证状态流转规则如下：

| 起始状态 | 触发动作 | 目标状态 | 说明 |
|---|---|---|---|
| `UNVERIFIED` | 完成校园邮箱验证并提交校内身份认证资料 | `PENDING_REVIEW` | 用户完成邮箱验证后首次提交认证资料 |
| `PENDING_REVIEW` | 管理员审核通过 | `VERIFIED` | 用户获得已认证学生能力 |
| `PENDING_REVIEW` | 管理员审核驳回 | `REJECTED` | 用户认证申请被驳回 |
| `REJECTED` | 用户重新提交认证资料 | `PENDING_REVIEW` | 允许根据规则重新提交 |
| `VERIFIED` | 管理端治理限制 | `SUSPENDED` | 仅由治理动作触发，不属于普通认证审核 |

普通认证流程不得直接产生 `SUSPENDED` 状态。

令牌状态需要支持签发、刷新、轮换、撤销和过期。

#### 4.4.5 主要流程

1. 用户通过邮箱密码或邮箱验证码注册/登录。
2. 后端签发 access token 与 refresh token。
3. 未认证用户提交个人身份信息与校内身份材料。
4. 用户完成校园邮箱验证。
5. 用户查看认证状态。
6. 管理员审核认证资料并回写认证结果。
7. 用户刷新令牌或退出登录。

V1 身份验证由个人信息、校内身份材料和校园邮箱验证共同构成。手机号可作为账号联系方式或辅助信息保存，但不作为登录凭证，也不作为校内身份认证的证明依据。用户必须先完成校园邮箱验证，才能提交认证资料并进入 `PENDING_REVIEW`；未完成校园邮箱验证的用户仍停留在 `UNVERIFIED`。

校内身份认证资料提交不设置重新提交次数限制或冷却时间。系统应在提交界面明确标注需要提交的材料，并在客户端与后端进行必填项完整性校验；资料不完整时不得打包提交。认证被驳回后，用户可根据管理员驳回原因补充或更换材料后重新提交。

认证资料被驳回后，系统不保留原提交的敏感资料内容，也不提供历史提交材料回看能力。被驳回申请中的个人敏感明细和材料附件应被删除或解除业务引用，避免长期留存学生个人信息。系统仅保留当前认证状态、驳回原因、处理时间等最小必要的非敏感结果信息，用于用户理解驳回原因并重新提交。

认证审核通过后，系统不长期保留学生证、校园卡等认证材料图片或 PDF 原件。审核完成后应删除认证材料原件或解除业务引用，仅保留认证结果、认证时间、认证方式、校园邮箱和必要的脱敏身份摘要。后续日常业务不得依赖回看认证材料原件；若出现身份争议，应通过重新认证或人工补充材料处理。

处于 `PENDING_REVIEW` 状态的认证材料不设置额外的最长保留时间，也不新增“审核逾期”状态。V1 按管理流程要求在较短时间内完成人工审核，通常不超过 1 天；当前不为低概率逾期场景引入定时过期、自动删除和重新提交流程。

校园邮箱验证采用学校邮箱域名白名单 + 邮件验证码方式：系统先校验邮箱域名是否属于允许的学校域名，再向该邮箱发送验证码，用户完成验证码校验后视为完成校园邮箱验证。

校园邮箱验证码有效期为 10 分钟；同一邮箱验证码发送后 60 秒内不得重复发送；同一验证码连续错误 5 次后失效，需要重新发送验证码。

邮箱验证码登录复用同一套验证码安全规则，即 10 分钟有效、60 秒重发间隔、连续错误 5 次失效。登录验证码与校园邮箱认证验证码用途不同，后端需要区分验证码使用场景。

邮箱密码登录要求密码长度至少 8 位。后端使用 BCrypt 存储密码哈希，不保存明文密码，不自研密码哈希方案。

密码登录需要设置失败限制。连续 5 次密码登录失败后，系统对该账号的密码登录临时锁定 15 分钟；锁定期内拒绝继续使用密码登录。密码登录成功后，清零该账号的密码登录失败计数。邮箱验证码登录不使用账号锁定机制，继续按验证码有效期、重发间隔和错误次数失效规则处理。

V1 提供密码重置流程。用户通过邮箱验证码验证后重置密码，验证码规则复用邮箱验证码安全规则；密码重置后新密码仍需满足至少 8 位并使用 BCrypt 哈希存储。

密码重置成功后，后端必须撤销该用户全部已签发的 refresh token。其他设备在 access token 过期后不得继续刷新登录态，需要重新登录。该规则用于降低密码泄露、邮箱验证码找回后旧设备继续持有长期登录态的安全风险。

V1 令牌有效期采用：access token 有效期 30 分钟，refresh token 有效期 14 天。用户刷新登录态时，后端签发新的 access token，并轮换 refresh token；旧 refresh token 在成功刷新后立即失效，不得再次用于续期。服务端需要保存 refresh token 的有效状态、过期时间、撤销时间和轮换关系，以支持退出登录、密码重置、异常 token 复用识别和后续设备管理扩展。

用户普通退出登录时，仅撤销当前设备或当前会话对应的 refresh token，不影响同一账号在其他设备上的有效登录态。系统另设“退出所有设备”能力，用于撤销该账号当前全部有效 refresh token。该规则要求 `RefreshTokenRecord` 能够区分会话或设备标识，并记录 token 签发来源、失效原因和撤销时间。

同一账号允许多设备或多客户端会话同时在线。V1 正式展示与主要实现重心仍以 Win11 桌面 PC 端为主，不要求实现完整设备管理页面，也不将移动端作为 V1 必交付范围；但后端账号、令牌和接口设计需要保留后续移动端接入空间，避免将登录态、会话记录和业务接口绑定为单一 PC 客户端专用实现。

校园邮箱域名白名单 V1 按单校配置，只允许项目目标学校的官方邮箱域名；配置方式应保留后续扩展为多校域名配置表的空间。

校园邮箱作为校内身份的关键依据，V1 强制遵循一人一校园邮箱一账号原则。用户更换校园邮箱后，原账号信息保留，但必须重新按照注册/认证流程完成校园邮箱验证、认证资料提交和管理员审核；重新认证完成前不得获得已认证学生互动能力。

同一个校园邮箱不得重复注册多个账号。注册、校园邮箱验证和邮箱更换流程均需要校验校园邮箱唯一性；若用户遗忘历史账号或密码，应通过账号找回或密码重置流程恢复原账号访问，而不是使用同一校园邮箱创建新账号。该约束用于保持认证记录、信用摘要、投诉治理和历史业务记录的连续性。

用户更换校园邮箱后，认证状态回退到 `UNVERIFIED`。系统保留其历史发布、会话、评价和信用摘要，但冻结发布、发起联系、解锁联系方式、评价和投诉等互动能力，直到重新完成认证并回到 `VERIFIED` 状态。

邮箱更换后的重新认证期间，历史发布继续对外展示，但需要标记账号认证状态变化；其他用户不得基于这些历史发布发起新的联系或解锁联系方式。已存在的历史会话和评价链路保留。

#### 4.4.6 接口与客户端关系

Qt 客户端通过 API Client 层调用账号与认证相关 REST 接口。令牌保存通过 `SecureTokenStore` 完成，普通配置不得保存 token。认证失败、令牌过期和权限不足均按统一错误响应结构处理。

#### 4.4.7 测试与追踪入口

后续测试至少覆盖注册/登录、校园邮箱唯一性、密码登录失败锁定、多会话登录、令牌刷新、refresh token 轮换、旧 refresh token 复用失败、普通退出当前会话、退出所有设备、认证资料提交、认证资料完整性校验、认证驳回后敏感资料删除、认证通过后材料原件删除、认证状态查询、认证状态权限限制、令牌撤销和密码重置后 refresh token 全部失效。当前不设计认证审核逾期状态和待审核材料自动过期测试。测试用例需要回链到需求编号、接口契约和状态流转。

### 4.5 P0-2 需求发布与审核

#### 4.5.1 模块定位

需求发布与审核模块承接已认证学生发布搭子需求的完整发布物生命周期，覆盖草稿保存、提交审核、管理员审核、审核通过后进入广场展示、审核驳回后修改重提、发布者主动下架等能力。

本模块聚焦“发布物从创建到可见或不可见”的状态与审核闭环，不覆盖广场推荐排序、发起联系、联系方式解锁、评价与信用摘要变化。广场展示后的发现与排序归入“广场发现与推荐”模块；基于需求发起邀约、站内消息和联系方式交换归入“低压力联系”模块；活动结束后的评价与信用变化归入“评价与信用摘要”模块。

第一轮核心对象包括：

- `PartnerPost`
- `Attachment`
- `ModerationRecord`

#### 4.5.2 参与角色

- 已认证学生：可新建草稿、编辑草稿、提交审核、查看自己的审核结果、修改被驳回需求并重新提交、主动下架自己的已发布需求。
- 管理员：负责审核待审核需求，给出通过或驳回结果；具体治理操作归入管理端治理模块，但审核结果会回写发布物状态。
- 未认证用户：不可新建草稿、提交审核或发布需求。

只有认证状态为 `VERIFIED` 的用户可以新建草稿、提交审核、修改草稿和下架自己的已发布需求。`UNVERIFIED`、`PENDING_REVIEW`、`REJECTED`、`SUSPENDED` 用户不得发布需求或提交审核。客户端应根据认证状态控制发布入口可见性和可操作性；后端接口必须进行认证状态校验，不能只依赖客户端隐藏入口。

#### 4.5.3 状态与约束

需求发布状态采用以下状态集合：

| 状态 | 含义 | 说明 |
|---|---|---|
| `DRAFT` | 草稿 | 发布者尚未提交审核，可继续编辑 |
| `PENDING_REVIEW` | 待审核 | 发布者已提交，等待管理员审核 |
| `PUBLISHED` | 已发布 | 审核通过，进入广场展示范围 |
| `REJECTED` | 已驳回 | 审核未通过，发布者可修改后重新提交 |

P0-2 当前不保留 `WITHDRAWN` 状态，也不引入 `EXPIRED`、`CLOSED`、`ARCHIVED` 等状态。活动是否结束、评价是否生成、历史是否归档等问题后续分别在低压力联系、评价与信用摘要或扩展性设计中确认，不提前混入需求发布与审核模块。

草稿与正式需求不拆分为两个独立核心领域对象。P0-2 使用 `PartnerPost` 统一承载需求发布物，草稿通过 `PartnerPost.status = DRAFT` 表达。草稿阶段允许部分字段暂未完整；提交审核时必须执行完整性校验，校验通过后状态进入 `PENDING_REVIEW`。文档中不再将 `PostDraft` 作为独立核心领域对象，仅将“草稿”作为 `PartnerPost` 的状态或客户端草稿视图表达。

`PENDING_REVIEW` 状态下发布内容冻结，发布者不得直接编辑需求内容。若发布者需要修改内容，必须先撤回审核，使 `PartnerPost.status` 从 `PENDING_REVIEW` 回到 `DRAFT`；修改完成后再次提交审核，并重新进入审核队列。后端编辑接口必须拒绝对 `PENDING_REVIEW` 状态内容的直接修改。

`PUBLISHED` 状态下发布者不得直接编辑已公开内容。若发布者需要修改已发布需求，应先主动下架，使 `PartnerPost.status` 从 `PUBLISHED` 进入 `DRAFT`，修改完成后重新提交审核；审核通过后才能再次进入 `PUBLISHED`。该规则保证所有对外公开内容均经过审核，避免已发布内容绕过先审后发机制。

`REJECTED` 状态表示上一版审核未通过的结果态。发布者开始编辑被驳回需求时，`PartnerPost.status` 应从 `REJECTED` 回到 `DRAFT`；修改完成后重新提交审核并进入 `PENDING_REVIEW`。系统应保留上一版驳回原因用于用户理解修改方向，但不得将“已驳回且内容已修改”混合为同一状态。

管理员审核需求发布时，V1 只提供“通过”和“驳回”两种审核结果。审核通过后，`PartnerPost.status` 从 `PENDING_REVIEW` 进入 `PUBLISHED`；审核驳回后，`PartnerPost.status` 从 `PENDING_REVIEW` 进入 `REJECTED`，且管理员必须填写驳回原因。P0-2 不设置“打回补充但仍保持待审核”“转交其他管理员”“暂缓审核”等复杂审核结果。

P0-2 不设置独立的发布审核记录对象。审核结果、审核时间、驳回原因等仅作为 `PartnerPost` 当前审核状态所需的必要字段保存，用于状态回显和发布者修改。审核动作不额外形成完整历史留痕；如后续管理端治理模块需要审计管理员操作，再在治理模块中统一确认，不在 P0-2 中提前引入。

在不设置独立审核记录的前提下，`PartnerPost` 只保留最近一次驳回原因。发布者开始编辑并回到 `DRAFT` 后，仍可在本次修改界面查看最近一次驳回原因；再次提交审核后，客户端可隐藏该原因，后端可清空或覆盖该字段。系统不提供历史驳回原因列表。

发布需求审核通过后，`PartnerPost.status` 立即进入 `PUBLISHED`，并进入广场可展示范围。P0-2 只负责发布物是否可见的状态开关，不负责广场列表中的排序、筛选、推荐权重和展示策略；这些内容后续归入“广场发现与推荐”模块。广场列表默认只读取 `PUBLISHED` 状态的需求。

发布者主动下架已发布需求后，不允许直接恢复发布。主动下架使 `PartnerPost.status` 从 `PUBLISHED` 回到 `DRAFT`；如需重新展示，发布者必须重新提交审核，审核通过后才能再次进入 `PUBLISHED`。系统不保留 `WITHDRAWN` 状态，主动下架作为操作记录或审核相关留痕保存。主动下架不要求发布者填写下架原因，系统仅记录必要操作留痕，例如发布物、操作者、操作类型和操作时间。

同一用户允许同时发布多个 `PUBLISHED` 需求，但 V1 设置宽松数量上限：同一用户最多同时保留 10 条 `PUBLISHED` 需求。达到上限后，用户可以继续保存草稿，但不得提交新的需求进入审核；管理员审核通过前也需要校验该用户当前已发布数量，避免审核通过后突破上限。

需求发布中的附件图片作为发布内容的一部分进入审核。若 `PartnerPost` 携带附件图片，管理员审核页需要提供附件预览；审核通过后附件随需求进入可展示范围，审核驳回后附件不对外展示。附件可见性跟随 `PartnerPost.status`，避免通过图片绕过先审后发机制。

`PartnerPost` 提交审核时至少需要满足以下必填字段完整性：

- 标题。
- 搭子类型。
- 活动时间或时间偏好。
- 地点或地点偏好。
- 人数/对象要求。
- 正文描述。
- 联系方式交换偏好。

附件图片作为选填内容，不作为所有搭子需求的必填项。客户端需要在提交审核前进行表单校验，后端提交审核接口需要再次校验必填字段，避免绕过客户端校验提交无效需求。

需求发布表单需要支持自动保存草稿，但自动保存限定在 Qt 客户端本地缓存中完成，不因每次输入变更立即同步到后端。用户显式点击“保存草稿”或“提交审核”时，客户端才调用后端接口写入或更新 `PartnerPost`。后端只处理显式保存草稿和提交审核，不将客户端本地自动保存作为服务端核心流程。

客户端本地自动保存草稿不要求加密存储，但不得以 txt、docx、明文 json 等容易被普通文本查看器直接读取和误修改的格式保存。Qt 客户端应使用专用本地缓存格式，并加入基础完整性校验或版本标识，降低误修改和误读取风险。草稿本地缓存不得使用 `QSettings` 保存，且不应包含 access token、refresh token、密码等敏感凭据。

#### 4.5.4 第一轮收束说明

需求发布与审核模块第一轮正文补全已收束。当前已确认模块边界、参与角色、认证状态前置条件、`PartnerPost` 四态状态集合、草稿与正式需求的对象关系、提交审核必填字段、待审核与已发布内容编辑规则、驳回后修改规则、管理员审核结果、审核通过后的广场展示入口、主动下架规则、同一用户发布数量上限、附件图片审核、审核数据最小化和客户端本地草稿缓存策略。后续若广场发现、低压力联系或接口契约发现冲突，再回到本模块补充修订。

### 4.6 P0-3 广场发现与推荐

#### 4.6.1 模块定位

广场发现与推荐模块承接已发布需求的广场列表、筛选、搜索、需求详情查看和基础排序/推荐入口；其输入以前序模块中 `PUBLISHED` 状态的 `PartnerPost` 为准。

本模块负责让用户发现需求，并基于标题、类型、时间、地点、对象要求、发布者摘要和信用摘要等信息判断是否感兴趣；不覆盖发起邀约、站内消息、联系方式卡片或联系方式解锁。需求详情页可以提供联系入口，但入口触发后的业务动作归入 P0-4 “低压力联系”模块。

未认证用户允许查看广场列表和需求详情，但只能查看 `PUBLISHED` 需求的公开信息。未认证用户在需求详情页看到的联系入口应处于禁用或引导认证状态，不得发起邀约、站内消息、联系方式解锁或其他真实互动。后端需要区分公开浏览权限和互动权限，不能仅依赖客户端隐藏联系入口。

广场列表和需求详情不对未登录游客开放。V1 要求用户至少完成登录后才能浏览广场；登录但未认证用户可以浏览 `PUBLISHED` 需求的公开信息，只有 `VERIFIED` 用户可以进入后续互动链路。该规则将“登录”作为浏览门槛，将“校内身份认证”作为互动门槛。

广场列表 V1 基础筛选条件包括搭子类型、时间偏好、地点关键词、人数/对象要求。发布者信用摘要在列表项和详情页中展示，作为用户判断参考，但不作为 V1 硬筛选条件；P0-3 不提供按信用摘要区间筛选的能力。

广场搜索范围限定在需求标题和正文描述。`PostSearchCriteria.keyword` 仅匹配 `PartnerPost.title` 与 `PartnerPost.description`；搭子类型、时间偏好、地点关键词、人数/对象要求通过结构化筛选表达，不混入全文搜索语义。

广场默认排序采用发布时间倒序。V1 默认按 `PUBLISHED` 需求的发布时间从新到旧展示，`PostSortStrategy` 默认值为 `LATEST`；同一发布时间下使用稳定次序字段补充排序，例如 `postId` 或创建时间。P0-3 暂不引入复杂个性化推荐算法，但保留排序策略对象用于后续扩展推荐排序。

广场列表需要分页加载。后端列表接口采用分页查询，默认每页 20 条，并明确 `page`、`size`、`sort` 等请求参数；Qt 客户端交互明确采用滚动加载，在用户接近列表底部时加载下一页。筛选、搜索和排序条件变化时，客户端应重置分页状态并从第一页重新加载。

需求详情页不展示发布者完整个人资料。`PostDetailView` 只展示发布者公开摘要，包括昵称/展示名、认证状态、信用摘要，以及必要的学院、年级等脱敏摘要；不得展示真实姓名、学号、校园邮箱、手机号等敏感信息。后端详情接口不得直接返回 `UserProfile` 全量字段，联系方式交换应留给 P0-4 “低压力联系”模块处理。

需求详情页可以展示附件图片原图，但仅限经过审核且随 `PUBLISHED` 需求公开的附件。附件访问仍需通过后端权限校验和后端中转，不直接暴露对象存储长期公开地址。详情接口可以返回附件引用或受控访问入口，不返回对象存储裸 URL；未登录用户、非 `PUBLISHED` 状态需求或无权限场景不得访问附件。

广场列表项可以展示第一张附件缩略图，但不直接加载附件原图。缩略图仅来自已审核通过且随 `PUBLISHED` 需求公开的附件，并通过后端受控访问入口加载；无附件时使用统一占位样式。该规则用于提高列表浏览效率，同时避免滚动加载时大量加载原图。

广场列表项不展示完整正文描述。`PostListItemView` 展示正文摘要，默认取正文描述前 30 个中文字符或等效长度字符；完整正文仅在 `PostDetailView` 中展示。正文摘要用于提升列表浏览的信息密度，避免每个列表项过于单薄，同时控制滚动列表高度和阅读负担。

广场列表不提供“只看自己发布的需求”筛选。用户自己的 `PUBLISHED` 需求如果出现在广场列表中，可以像其他需求一样被点开查看详情；但编辑、下架、重新提交审核等管理动作不在广场筛选中承载，应通过“我的发布”或发布管理入口处理。

用户不能在广场对自己发布的需求发起联系。自己的需求详情页可以正常查看，但联系入口应禁用或显示“这是你发布的需求”等提示。P0-3 详情视图需要识别 `ownPost` 展示状态；P0-4 低压力联系接口后续也必须校验发起者不能等于发布者。

V1 不实现收藏/保存需求功能。P0-3 不设计收藏按钮、我的收藏列表、`FavoritePost` 对象或收藏接口；但广场列表项和详情视图应避免将收藏能力排除在布局和对象扩展之外，后续可通过新增 `FavoritePost` 或同类用户-需求关系对象扩展收藏能力。

#### 4.6.2 第一轮收束说明

广场发现与推荐模块第一轮正文补全已收束。当前已确认模块边界、登录与认证可见性、基础筛选条件、搜索范围、默认排序、滚动分页、详情页发布者公开摘要、详情页附件展示、列表缩略图、列表正文摘要、自有需求展示限制和收藏扩展预留。后续若低压力联系、评价信用或接口契约发现冲突，再回到本模块补充修订。

### 4.7 P0-4 低压力联系

#### 4.7.1 模块定位

低压力联系模块承接用户基于已发布需求发起邀约或站内联系、建立会话、发送站内消息、交换联系方式卡片和联系方式解锁等能力。该模块负责从需求详情页产生受控互动，并在双方具备意愿后交换联系方式。

本模块不扩展为完整即时聊天系统。V1 站内消息作为轻量沟通载体存在，不设计在线状态、已读回执、输入中、实时推送等完整 IM 能力；消息传输方式延续既有技术约束，V1 采用 REST 轮询，并为后续 WebSocket 演进预留边界。

第一轮核心对象包括：

- `Conversation`
- `Message`
- `ContactCard`
- `ContactUnlockRecord`
- `PartnerPost`

#### 4.7.2 参与角色与前置条件

发起低压力联系时，发起者必须处于 `VERIFIED` 状态，需求发布者在发起时也必须仍处于 `VERIFIED` 状态。若发布者因校园邮箱更换、重新认证、治理限制等原因不再具备已认证学生能力，则不得基于其历史发布发起新的联系。

发起联系还需要满足以下前置条件：目标需求处于 `PUBLISHED` 状态；发起者不能等于发布者；发起者与发布者均未处于阻断该互动的治理限制状态。后端发起联系接口必须执行上述校验，客户端禁用入口不能替代后端校验。

发起联系成功后，系统立即创建一个绑定目标 `PartnerPost` 的 `Conversation`，并发送一条初始邀约消息。该会话参与者为发起者和需求发布者；初始邀约消息用于表达联系意图。联系方式卡片在会话创建时保持未解锁状态，后续需要按联系方式解锁规则处理。

同一两个用户之间只能存在一个有效 `Conversation`。用户基于某个 `PartnerPost` 发起联系时，系统应先按参与双方查找是否已有有效会话；若已有会话，则不重复创建新会话，而是在既有会话中追加本次邀约上下文和初始邀约消息，保留先前聊天记录。该规则不按 `postId + initiatorId` 限制会话唯一性，而按用户双方关系限制有效会话唯一性。

P0-4 不单独设计 `Invitation` 对象。根据具体 `PartnerPost` 发起的邀约上下文直接由 `Conversation` 及其消息记录共同维护，不再要求划分每一条消息归属于哪一次邀约。后续如同一双方因新的需求再次沟通，系统复用既有会话并追加新的初始邀约消息即可。

初始邀约消息允许用户完全自由输入，不采用固定系统模板或“模板 + 附言”模式。为控制消息质量和界面展示，初始邀约消息与普通站内文本消息统一采用短消息约束：消息去除首尾空白后必须非空，长度上限为 30 个中文字符或等效长度字符。V1 不引入图片、附件、表情等非纯文本消息类型。

#### 4.7.3 消息边界

P0-4 的站内消息定位为低压力联系入口，不承担完整即时通讯系统职责。消息内容以简短文本为主，初始邀约消息和普通站内文本消息共用同一套非空校验、长度校验和服务端约束。Qt 客户端应在输入框中给出剩余字数或超限提示；后端接口必须对空消息、全空白消息和超长消息进行拒绝，客户端校验不能替代后端校验。

`Conversation` 设置 `ACTIVE` 与 `CLOSED` 两种业务状态。会话创建后默认为 `ACTIVE`；任一参与方可以主动关闭会话。`CLOSED` 状态下不允许继续发送普通消息或解锁联系方式，但历史消息保留，用于用户回看和必要治理。若同一双方后续因新的 `PartnerPost` 再次发起联系，系统复用既有会话并通过新的初始邀约消息重新激活为 `ACTIVE`，不得在没有新需求和新初始消息的情况下静默恢复会话。

联系方式卡片默认处于锁定状态。只有当双方都在同一 `ACTIVE` 会话中明确确认愿意交换联系方式后，系统才解锁双方联系方式；单方确认不能查看对方联系方式。联系方式解锁结果由 `ContactUnlockRecord` 记录双方确认状态和最终解锁时间，后端返回联系方式前必须校验会话状态、参与关系和双方确认结果。

V1 的 `ContactCard` 仅支持用户主动填写并确认可交换的微信号、手机号和 QQ 号。用户可以填写其中一种或多种，但至少需要维护一种可交换联系方式。校园邮箱不作为联系方式卡片内容，其用途限定为账号注册、登录和校内身份认证，不默认进入社交交换链路。联系方式解锁后，系统只返回对方已维护且允许交换的联系方式项。

用户发起联系前必须已经维护至少一种有效的可交换联系方式，否则不得发起联系，应先进入联系方式维护流程。需求发布者可以在尚未维护联系方式的情况下发布需求并接收邀约、进行站内沟通，但在双方联系方式解锁前也必须补全联系方式卡片。联系方式解锁接口需要同时校验双方均存在有效 `ContactCard`。

联系方式解锁记录只保存双方已完成解锁的事实，不保存联系方式值的历史快照。用户后续修改联系方式后，已解锁会话中对方再次查看时应读取最新 `ContactCard` 内容；若用户删除到没有任何可交换联系方式，则已解锁会话中对方暂时看不到联系方式，并提示对方暂未维护可交换联系方式。该规则避免旧手机号、旧 QQ 号或旧微信号在系统中长期残留。

V1 不提供站内消息撤回或删除能力。`Message` 一经发送即作为会话历史保留，客户端不提供撤回、删除单条消息或删除服务端会话记录的入口。该约束用于保持会话上下文完整，并为后续投诉、申诉和管理员治理提供必要依据。

站内消息链路需要提供轻量举报入口。V1 按会话维度举报，不提供逐条消息举报；用户可在会话页发起“举报该会话”，选择举报原因并填写补充说明。P0-4 只负责提供入口和上下文挂接，系统应将会话、双方用户、关联需求和最近消息上下文提交给后续投诉/申诉模块处理，具体案件状态、证据补充和管理员处置流程归入 P1 “投诉申诉与案件”。

V1 站内消息采用 REST 轮询，并支持会话列表未读数和总未读数。系统需要记录每个用户在每个会话中的已读位置，例如 `lastReadMessageId` 或等价字段；用户进入会话详情后，将该会话中对方发送且尚未计入已读位置的消息标记为已读。该机制仅用于当前用户自己的未读提示，不向对方展示“已读”状态，不形成已读回执。

V1 站内消息轮询频率统一按 1 秒一次设计。Qt 客户端在会话列表和会话详情页均可按 1 秒间隔查询消息更新、会话未读数和总未读数；网络异常时应避免阻塞界面，并在恢复后重新拉取最新数据。该方案以桌面端接近即时的使用体验为优先，当前规模下不认为 1 秒轮询会造成不可接受的资源占用；后续如进入更高并发或移动端常驻场景，可在保持业务接口边界稳定的前提下演进到 WebSocket 或其他推送机制。

会话详情页消息查询采用增量拉取与历史分页结合的方式。进入会话时先加载最近一页历史消息，随后客户端维护当前最大 `messageId`，每 1 秒通过 `afterMessageId` 拉取新增消息；当用户向上滚动查看历史时，再按分页参数加载更早消息。消息查询接口需要支持最近一页、更早一页和某消息之后新增消息三类查询场景，避免轮询时重复拉取完整消息列表。

会话关闭后仍保留历史消息记录，但不再提供联系方式查看能力。`CLOSED` 表示一段联系关系已经终结，即使关闭前双方已经完成联系方式解锁，系统也不应在关闭后的会话中继续展示对方联系方式。关闭后不得发送新消息、不得发起新的联系方式解锁确认，也不得通过既有解锁记录继续访问联系方式；如同一双方后续因新的 `PartnerPost` 再次发起联系，需要通过新的初始邀约消息重新激活会话，并重新完成双方联系方式解锁确认。

关闭后的会话不能由任一方在旧会话中直接手动重新打开。会话重新激活必须通过新的 `PartnerPost` 邀约触发，并重新满足发起联系前置条件、联系方式卡片要求和初始邀约消息校验。为支持后续复联场景，邀约帖子可以在发布或发布后由发布者手动指定，定向推送给先前有过沟通记录的用户；该定向推送仅作为新需求邀约入口，不绕过对方是否接受沟通、会话状态和联系方式重新解锁规则。

定向推送的接收方范围限定为与发布者存在历史 `Conversation` 的用户。V1 不允许发布者按学号、手机号、邮箱、昵称或其他个人标识搜索任意用户进行定向推送。后端需要校验发布者与接收方存在历史会话记录，且双方当前不存在阻断互动的治理限制；通知模块后续负责承接定向推送通知。

P0-4 需要支持用户拉黑功能。任一用户拉黑对方后，系统应立即阻断双方后续互动：拉黑方不再收到被拉黑方发送的站内消息、定向推送或新的邀约入口；被拉黑方也不得继续查看拉黑方在该关系链路中的消息内容、联系方式或后续定向推送信息。拉黑关系应作为会话、发起联系、消息发送、联系方式查看和定向推送接口的统一前置校验条件。

用户可以主动解除拉黑，但解除拉黑只移除阻断关系，不自动恢复旧会话互动。若旧会话已关闭，则仍保持 `CLOSED`，历史联系方式仍不可见；双方后续如需再次沟通，仍需通过新的 `PartnerPost` 邀约、发送新的初始邀约消息，并重新完成联系方式解锁确认。解除拉黑不应自动通知对方，也不应自动恢复会话、消息可见性或联系方式可见性。

拉黑后，该关系链路中的历史会话消息在双方普通客户端视图中均不可见。客户端会话列表不再展示该会话，普通会话详情接口也不得向任一方返回历史消息内容；但服务端仍保留消息记录，用于投诉、申诉和管理员治理。解除拉黑后，也不自动恢复历史消息可见性；若双方后续通过新的 `PartnerPost` 邀约重新建立联系关系，再按新的会话激活和消息展示规则处理。

拉黑动作触发后，当前双方 `Conversation` 应立即进入 `CLOSED`，并记录关闭原因或阻断来源为 `BLOCKED`。会话业务状态仍只保留 `ACTIVE` 与 `CLOSED`，不额外增加 `BLOCKED` 状态；拉黑语义由 `BlockRelation` 或等价黑名单关系对象表达。发消息、联系方式解锁、定向推送和重新激活会话时，应同时校验 `Conversation.status` 与拉黑关系。
