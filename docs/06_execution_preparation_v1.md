- 服务器 smoke test 至少记录：部署或重启结果、服务健康检查、关键接口最小调用、数据库迁移或连通性、对象存储连通性，以及本批次相关的环境变量或私有配置是否就位。

## 2. 模块拆分与职责边界

### 2.1 学生端业务模块

1. `账号准入与身份认证模块`
   - 负责：登录注册、校园邮箱验证码、学号认证资料提交、认证状态回显
   - 不负责：资料审核裁定、账号处罚
   - 上游依赖：无
   - 下游输出：`user_id`、登录态、认证状态

2. `个人资料与信用展示模块`
   - 负责：昵称/头像/简介/标签维护，个人主页、信用摘要/靠谱分摘要、评价摘要展示
   - 不负责：信用摘要/靠谱分计算、评价裁定
   - 上游依赖：账号准入、评价与信用摘要模块
   - 下游输出：对外展示用的用户可信度快照

3. `需求发布与我的发布模块`
   - 负责：3 步发布表单、草稿保存、场景切换确认、提交审核、驳回后修改重提、关闭与再次发布
   - 不负责：公开广场排序、审核裁定
   - 上游依赖：身份认证已通过或至少具备可发布资格
   - 下游输出：`PartnerPost` 与草稿数据

4. `广场发现与推荐模块`
   - 负责：首页推荐分区、广场列表、筛选排序、详情页展示、推荐理由摘要
   - 不负责：会话创建后的沟通
   - 上游依赖：已审核通过的需求、规则推荐结果、用户可信度摘要
   - 下游输出：点击详情、发起私信、举报需求

5. `低压力联系模块`
   - 负责：消息列表、会话页、文本/图片/联系方式卡片、联系方式解锁状态、消息举报
   - 不负责：评价裁定、外部联系后的线下履约真实性判断
   - 上游依赖：需求详情、会话上下文、举报模块
   - 下游输出：联系记录、评价候选关系、消息举报线索

6. `评价与信用摘要模块`
   - 负责：待评价列表、评价表单、评价提醒、已过期未评状态、信用摘要/靠谱分变更记录展示
   - 不负责：申诉裁定
   - 上游依赖：合作/联系记录、场景类型
   - 下游输出：评价记录、信用摘要/靠谱分变更日志、争议入口

7. `投诉/申诉模块`
   - 负责：案件提交、证据上传、进度查看、被投诉方回应入口
   - 不负责：案件最终裁定
   - 上游依赖：需求/消息/评价/用户关联关系
   - 下游输出：案件记录、补证据记录、通知触发

8. `通知模块`
   - 负责：审核结果、评价提醒、案件进度、举报受理等统一通知聚合
   - 不负责：具体业务裁定
   - 上游依赖：审核、评价、案件、私信事件
   - 下游输出：用户可见提醒状态

### 2.2 管理端业务模块

1. `审核工作台模块`
   - 负责：需求、资料、头像昵称、评价等内容审核
   - 主结构：按内容类型分 tab，tab 内按风险优先级 + 提交时间排序

2. `案件处理模块`
   - 负责：投诉/申诉案件查看、双方回应、裁定结果输出
   - 输出：删除/纠正评价、扣减/恢复信用摘要/靠谱分、警告、禁言、封禁等结果

3. `用户治理模块`
   - 负责：用户检索、详情查看、状态处置、历史处罚查看
   - 与案件处理的关系：案件是触发来源，用户治理是长期状态落点

4. `信用摘要/靠谱分干预模块`
   - 负责：分值调整、原因记录、操作留痕
   - 与评价模块的边界：评价模块产生规则分，干预模块处理人工修正

5. `基础看板模块`
   - 负责：待审核量、待处理案件量、活跃场景分布等最小运营信息
   - 不负责：复杂 BI 分析

### 2.3 跨模块共享能力

- `对象关联能力`
  - 会话绑定原始需求
  - 评价绑定相关需求与双方用户
  - 案件绑定需求/消息/评价/用户
- `文件附件能力`
  - 头像
  - 消息图片/截图
  - 投诉/申诉证据
- `状态与通知分发能力`
  - 审核、案件、评价提醒都依赖统一状态变更通知
- `操作留痕能力`
  - 审核记录
  - 分值调整记录
  - 案件处理记录

## 3. 核心状态机与状态流转

### 3.1 用户认证状态机

`未认证 -> 待审核 -> 已通过 / 已驳回`

- 进入 `未认证`：首次注册完成但未提交认证资料
- `未认证 -> 待审核`：提交学号、学院、年级、昵称等认证资料
- `待审核 -> 已通过`：管理员审核通过
- `待审核 -> 已驳回`：管理员驳回并给出原因
- `已驳回 -> 待审核`：用户修改后再次提交

边界说明：

- `认证驳回` 不等于处罚
- 未认证用户可浏览，但发私信、发布需求等关键动作需要受限

### 3.2 资料审核状态机

`未提交 -> 待审核 -> 已通过 / 已驳回`

- 作用对象：昵称、头像、简介等公开资料
- 与身份认证分离原因：身份通过不代表公开资料可直接展示

### 3.3 搭子需求状态机

`草稿 -> 待审核 -> 已发布 -> 已关闭 / 已过期`

补充分支：

- `待审核 -> 已驳回`
- `已驳回 -> 待审核`
- `已发布 -> 已关闭`
- `已发布 -> 已过期`
- `已过期 -> 再次发布` 实际应生成新需求，不回写旧需求状态

约束：

- `草稿` 不进入广场
- `待审核` 不进入广场
- `已驳回` 需携带驳回原因与修改动作
- `已关闭` 表示用户主动结束招募
- `已过期` 表示时间自然到期

### 3.4 私信会话状态机

`已创建 -> 沟通中 -> 联系方式已解锁 -> 已沉默/已结束`

其中有一条并行子状态：

- `联系方式未解锁 -> 联系方式已解锁`

触发规则：

- 会话从需求详情发起时进入 `已创建`
- 任一方发送首条有效消息后仍保持 `联系方式未解锁`
- 双方各发送 1 条有效消息后进入 `联系方式已解锁`
- 长时间无新消息可归入 `已沉默`
- 若需求关闭、评价完成且双方无继续沟通，可标记 `已结束`

待实现定义：

- “有效消息”默认指非空文本或成功上传图片，不含系统提示

### 3.5 评价任务状态机

`待评价 -> 提醒中 -> 已提交 / 已过期未评`

说明：

- `待评价` 是业务任务状态，不等同于 `Review` 记录已创建
- 进入条件：平台认定一次联系/合作已结束，进入评价窗口
- `提醒中` 是 `待评价` 的展示态，不一定单独落库
- `已提交`：用户完成评价
- `已过期未评`：超过评价期限自动关闭入口

配对规则：

- 一次合作应生成双方各自独立的评价任务
- 单方提交不阻塞另一方状态流转

### 3.6 案件状态机

`待补证据 / 待对方回应 / 处理中 / 已结案`

建议流转：

- 提交案件且证据不足时：`待补证据`
- 证据基本齐备后通知对方：`待对方回应`
- 管理员开始审理：`处理中`
- 输出裁定后：`已结案`

补充：

- `已结案` 必须伴随 `decision_result`
- 若需要再次补充材料，不回退成新案件，优先在原案件内追加回应记录

### 3.7 账号处置状态机

`正常 -> 警告中 / 临时禁言 / 封禁`

说明：

- `警告中` 更接近留痕标签，可与 `正常` 并存显示在管理端
- `临时禁言` 影响发私信、发布、评论等动作
- `封禁` 影响整体使用权限

## 4. 接口输入输出细化

以下只细化 execution 初期最关键的接口契约，优先覆盖最小闭环。

### 4.1 账号准入与身份认证

1. `发送校园邮箱验证码`
   - 输入：`campus_email`
   - 输出：`request_id`、发送结果、限流提示
   - 规则：仅允许学校白名单域名

2. `验证码登录/注册`
   - 输入：`campus_email`、`code`
   - 输出：`access_token`、`user_id`、`is_new_user`、`auth_status`

3. `提交认证资料`
   - 输入：`student_no`、`college`、`grade`、`nickname`
   - 输出：`auth_status=待审核`、提交时间

4. `获取当前认证状态`
   - 输出：`auth_status`、`reject_reason`、`allowed_actions`

### 4.2 个人资料

1. `获取我的资料`
   - 输出：基础资料、资料审核状态、信用摘要/靠谱分摘要、评价摘要标签

2. `更新我的资料`
   - 输入：`nickname`、`avatar`、`bio`、`tags`
   - 输出：最新资料、`profile_review_status`
   - 规则：涉及公开展示字段时应重新进入资料审核

### 4.3 需求发布与我的发布

1. `创建或保存草稿`
   - 输入：
     - 公共字段：`scene_type`、`title`、`summary`、`time_text`、`location_text`、`participant_count`、`gender_preference`、`expectation_tags`、`expires_at`
     - 场景字段：`extra_payload`
   - 输出：`post_id`、`post_status=草稿`

2. `提交审核`
   - 输入：`post_id`
   - 输出：`review_status=待审核`
   - 前置条件：字段校验通过、用户具备发布资格

3. `获取我的需求列表`
   - 输入：`status_tab`、分页参数
   - 输出：列表项、主状态、驳回原因摘要、可执行动作

4. `修改后重提`
   - 输入：`post_id` + 修改字段
   - 输出：`review_status=待审核`

5. `关闭需求`
   - 输入：`post_id`
   - 输出：`post_status=已关闭`

### 4.4 广场与推荐

1. `获取首页轻量推荐`
   - 输入：当前用户、场景偏好或最近行为上下文
   - 输出：
     - `recommendation_cards`
     - 每条含 `post_id`、卡片摘要、`recommend_reason_summary`

2. `获取广场列表`
   - 输入：
     - `scene_type`
     - `time_filter`
     - `location_filter`
     - `gender_filter`
     - `tag_filters`
     - `sort_by`
   - 输出：
     - 列表卡片
     - 分页信息
     - 当前筛选条件回显

3. `获取需求详情`
   - 输入：`post_id`
   - 输出：
     - 需求完整字段
     - 发起人可信度摘要
     - 当前用户可执行动作，如 `can_message`、`can_report`

4. `举报需求`
   - 输入：`post_id`、`reason`、可选附件
   - 输出：受理状态、案件入口

### 4.5 私信与联系方式解锁

1. `创建或打开会话`
   - 输入：`related_post_id`、`target_user_id`
   - 输出：
     - `conversation_id`
     - 会话状态
     - 联系方式是否已解锁

2. `获取会话列表`
   - 输出：关联需求、未读数、最近消息摘要、会话状态

3. `获取消息列表`
   - 输入：`conversation_id`
   - 输出：消息流、关联需求卡片、联系方式解锁状态

4. `发送消息`
   - 输入：
     - `conversation_id`
     - `message_type=text|image|contact_card`
     - 对应 payload
   - 输出：
     - `message_id`
     - 发送时间
     - 是否触发联系方式解锁变化
   - 规则：
     - `contact_card` 发送前必须满足已解锁

5. `举报消息`
   - 输入：`message_id`、`reason`、可选补充说明
   - 输出：举报受理结果、案件编号

### 4.6 评价与信用摘要

1. `获取待评价任务`
   - 输出：关联对象、场景类型、截止时间、当前状态

2. `提交评价`
   - 输入：
     - `review_target_user_id`
     - `related_post_id`
     - `scene_type`
     - 通用维度结果
     - 协作维度结果
     - `review_tags`
     - `comment_text`
   - 输出：
     - `review_id`
     - `review_status`
     - 信用摘要/靠谱分是否已变更

3. `获取信用摘要/靠谱分明细`
   - 输出：当前分值、最近变更记录、来源说明

### 4.7 投诉 / 申诉

1. `创建案件`
   - 输入：
     - `case_type=投诉|申诉`
     - `target_type=需求|消息|评价|用户|处理结果`
     - `target_id`
     - `description`
     - `evidence_list`
   - 输出：`case_id`、`case_status`

2. `补充证据`
   - 输入：`case_id`、附件、补充说明
   - 输出：最新案件状态

3. `提交案件回应`
   - 输入：`case_id`、`response_text`、`evidence_list`
   - 输出：回应记录、最新案件状态

4. `获取我的案件列表/详情`
   - 输出：案件状态、处理节点、对方是否已回应、结果摘要

### 4.8 管理端

1. `获取待审核队列`
   - 输入：`content_type_tab`、`risk_level`、分页参数
   - 输出：待处理数量、列表项、风险摘要

2. `审核内容`
   - 输入：`target_type`、`target_id`、`decision=通过|驳回`、`reason`
   - 输出：最新审核状态、通知是否已触发

3. `获取案件队列`
   - 输入：`case_status`、`case_type`
   - 输出：案件列表、待回应/待补证据提醒

4. `处理案件`
   - 输入：`case_id`、`decision_result`、`decision_note`、可选附带处置动作
   - 输出：案件结案结果、关联对象更新结果

5. `用户治理动作`
   - 输入：`user_id`、`action_type=警告|禁言|封禁|恢复`
   - 输出：用户新状态、操作记录

6. `信用摘要/靠谱分人工调整`
   - 输入：`user_id`、`change_value`、`reason_code`、备注
   - 输出：新分值、调整记录

## 5. 开发顺序与最小交付闭环

### 5.1 推荐的实施切片

1. `M0：准入与可发布基础`
   - 登录注册
   - 校园邮箱验证码
   - 身份认证提交流程
   - 个人资料最小编辑

2. `M1：发布-审核-广场闭环`
   - 3 步发布表单
   - 草稿保存
   - 我的发布状态页
   - 管理端需求审核
   - 广场列表与详情页

3. `M2：联系闭环`
   - 会话创建
   - 消息列表/会话页
   - 文本/图片发送
   - 联系方式卡片解锁
   - 消息举报

4. `M3：信用闭环`
   - 待评价任务
   - 评价提交
   - 信用摘要/靠谱分规则记账
   - 我的页信用摘要

5. `M4：争议治理闭环`
   - 投诉/申诉提交
   - 证据补充
   - 被投诉方回应
   - 管理端案件处理
   - 人工分值干预与处罚

6. `M5：体验补完`
   - 首页轻量推荐
   - 通知中心
   - 基础看板
   - 资料审核、评价审核等扩展审核面

### 5.2 最小交付闭环定义

如果要尽快形成“可演示、可验证、可继续扩展”的第一条工程闭环，建议先做到：

1. 学生完成登录与认证提交
2. 学生发布需求并进入待审核
3. 管理员审核通过
4. 另一名学生在广场看到需求并发起私信
5. 双方各发送 1 条有效消息，联系方式解锁

这个闭环暂不要求：

- 规则推荐必须完成
- 评价与信用摘要/靠谱分必须上线
- 投诉/申诉必须上线

但要预留：

- 会话绑定需求
- 后续评价任务挂接点
- 案件与举报的对象关联

### 5.3 推荐的实现顺序原则

- 先做“对象能流转”，再做“分值和治理”
- 先做“学生闭环”，再做“管理员扩面”
- 先做“可解释规则”，再做“复杂策略优化”
- 先做“关键状态露出”，再做“长尾增强能力”

## 6. 风险点与待确认技术决策

### 6.1 业务规则待确认

1. `评价触发时机`
   - 当前仅确认“活动或合作结束后”进入评价
   - 仍需明确谁来触发结束：用户手动、到期自动、管理员介入，还是多条件组合

2. `有效消息定义`
   - 当前建议：非空文本或图片即算
   - 仍需防止用户用无意义字符刷解锁

3. `评价期限`
   - 已确认有 `已过期未评`
   - 但具体期限和提醒频率尚未固定

4. `信用摘要/靠谱分分值表`
   - 当前只有原则，没有行为到分值的映射表
   - execution 前至少需要 V1 规则表

5. `推荐排序权重`
   - 当前仅确认维度，不含权重
   - 如果首页要展示推荐，至少要先给出可解释的初版排序口径

### 6.2 执行成本风险

1. `先审后发的人力成本`
   - 所有需求先审后发会直接抬高冷启动阶段的人力要求
   - 需明确审核 SLA，否则 `待审核` 状态会拖慢主路径

2. `站内私信与证据链保存`
   - 私信既是沟通能力也是争议证据来源
   - 图片、截图、消息举报需要稳定附件与留痕方案

3. `管理端范围较大`
   - 当前 V1 已包含审核、案件、用户治理、分值干预
   - 若一次性全做，execution 容易失控

### 6.3 课程叙事风险

1. `PPT 与当前收敛结论存在差异`
   - PPT 中曾写“不做即时通讯，转微信”
   - 当前已确认第一版需要站内私信
   - 后续汇报时必须明确说明：这是在后续规划与原型评审中做出的升级决策

2. `PPT 中的量化指标尚不应当作当前实现验收`
   - 匹配成功率、周活跃、满意度更适合作为内测目标
   - 不应在 execution 启动前误设为开发完成标准

### 6.4 建议优先拍板的技术决策

以下决策足以影响 execution 开工顺序，但不需要现在展开到深架构层：

1. 校园邮箱验证码发送方式与邮箱白名单规则
2. 图片/证据附件的统一存储方式
3. 私信第一版是否要求实时到达，还是允许准实时刷新
4. 审核与案件通知采用站内通知为主，是否需要额外邮件提醒
5. 评价截止时间、提醒节奏、案件处理时限的默认口径
