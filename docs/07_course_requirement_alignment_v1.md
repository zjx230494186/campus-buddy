## 1. 新增硬约束

根据老师最新要求，项目后续必须满足以下硬约束：

1. 需求管理与软件建模需在 `CodeArts` 平台上完成，而不是只在本地文档中沉淀。
2. 编码实现阶段需使用 `CodeArts 智能体` 进行代码编写。
3. 当前已知老师指定使用的编码模型为 `GLM5.1`。
4. 最终交付不仅是源代码，还包括：
   - CodeArts 需求管理产物
   - CodeArts 建模产物
   - 编码实现报告
   - README
   - 其他中间文档与支撑材料
5. 编码实现报告中必须明确：
   - 项目简介
   - Prompt
   - Vibe Coding 过程
   - AI 生成后人工做了哪些修改
   - 存在哪些问题
   - 系统功能分别来源于哪条 `IR` 或 `US`
6. 项目需要能交付“可真实运行的最简化可实行版本（MVP）”，但不要求一开始做成完整产品。

## 2. 对当前项目推进方式的直接影响

### 2.1 不能再把“本地高质量设计文档”视为最终主交付结构

当前 `04/05/06` 文档仍然有价值，但它们现在更适合作为：

- CodeArts 需求录入前的收敛底稿
- CodeArts 建模前的语义基线
- 弱模型编码前的执行说明书

而不再是唯一主交付面。

### 2.2 不能默认按“强模型一次性拉通代码实现”推进

因为老师明确要求使用指定智能体编码，且当前已知执行模型为 `GLM5.1`，所以后续实现策略必须从：

- “由当前线程直接高强度完成架构与代码”

转为：

- “先把边界、规则、模块、任务切片、输入输出和验收口径拆得足够细”
- “让较弱模型也能基于这些材料稳定生成正确代码”
- “人工负责校正、复核、拼装与溯源”

### 2.3 每个阶段都要兼顾“平台录入形态”

后续不只是回答“项目该怎么做”，还要回答：

- 这部分内容在 CodeArts 里对应什么对象
- 需要录成哪一层需求
- 需要配套哪些图
- 后续代码如何回链到 IR / US

## 3. 新的推进原则

### 3.1 总原则

项目后续以“可被弱模型执行的结构化规划”作为核心中间产物，而不是直接追求本地代码规模。

### 3.2 具体原则

1. `先边界，后实现`
   - 先把系统边界、角色边界、模块边界、阶段边界说清楚，再进入 CodeArts 录入和编码。
2. `先映射，后展开`
   - 先把已有文档映射到 RR / SF / IR / US，再展开细化。
3. `先最小闭环，后补全范围`
   - 仍坚持 MVP，但 MVP 的定义要同时满足课程交付和系统可运行。
4. `先可执行任务单，后交给智能体`
   - 不直接把“大功能”交给较弱模型，而是拆成小任务、明确输入输出、明确验收标准。
5. `先需求溯源，后代码扩写`
   - 每个实现项都应能回链到某条 IR / US，避免后续报告无法说明来源。

## 6. 适配 `GLM5.1` 的执行策略

为了让老师指定的 `GLM5.1` 模型也能把项目稳定搭起来，建议统一采用以下方式喂给模型：

1. 单次只给一个明确目标
   - 例如“只生成认证模块的 API 和数据模型草案”，不要同时夹带广场、审核、私信。
2. 提供固定格式输入
   - 背景
   - 对应 IR / US
   - 输入数据结构
   - 输出数据结构
   - 边界限制
   - 不做事项
   - 验收标准
3. 保持任务粒度小
   - 以一个模块中的一个子闭环为单位，而不是整站。
4. 先生成，再复核，再拼装
   - 不期待一次生成即正确。
5. 所有 Prompt 和人工修订都留痕
   - 便于最终写编码实现报告。

### 6.1 skill 流程层的增强思路

在不能更换模型的前提下，优先增强“流程”，而不是只增强“提示词长度”：

1. `固定任务模板`
   - 每次给模型的任务输入都保持同一结构，降低结果漂移。
2. `固定上下文包`
   - 只提供当前任务需要的需求、接口、状态机、数据结构片段，不把全项目一次性塞给模型。
3. `固定输出格式`
   - 要求模型按“修改文件 / 核心逻辑 / 风险点 / 自测建议”输出，方便人工复核。
4. `固定复核流程`
   - 生成后先人工检查，再决定是否进入下一块。
5. `固定回链要求`
   - 每张任务卡都标注来源 IR / US，避免后续报告断链。

### 6.2 提示词工程层的增强思路

对 `GLM5.1` 更有效的，不是泛泛要求“写得更好”，而是把 prompt 工程化：

1. 明确角色
   - 例如“你现在只负责认证模块的 API 层，不负责页面、不负责私信、不负责推荐”。
2. 明确边界
   - 写清楚“要做什么 / 不做什么 / 不要修改什么”。
3. 明确输入输出
   - 输入字段、输出字段、状态变化、报错口径尽量写死。
4. 明确验收标准
   - 至少给出 2 到 5 条可检查标准。
5. 明确已有上下文
   - 指向具体文档片段或任务卡，而不是笼统说“参考项目文档”。

### 6.3 对项目推进方式的进一步结论

既然指定执行器是 `GLM5.1`，那么本项目后续最核心的工程能力就不再只是“我们能否自己把代码写出来”，而是：

- 我们能否把项目压缩成适合 `GLM5.1` 理解和执行的 skill 流程
- 我们能否把每一步任务写成高质量、低歧义的提示词包
- 我们能否把模型输出稳定接回需求、建模和报告链路

因此，后续线程优先级可以进一步明确为：

1. 需求映射
2. 建模清单
3. `GLM5.1` 任务卡与 Prompt 模板
4. 指定智能体编码执行
5. 人工复核与报告留痕

## 7. 当前最推荐的下一步

在新要求下，最合理的下一步不再是直接启动 `M0 + M1` 编码，而是先做：

1. `CodeArts 需求映射线程`
   - 把已有文档系统化转成 RR / SF / IR / US
2. `CodeArts 建模规划线程`
   - 先明确用例图、活动图、类图、顺序图分别画什么
3. `弱模型实现任务拆分线程`
   - 把 `M0 + M1` 拆成智能体可执行任务卡

## 8. 当前结论

`big_homework` 项目并未失去此前的规划价值，但其推进主线已从“直接进入实现启动”变为：

- 继续确定边界
- 按课程要求补齐 CodeArts 需求与建模结构
- 将实现规划压缩成弱模型可稳定执行的任务卡
- 再进入指定智能体编码阶段

这意味着当前项目最优先的工作已经从“代码起盘”切换为“交付结构重排 + 任务拆解能力建设”。
