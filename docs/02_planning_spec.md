## 2. 从 PPT 提取到的已知信息

### 2.1 问题域

- 项目工作名为 `校园搭子`
- 关注问题是“校园内找合适同伴困难”
- 材料声称基于 125 人问卷与 12 人深度访谈
- 问卷显示：
  - 51.2% 的学生表示“想找人一起做某件事，但找不到合适的人”
  - 58.4% 的学生每周多次遇到这类问题
- 问卷排序中，`找搭子困难` 被列为最高优先级问题

### 2.2 目标用户

- 在校大学生，以大一到大四为主
- 不是“所有学生”，而是：
  - 有明确结伴场景需求
  - 社交圈有限
  - 对伙伴质量有要求
  - 不太愿意主动联系陌生人

### 2.3 当前高频场景

- 饭搭子
- 考研/学习搭子
- 运动搭子
- 小组作业队友
- 大创项目队友

### 2.4 当前痛点归纳

- 信息触达率低：
  - 群聊、校园墙、朋友圈中的信息容易被淹没
- 匹配精度不足：
  - 很难按时间、地点、技能、水平、作息、性别等条件找到合适对象
- 伙伴稳定性差：
  - 爽约、划水、拖延、半途退出等问题明显
- 随机组队风险高：
  - 小组作业、大创项目中尤为突出
- 社交压力存在：
  - 用户不好意思频繁发消息或主动联系陌生人

### 2.5 当前替代方案与缺陷

- 宿舍群 / 班级群：消息淹没快、时效性差、社交压力大
- 朋友圈：触达面窄、信息不结构化
- 校园墙：信息碎片化、搜索筛选弱、流程成本高
- 线下碰运气：成功率低、时间成本高
- 熟人推荐：范围有限、难以匹配具体条件

### 2.6 PPT 中提出的 MVP 方向

- 核心假设：
  - `结构化需求 + 精准匹配 + 信用背书` 能提升搭子匹配成功率
- P0（PPT 中写为必须做）：
  - 结构化需求发布
  - 精准匹配与筛选
  - 活动后互评 / 信用评价体系
- P1（PPT 中写为应该做）：
  - 一键闪约
  - 匿名意向确认
- P2（PPT 中写为暂时不做）：
  - 即时通讯
  - 积分商城 / 奖励体系
  - 跨校匹配

### 2.7 PPT 中明确排除项

- 不做即时通讯，匹配成功后引导到微信
- 不做跨校，先聚焦本校
- 不做积分商城

### 2.8 PPT 中出现的角色与功能线索

- 角色：
  - 学生用户
  - 系统管理员
- 用例图中出现的功能线索：
  - 浏览广场
  - 智能精准筛选
  - 发布需求
  - 选择场景
  - 实名认证
  - 一键闪约
  - 查看靠谱分/勋章
  - 评价伙伴
  - 强制互评搭子
  - 审核内容
  - 调节信用分
  - 处理违规
  - 分析数据看板

### 2.9 PPT 中给出的验证口径

- 匹配成功率 ≥ 60%
- 周活跃用户使用频次 ≥ 3 次
- 用户满意度 ≥ 4/5

## 8. V1 范围结论

### 8.1 产品定位

- `校园搭子` 是一个面向校内学生的纯 Web 同伴匹配系统。
- 项目目标是可真实落地部署，而不是只做课堂展示。
- 第一版服务单校场景，但数据结构预留 `school_id`，为后续跨校扩展留空间。

### 8.2 第一版核心角色

- 学生用户：
  - 完成学号 + 校园邮箱认证
  - 发布搭子需求
  - 浏览广场与筛选
  - 接收系统推荐
  - 站内私信沟通
  - 活动后互评
  - 发起投诉 / 申诉
- 系统管理员：
  - 审核搭子需求
  - 审核用户资料 / 头像 / 昵称
  - 审核评价内容
  - 处理投诉与申诉
  - 管理用户状态
  - 干预靠谱分

### 8.3 第一版核心流程

1. 学生注册并完成学号 + 校园邮箱认证。
2. 学生选择场景类型，填写统一字段与场景化字段。
3. 搭子需求进入先审后发流程。
4. 审核通过后进入广场。
5. 其他学生通过广场筛选或系统推荐发现需求。
6. 双方通过站内私信沟通，私信支持文本、图片 / 截图、联系方式卡片与举报。
7. 活动或协作结束后进行互评。
8. 系统根据评价与履约情况调整靠谱分。
9. 用户可对评价、投诉或处理结果发起申诉。
10. 管理员查看证据与双方回应后处理申诉，并可执行信用分干预、警告、禁言或封禁。

### 8.4 第一版明确暂不做

- 跨校匹配
- 积分商城 / 奖励体系
- 复杂算法推荐
- 已读状态、撤回、黑名单等私信增强能力
- `一键闪约` 与 `匿名意向确认` 暂列为待评估增强项
