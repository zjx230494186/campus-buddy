# 22 CodeArts 长时间无人监督提示词工程优化 V1

本文基于 CodeArts 在 P0 账号认证持久化与 JWT 开发中的实际表现，沉淀后续提示词设计方法。目标是利用其强执行、强调试能力，同时约束其改动半径、Git 边界和测试节奏。

## 1. 模型表现评估

当前 CodeArts 智能体表现为：

| 维度 | 当前判断 |
|---|---|
| 代码生成能力 | 强，能推进 Spring Boot / JPA / Flyway / JWT 等完整后端链路 |
| 调试恢复能力 | 强，能连续处理 Maven、Testcontainers、Flyway、权限和测试配置问题 |
| 测试意识 | 有，能写测试并运行回归 |
| 测试节奏纪律 | 中等偏弱，容易把多个闭环一次性铺开 |
| Git 边界控制 | 弱，曾使用过大范围暂存，导致构建产物、生成物和备份目录进入基线提交 |
| 安全边界意识 | 较强，目前未见真实云密钥或数据库密码写入 |
| 无监督开发可靠性 | 可用，但必须有批次边界、停止条件和提交审计 |

一句话结论：它适合作为“高执行力工程代理”，不适合作为完全放飞的项目负责人。提示词必须提供任务预算、停止条件、Git 护栏和审计步骤。

## 2. 核心策略

不要给它一个“大目标”，而要给它一个“带预算、带停机条件、带提交边界的任务队列”。

不推荐：

```text
继续完成 P0 账号认证模块。
```

推荐：

```text
本轮是 P0 账号认证无人监督开发批次 01。最多推进 3 个小闭环。
每个闭环必须测试先行、单独记录红灯/绿灯、单独复核 git diff。
禁止 git add . 和 git add -A。
如果连续两次测试失败原因不同，停止并输出诊断报告。
```

## 3. 无人监督开发五段式

### 3.1 运行前准备

- 使用独立分支，例如 `ai/p0-auth-batch-01`。
- 开始前运行 `git status --short --branch`。
- 检查 `.gitignore`。
- 若发现构建产物、缓存、密钥或临时日志进入 Git 状态，先停下来做清理，不继续写业务代码。

### 3.2 任务预算

每轮提示词必须写明：

- 最多推进几个小闭环。
- 每个闭环最多修改多少个源代码文件。
- 同一个失败最多尝试修复几次。
- 是否允许新增依赖。
- 是否允许修改 API 契约。
- 是否允许创建提交。

推荐默认：

- 每轮最多 2-3 个小闭环。
- 每个闭环最多 8 个源代码文件。
- 同一类失败最多修复 2 次。
- 本轮最多 1 个开发提交。

### 3.3 自主批处理

允许 CodeArts 在一个批次内自主推进多个连续小闭环，但完成本批次后必须停下，不得自行进入下一批次。

示例拆分：

1. 仓库清理与 Git 卫生。
2. Flyway + 表结构 + Repository。
3. 注册持久化。
4. 登录持久化。
5. JWT 签发。
6. JWT 过滤器与受保护接口。
7. 文档与交接整理。

### 3.4 停止条件

出现以下情况必须停止：

- 需要新增超过 10 个源代码文件。
- 同一测试失败修复超过 2 次仍未通过。
- 需要修改既有 API 契约。
- 需要引入新中间件、新云服务、新 IAM 或新密钥。
- Docker/Testcontainers/数据库环境不可用。
- 发现敏感信息可能进入仓库。
- `git status` 出现 `target/`、`build/`、`.ninja_*`、`.env`、私钥、临时测试日志等不应提交内容。
- 为了通过测试需要删除关键测试、弱化断言或 mock 掉本轮核心路径。

### 3.5 每轮审计

每轮结束前必须执行：

```powershell
git status --short --branch
git diff --name-only
git diff --stat
```

如已提交，还需执行：

```powershell
git show --stat --oneline HEAD
```

并在验证记录中写明：

- 提交了哪些文件。
- 没有提交哪些临时文件。
- 测试命令与结果。
- 未完成事项。

## 4. Git 护栏

禁止：

```powershell
git add .
git add -A
```

只允许：

```powershell
git add <明确文件路径>
```

如果出现以下路径或文件，必须停止并清理：

```text
backend/target/
desktop/build*/
.ninja_*
CMakeFiles/
.env
*.key
*.pem
*.p12
*.pfx
*secret*
*credential*
backend/test_output*.txt
backend/test_error*.txt
```

## 5. 当前项目的即时处理建议

当前仓库已有基线提交：

```text
447b0af chore:-establish-project-baseline-and-backend-configuration
```

该提交存在 Git 卫生问题：包含 Qt 构建目录、`.ninja_*`、生成物、备份目录等不应作为长期源码基线的内容。

当前工作区还存在 P0 认证持久化/JWT 的未提交改动，以及若干 Maven/Testcontainers 调试输出：

```text
backend/test_output*.txt
backend/test_error*.txt
backend/test_flyway_out*.txt
```

因此下一轮不应继续加业务功能，应先让 CodeArts 执行“仓库审计与收束批次”：

1. 不写新业务代码。
2. 审计 `HEAD` 基线提交污染。
3. 审计当前未提交改动。
4. 更新 `.gitignore`。
5. 移除 Git 跟踪中的构建产物和备份目录，但不得删除本地文件，除非用户确认。
6. 删除或保留临时测试日志前先说明建议。
7. 运行当前可运行测试。
8. 给出是否适合提交 P0 认证开发改动的判断。

## 6. 标准无人监督提示词骨架

```text
你将继续接手《校园搭子平台》开发。本轮是“无人监督安全开发批次”，你可以自主推进，但必须严格受边界约束。

总原则：
- 不是一次性完成整个模块，而是完成一个可测试、可提交、可回滚的小闭环批次。
- 必须保持：测试先行 -> 红灯 -> 实现 -> 绿灯 -> 留档 -> Git diff/status 复核 -> 明确提交建议。

本轮目标：
- <填写一个明确目标>

本轮允许做：
- <允许项>

本轮明确不做：
- <禁止项>

开发预算：
- 最多完成 <N> 个小闭环。
- 每个小闭环最多修改 <N> 个源代码文件。
- 同一个失败最多尝试修复 2 次。

停止条件：
- <列出必须停止的条件>

Git 纪律：
- 开始前运行 `git status --short --branch`。
- 提交前运行 `git diff --name-only` 和 `git status --short --branch`。
- 禁止 `git add .` 和 `git add -A`。
- 只能 `git add <明确文件路径>`。

完成后输出：
1. 本轮完成了什么。
2. 哪些测试先红后绿。
3. 最终测试命令与结果。
4. validation 记录路径。
5. git status 摘要。
6. 是否创建提交，提交哈希是什么。
7. 明确未完成事项。
8. 下一轮建议。
```

## 7. 后续执行原则

- 可以让 CodeArts 长时间无人监督，但每轮必须是“短批次无人监督”，不能跨越多个模块。
- 每一轮结束后先做审计轮，再决定是否进入下一轮开发。
- 用户和本线程负责设计批次与验收门；CodeArts 负责连续写代码、跑测试、修复和留档。

## 8. Round 03 后新增优化：提交门禁独立成轮

Round 03 表明 CodeArts 能完成仓库清理和测试修复，但在真正创建提交前，仍需要单独的“提交门禁轮”。原因是当前项目基线曾包含生成物和构建产物，清理动作与业务代码改动混在同一工作区内，如果直接提交，容易再次形成边界不清的提交。

后续凡是出现“大批量清理 + 业务代码 + 文档更新”混合状态，应先进入提交门禁轮，而不是继续开发。

提交门禁轮必须检查：

- 当前是否只有预期分支和预期提交历史。
- `git status --short --branch` 是否包含构建产物、缓存、临时日志或密钥。
- `git ls-files` 是否仍跟踪 `desktop/build*`、`.ninja_*`、`.arts/`、`backend/target/`、`ppt_extracted.txt` 等明确污染项。
- 对 `latex_work/build_*/*.pdf`、`latex_work/build_*/*.docx` 等可能属于课程交付物的历史产物，应分类记录，不得擅自删除。
- 测试是否已复跑并通过。
- 敏感信息是否已复查。
- 提交是否可以拆分为“仓库卫生提交”和“业务功能提交”。

推荐提交顺序：

1. `chore(repo): clean generated artifacts and record git hygiene`
2. `feat(auth): persist authentication flow and introduce jwt`

如果无法拆分，或拆分后任一提交无法解释其独立价值，应停止并输出原因。
